# Council review — 1779273122

**Problem:** For "Gate 2 — compliance + auth" of the jarvis tutor redesign, what auth stack should be built? The redesign spec is internally inconsistent: §4.1/§5.1/§15 name "Auth.js v5 + Drizzle" (a JS/TS stack) for auth inside a Kotlin Ktor backend — factually impossible (Auth.js cannot run in a JVM). §15 falsely claims it is "already-shipped". Path B is council-locked as Kotlin/Ktor + Python-ML-sidecar only.

**Proposed approach (Option A):** Treat "Auth.js v5 + Drizzle" as a stale spec error. Build Gate 2 auth Kotlin-native: extend existing Ktor/Exposed auth into magic-link multi-user auth — Resend HTTP API for email delivery, Postgres-backed users/sessions/verification_tokens with Row-Level Security, drop the `accounts` table. Counter (Option B): a standalone Node Auth.js service as a cross-subsystem identity layer (§15.1 forward-compat).

**Project context:** Solo dev educational tutoring web app for ~10 friends. Kotlin/Ktor backend (~715 tests) + Vite/React frontend, SQLite + Exposed, one 2GB VPS. Path B council-locked (council-1779022799) = Kotlin + Python-ML-sidecar only. Gate 1 (viz foundation) shipped. Prod has 871 live FSRS cards in SQLite.

## 🔴 Devil's Advocate

AGENT: Devil's Advocate
STANCE: REJECT

REASONING: Option A's framing — "honors Path B, reuses working Kotlin code" — is a sleight of hand that smuggles a Postgres cutover with Row-Level Security past review under the cover of "just extend the existing auth." The verified reality: today's auth is three ~60-line files on SQLite (`Users.kt`/`Sessions.kt`/`Tokens.kt`), and `TutorDb.connect` hard-wires `jdbc:sqlite:` plus WAL/foreign-key pragmas with zero `accounts`/`verification_tokens` tables and zero Postgres anywhere in 200 .kt files or 715 tests. "Reuse working Kotlin code" is false: the moment you adopt Postgres+RLS you must rewrite the connection layer, port every Exposed table that currently relies on SQLite semantics, run a 871-card data migration that has no rollback story, and operate a second stateful database engine on a 2GB shared VPS. The proposal answers the auth-library question correctly (Auth.js-in-JVM is impossible) but does so while making a far riskier infrastructure bet that has nothing to do with the auth question and was never independently justified.

KEY CONCERN: Option A conflates two unrelated decisions and lets the easy one (drop the impossible Auth.js, keep Kotlin magic-link) launder the dangerous one (SQLite→Postgres + RLS). RLS is the tell: it is a Postgres feature that exists to defend a database against compromised or multi-tenant application code — for a single-process Kotlin monolith serving ~10 trusted friends where the app already scopes every query by `userId`, RLS adds a whole second, redundant authorization model with no threat it actually closes. Magic-link multi-user auth does NOT require Postgres at all. CONDITIONAL minimum: Gate 2 ships magic-link auth on the existing SQLite/Exposed stack with a new `verification_tokens` table and nothing else; the Postgres cutover is split into its own gate with its own council, a reversible migration plan for the 871 FSRS cards, and an explicit written answer to "what concrete attack does RLS stop that `WHERE user_id = ?` does not."

## 📚 Domain Expert

AGENT: Domain Expert
STANCE: APPROVE

REASONING: Option A is the established pattern, not a reinvented wheel. "Magic-link auth without a JS framework" is a fully solved problem on the JVM: the canonical recipe is a single-use, hashed, short-TTL token row + an opaque server-side session ID in an httpOnly/SameSite cookie — exactly what the existing `TokensTable`/`SessionRepo` already implement. Ktor ships `ktor-server-auth` + `ktor-server-sessions` (both already declared in `build.gradle.kts`) precisely so you don't hand-roll a framework; the right move is to register a `session<T>` provider + an `authenticate("session")` block rather than the current 35-site repeated `call.request.cookies["jarvis_session"]` lookup. Auth.js/NextAuth + the Drizzle adapter is a Node-runtime library and literally cannot run in a JVM; treating §4.1/§5.1/§15 as stale spec error is correct, and the `accounts` table is dead weight (NextAuth OAuth-provider join table — with magic-link only, dropping it is right). Option B's standing Node identity service contradicts the council-locked Path B two-runtime rule and would be a textbook YAGNI for subsystems that do not exist.

On the secondary question: do NOT do a full SQLite→Postgres domain cutover in Gate 2. Flyway `V001__baseline` should describe only the auth + compliance tables on a fresh Postgres, run dual-DB (Postgres for auth, SQLite for domain) until a later gate, and treat the 871-card FSRS migration as its own gate with rehearse-on-a-copy + row-count-assert + checksum verification.

KEY CONCERN: One real divergence in the current code that Gate 2 must fix: `SessionRepo.create` stores the raw session ID directly as the table primary key — session IDs are bearer credentials, so a leaked DB backup or SQL-injection read exposes live, directly-usable session IDs. Hash the stored session lookup key just as `TokensTable` already hashes the token. The magic-link token must be single-use with a short TTL (~10–15 min) and consumed atomically (delete-on-redeem in the same transaction as session creation) — the current `tokens` table has no `expires_at` and `findUserIdByToken` does not consume on read, so a magic link would currently be infinitely replayable.

## ⚙️ Pragmatist

AGENT: Pragmatist
STANCE: CONDITIONAL

REASONING: Option A is the only buildable plan — Auth.js cannot run in a JVM, and a standalone Node identity service means a third runtime (Kotlin + Node + future Python) competing for ~2GB on one VPS, which is operationally absurd for a solo dev serving ~10 users during finals. But Option A as written smuggles in two cost bombs that are NOT auth: a Postgres migration and Row-Level Security. RLS for a 2-scope (OWNER/FRIEND) app where the Ktor layer already gates every request is enterprise ceremony that buys nothing and adds a debugging surface (silent empty result sets) one person will hate. The honest scope: keep SQLite, extend the existing `tokens`/`SessionRepo`/`UsersTable` code into multi-user magic-link via the Resend HTTP API, enforce scope in app code as you already do — 90% of the result at 20% of the cost.

KEY CONCERN: Scope creep disguised as "auth." Gate 2 should ship magic-link multi-user on the existing SQLite + Kotlin stack and explicitly DEFER both the Postgres cutover and RLS. A full SQLite→Postgres cutover risking 871 live FSRS cards has no user-visible payoff at 10 users and must be its own dedicated gate with a tested dump/restore + rollback path — never bundled into the auth gate.

## 🧱 First Principles

AGENT: First Principles
STANCE: APPROVE

REASONING: Strip the framing and the real goal is tiny: prove to ~10 known humans that they are who they say, gate a compliance surface, and never block yourself the solo dev. The existing Kotlin code already does identity (httpOnly cookie, CSRF double-submit, one-time token table, OWNER/FRIEND scope) — it is 90% of a magic-link flow; the only missing piece is "deliver the token by email instead of pasting a URL," which is one HTTP POST to Resend. "Auth.js v5 + Drizzle" is not a requirement, it is a diagram drawn by someone who confused NextAuth's reference schema for a design — it cannot run in a JVM, and Option B's "third Node process as global identity layer" spends a permanent ~150MB RSS and a whole runtime on a 2GB VPS to serve a login that happens maybe twice a week. From a blank sheet, for this exact situation, you would write exactly Option A.

KEY CONCERN: The genuine first-principles question is not "Auth.js vs Kotlin" — it is "do you need Postgres + RLS at all for 10 friends?" RLS is a heavyweight DB feature justified by untrusted multi-tenant isolation; here every row is owned by the dev or one of 10 vetted friends, and a `user_id` WHERE-clause filter in the repo layer is sufficient. Decouple the two cutovers: build Gate 2 auth Kotlin-native on the current SQLite, keep the 871 FSRS cards untouched, and do not let a stale spec drag a database migration + RLS into an auth gate that does not need either.

## ⚠️ Risk Analyst

AGENT: Risk Analyst
STANCE: CONDITIONAL

REASONING: Magic-link auth Kotlin-native is a HIGH-but-manageable risk surface — token replay, session fixation, and Resend deliverability are all standard, well-mitigated problems (the existing one-time `tokens` table already does single-use exchange; the existing `csrf` double-submit already exists). The genuinely CRITICAL risk is the secondary question: a full SQLite→Postgres cutover that must preserve 871 live FSRS cards, because FSRS card state (stability, difficulty, due timestamps, review logs) is irreplaceable user data — corrupt it and every user's scheduling history is silently wrong with no source of truth to restore from. RLS misconfiguration is the second real concern: in a self-rolled Kotlin/Exposed setup, RLS only fires if every connection sets the per-request user GUC correctly, and Exposed connection pooling makes it easy to leak a previous request's identity onto a reused connection — so RLS gives a false sense of safety while app-layer `WHERE user_id = ?` scoping is what actually protects you.

KEY CONCERN: The 871-card cutover is the catastrophic, non-recoverable failure mode — Gate 2 should stand up Postgres for auth/compliance tables ONLY and keep FSRS domain data on SQLite until a dedicated migration gate. Mandatory mitigations if any cutover happens: (1) row-count + per-card checksum assertion before/after, (2) immutable timestamped SQLite backup copied off-VPS before Flyway runs, (3) dry-run the migration against a copy of prod, (4) verify FSRS due-date math on migrated rows. For the auth itself: short link TTL (~10–15 min) + single-use enforced at the DB row, rotate session ID on login (kill fixation), keep the existing CSRF double-submit, treat RLS as defense-in-depth only.

KEY ISSUES:
- Postgres + RLS do not belong in Gate 2. RECOMMENDED PATH:
Build Gate 2 as Kotlin-native magic-link multi-user auth on the existing SQLite + Exposed stack. Add `email`/`expires_at` to a single-use token table, deliver links via the Resend HTTP API, hash stored session keys, rotate session ID on login, consolidate cookie handling into a Ktor session provider. Keep the AI-literacy first-login gate + Settings/Me + the 15 compliance docs in Gate 2 (they do not depend on Postgres). Spin the SQLite→Postgres cutover + RLS into a separate later gate with its own council and a rehearsed, reversible 871-card migration.
