## 📚 Domain Expert
STANCE: REJECT
REASONING: "Build all backend, then all UI, verify content later" is the layer-cake (horizontal-slice) anti-pattern, the exact failure the walking-skeleton (Cockburn) and end-to-end thin vertical slice norms exist to prevent — you wire one paper-thin path through every layer (ingest → audit → faithful-gate → queue → serve → render) so the integration seams are proven before widening any layer. The proposed Phase 5 builds a real UI over a pipeline whose load-bearing seam — offline SymPy+NLI audit writing `faithful` rows — has never executed once on the real corpus; `verification_audit` and `kc_verification_status` are empty, so the faithful-gated queue (TutorRoutes.kt:2157 DENYs every KC with no faithful row) returns nothing. The "Phases 0-4 done, backend-first" framing is a definition-of-done error: a verification-gated system's done bar is *one real claim verified and served end-to-end*, not *all handlers compiled and unit-green* — passing on TEST FIXTURES is the false-confidence thin-slicing eliminates. Correct order is A first (stand up the NLI/SymPy checker locally, run the live audit, land ≥1 real PA KC faithful), then B (UI over a proven path); option C (lesson-serve backend) is real missing scope but a second vertical slice — defer it.
KEY CONCERN: The single highest-risk, least-proven component is the offline SymPy+NLI audit checker (DOWN, unrun on real content), yet the proposed order verifies it LAST — building the most expensive layer (UI) on the dependency most likely to change the data contract or fail. Retire that risk FIRST.
