## 🔴 Devil's Advocate
AGENT: Devil's Advocate
STANCE: CONDITIONAL
REASONING: The MatrixGrid exception's stated justification is provably false. Its docblock claims gates read ONLY data-* attributes so moving originY is safe. But the frame-conjunction gate does NOT read data-* — it perceptually diffs PNG PIXELS of a crop. Moving a centered grid to top-packed + shrinking the viewBox changes those pixels. The only reason the coord-move is actually safe is that frame-conjunction AND svg-overlap are hard-bound to the mergesort corpus (/tutor/lectie-mergesort, /tutor/merge-compare, sort-merge-root) — they never visit matrix-grid/gauss/graphtree/class-diagram. So the conclusion holds by NON-COVERAGE, not the cited mechanism: the commit ships a load-bearing FALSE claim about WHY it's safe (the #1 ungrounded-claim pattern). Separately, the max-over-frames height is a measure-time estimate re-deriving wrap/descender via font*0.25 that must agree pixel-for-pixel with paint-time webfont metrics — the same font-advance mismatch the class-diagram D1 gate exists to catch. One extra wrapped line → last callout line lands below the hugged box edge with ~2px margin, and NO gate covers these families to catch it.
KEY CONCERN: The "gate-safe" justification is factually wrong about frame-conjunction (pixels, not data-*); the change is safe only because those gates don't cover these 5 families — so the hugging ships UNGATED for clip/overlap/freeze atop a false rationale. Lift condition: extend frame-conjunction + svg-overlap to these routes, OR correct the docblock to state the real reason (non-coverage) + add an explicit per-family no-clip assertion on the hugged bottom edge.

## 📚 Domain Expert
AGENT: Domain Expert
STANCE: APPROVE
REASONING: Per-family content-driven sizing via hand-computed formulas IS the established pattern, not the alternative. D3's margin convention (Bostock 2011) has each chart compute its own width/height into the viewBox — no generic auto-measure loop. Visx: explicit width/height props derived from data or a ParentSize wrapper that measures the CONTAINER, not SVG internals. Observable Plot computes dims from the mark set pre-render. Mermaid is the only major tool doing post-render getBBox(), precisely because it can't know extent until the DOM is live — a two-pass render with a frame flash that can't measure server-side. This codebase's no-clip harness runs in Node without a DOM → getBBox() is structurally incompatible. "Size to the max frame" for multi-frame figures is correct standard (a box that resizes between frames causes layout shift breaking reading + pixel gates). Graph-tree sparse on frame 1 is handled right: computeContentHeight takes MAX across frames.
KEY CONCERN: ClassDiagramFamily's mergedLayout spread puts viewBoxW/H last (`{...shellLayout, viewBoxW: layout.viewW, ...}`) so a caller-supplied viewBoxW is silently overridden — a latent API contract bug (not a blocker; no caller passes it today). Either document "family-derived wins" or flip the spread so caller overrides win with a Math.max minimum.

## 🧱 First Principles
AGENT: First Principles
STANCE: CONDITIONAL
REASONING: The real goal (PRODUCT.md) is "attention on the ONE thing; figures EXTRAORDINARY, not just not-floating." The change solves half. fixed-classdiagram.png = success (already-dense diagram → tighter, more authoritative). fixed-seqarray.png = acceptable (the array fills the box). But fixed-graphtree.png is the failure the framing suppresses: a 6-element array + one callout line; after hugging, the box is dominated by the array row, the tree communicates almost nothing, and the tight frame doesn't rescue it — it just shrinks the emptiness. For sparse/shallow figures the underlying data is too sparse for the real-estate regardless of viewBox. Content-driven viewBox is correct for dense figures; a local optimum for sparse ones where the real fix is to draw BIGGER/denser or add a second info layer (invariant, complexity, edge weights).
KEY CONCERN: fixed-graphtree.png is hugged but still mediocre — shrinking the frame around sparse content yields a small mediocre figure, not an extraordinary one. The gate should require minimum information DENSITY, not just no-void: if contentH < DENSITY_FLOOR (~120px meaningful content), scale up or augment, don't just frame tighter.

## ⚠️ Risk Analyst
AGENT: Risk Analyst
STANCE: APPROVE
REASONING: Traced every family's height formula vs diff + PNGs. Core safety holds: each family's internal SVG_W is literally 480, content packed into [0,480], so new defaults reproduce prior behavior byte-for-byte — the half-opt-in found (SortMergeFamily sets viewBoxH not viewBoxW) is safe by construction. MatrixGrid's new viewBoxH puts the callout in the TOP band (grows downward toward the grid), so a long callout can only collide with the grid (pre-existing 40px-band property) and can NEVER clip the bottom. SequenceArray `indexY+12` and GraphTree per-frame `wantTop+blockH+font*0.25` correctly model the lowest painted pixel; GraphTree's MAX-over-frames keeps the box stable and reuses the value in the renderer's clamp. MatrixGrid coord-move only shifts absolute y; trace-match/structure gates read data-* only; frame-conjunction compares deltas where all frames share the new layout → no false freeze/overlap.
KEY CONCERN: MEDIUM — GraphTree (and pre-existing ClassDiagram width-scale) cap at SVG_H=360: a deep tree exceeding 360 → computeContentHeight returns 360 while the renderer paints below → silent bottom clip. PRESERVED pre-existing ceiling (box was always 360), not introduced here, recoverable. Mitigation: the no-clip e2e gate must include a deep-tree fixture (depth ≥5) so the 360 ceiling is exercised, not just the shallow trees in the PNGs.
