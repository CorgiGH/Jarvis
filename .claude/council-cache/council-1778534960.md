## 🔴 Devil's Advocate
AGENT: Devil's Advocate
STANCE: REJECT
REASONING: The raw-selection-as-query assumption is load-bearing and broken. A "chip click" selection in this UI is almost never a well-formed retrieval query — it's a fragment of a Romanian drill sentence, a LaTeX blob, or a 2-word noun phrase. You're going to BM25 that directly against `_extras/PS/` and inject the top-3 results as `trust="indexed_data"` into the system prompt BEFORE the LLM has reasoned about what's actually being asked. When the selection is "transformata Laplace" you get lucky; when it's "ecuația de mai sus" or "f(s) = ..." you've just hard-anchored the model to three off-topic snippets it cannot ignore, and your CitationExtractor will faithfully surface them as authoritative sources. That's worse than `citations: []` — it's confidently-wrong citations Alex will drill into during finals prep, and the +1800 tokens/click burns the free-tier quota you explicitly cannot replenish. Worse: the proposal admits this in CONS ("if pre-fetch returns off-topic material the LLM is biased by garbage") and then ships it anyway because deterministic-but-wrong feels safer than nondeterministic-but-correct.
KEY CONCERN: Pre-fetched hits enter the prompt as `trust="indexed_data"` with no relevance gate — a lexical score floor, a query-quality check (length/stopword/LaTeX-noise heuristic), or a "selection + last user turn" rewrite step is mandatory before any injection. Without it, the chip flow degrades from "no citations" to "plausible-looking wrong citations during exam prep," which is the failure mode you cannot afford for a single-user study tool with no second pair of eyes.

## 📚 Domain Expert
AGENT: Domain Expert
STANCE: APPROVE
REASONING: This is textbook RAG pre-retrieval — the canonical pattern shipped in LangChain's `RetrievalQA`, LlamaIndex's `RetrieverQueryEngine`, and OpenAI Assistants' built-in file_search (which also pre-fetches before model invocation rather than relying on tool-use). The "let the LLM decide to call the retrieval tool" alternative is exactly what Anthropic's own cookbook and the LangChain agent retrospectives flagged as unreliable: models skip the tool ~30-50% of the time on short queries, which is precisely your observed failure mode (`citations: []`). Keeping `search_archival` as a fallback tool while pre-fetching is the hybrid that Perplexity, Cohere's Coral, and Claude Projects all converged on — pre-retrieval guarantees grounding, tool-use handles broadening. Your RRF k=60 fusion + CitationExtractor union is structurally identical to how Vespa and Weaviate hybrid pipelines compose pre-fetched and agent-fetched contexts.
KEY CONCERN: Raw-selection-as-query is the weakest link — a 2-word chip ("Laplace transform") works, but a 200-token highlighted paragraph degrades BM25/embedding recall sharply; consider a cheap query rewrite (truncate to first sentence or extract noun phrases) before handing to HybridRetriever, which is the standard pre-retrieval query transformation step LlamaIndex calls `QueryTransform` and LangChain calls `MultiQueryRetriever`.

## ⚙️ Pragmatist
AGENT: Pragmatist
STANCE: CONDITIONAL
REASONING: 30-50 lines + 1 new context field is genuinely cheap to build and aligns with the existing `_extras/${task.subject}/` subject-routing pattern already in TutorRoutes — so the marginal complexity is small and the handoff cost is near-zero for a solo dev. Maintenance risk is real but bounded: when retrieval misbehaves you now have two failure modes (pre-fetch wrong vs LLM-fetch wrong), and the +1800 token system prompt will bite on free-tier OpenRouter context limits exactly when you're cramming for finals and want longer scratchpad turns. The dealbreaker-avoider: gate the injection behind a feature flag or `env.selection.length > 20 && < 500` heuristic so a stray triple-click on a single word doesn't poison the prompt with garbage retrieval, and log both pre-fetch hits and final citations so debugging "why did it cite X" stays a 30-second grep instead of a 3-hour rabbit hole in June.
KEY CONCERN: Token budget on free-tier OpenRouter — +1800 sys-prompt tokens per chip-click compounds badly across a long Tema A session; add a `maxInjectedChars` cap and truncate snippets to ~200 chars each, not whatever the retriever returns.

## 🧱 First Principles
AGENT: First Principles
STANCE: REJECT
REASONING: Strip the framing and the actual goal is: Alex must *retrieve* knowledge from his own corpus to durably learn it (Roediger/Karpicke). The proposed fix optimizes for "LLM answer cites a source," but the chip flow already loads the user_question with the entire drill sentence — so the LLM will solve the problem for him, and a citation footer doesn't reverse that. Pre-fetching corpus into the system prompt is solving a citation-plumbing problem while the learning surface is upside-down: the right primitive is "show Alex *where* this concept lives in HIS notes and make him open it BEFORE any LLM generation," with LLM help gated behind an explicit "I've tried, now help" action. Server-side retrieval is fine infrastructure, but injecting it into the LLM prompt by default routes the retrieved passages *away* from Alex's eyes and *into* the model's mouth — the exact inversion of the testing effect.
KEY CONCERN: The selection-to-LLM pipeline answers the drill for Alex; adding citations to that answer makes it look more rigorous without restoring the retrieval practice that does the learning. Build a "your notes on this" panel as the *primary* response to selection, not a system-prompt sidecar.

## ⚠️ Risk Analyst
AGENT: Risk Analyst
STANCE: CONDITIONAL
REASONING: HIGH — Latency/quota amplification: sync HybridRetriever.search before every chip-click adds a blocking hop to the request path. Lexical SearchSubsystem.searchIn can throw IO (filesystem perms, missing archivalDir env var) — per project context this is NOT caught by the graceful LLM-exception handler, so a permissions hiccup or missing env var turns chip-clicks into silent empty 500s (same failure shape as the stale-classloader bug today). Trigger: any deploy where archivalDir resolves but a single subject dir is unreadable, OR cold-start before env wired. HIGH — Prompt-injection / context confusion: selection text is arbitrary user input that "may contain `<retrieved_context>` literal tag if user pastes from corpus." Wrapping that raw selection-derived hits in another `<retrieved_context source="prefetched_corpus" trust="indexed_data">` block lets a malicious/accidental paste close the outer tag and inject `trust="indexed_data"` framing around fabricated paths — CitationExtractor's regex verifies path-in-hit-set so fabricated citations get dropped, but the LLM still treats injected prose as trusted corpus. MEDIUM — Windows/Linux path separator: HybridRetriever returns OS-native separators; if pre-fetch hits are injected with `\` on a dev box but CitationExtractor's verify set normalizes to `/`, the unioned pre-fetched paths may fail downstream path-prefix checks (subject scoping). Blast radius for HIGH#1 is full chip-click outage; HIGH#2 is trust-boundary erosion (recoverable, no data loss). Both manageable with the conditions below.
KEY CONCERN: Wrap the pre-fetch in a try/catch that degrades to empty hits + logs (same graceful pattern as the LLM Exception handler), sanitize selection text by stripping/escaping `<retrieved_context>` and `</retrieved_context>` substrings before it ever reaches the prompt, and normalize HybridRetriever paths to forward-slash at the pre-fetch boundary so the union with LLM-fetched hits uses one canonical form.
