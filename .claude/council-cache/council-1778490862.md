## 📚 Domain Expert

AGENT: Domain Expert
STANCE: REJECT
REASONING: The critique misreads productive failure (Kapur, 2012, 2015): PF requires *initial attempt before instruction*, then worked-example reveal — which is exactly what the current "drill unlocked, worked visible if you scroll" design supports. Forcing definition→worked→drill order inverts this and triggers the expertise-reversal effect only for already-fluent learners (Kalyuga, Ayres, Chandler, Sweller 2003); for true novices like Alex, the right move is *faded worked examples* (Renkl & Atkinson 2003) — show worked first, fade steps over successive problems — NOT a permanent zero-knowledge mode. Two-mode user-toggled UIs are a known anti-pattern: Khan Academy's Mastery, ALEKS's Knowledge Space Theory, and Brilliant all ship *adaptive per-concept ordering* driven by response data, never a "I'm a beginner" switch (the self-assessment is unreliable per Dunning-Kruger and Kruger & Dunning 1999, and Alex specifically is in the low-skill/high-uncertainty zone where self-rated mode picks will be wrong). LLM-estimated task times teach miscalibration — Duolingo shows fixed XP not minutes, Anki shows card counts, Khan shows mastery percent; surfacing fake-precise minutes is worse than omitting them (planning fallacy, Buehler et al. 1994). Inline glossary is already ConceptInline's job (delta is content coverage, not architecture), and the "brutalist UI harms stressed learners" claim has no empirical backing — Mayer's multimedia principles (coherence, signaling, redundancy) favor *less* chrome, not more warmth; the affective-learning literature (Pekrun's control-value theory) ties emotion to perceived competence and autonomy, both of which a mode-toggle actively undermines.
KEY CONCERN: Adopting the critique replaces a research-aligned productive-failure + expertise-reversal-aware design with a self-assessed two-mode UI that no established adaptive-learning platform ships, against direct evidence (Kalyuga, Renkl, KA/ALEKS/Brilliant product decisions) — the correct fix for Alex's novice state is faded worked examples and adaptive scaffolding inside the existing single flow, not a beginner toggle.

## ⚙️ Pragmatist

AGENT: Pragmatist
STANCE: CONDITIONAL — approve D + cheap version of A/F as a single "starter hint" UX touch; reject B; defer C and E.
REASONING: Cost split is sharp. D (hide DAEMON DOWN / domain footer behind a debug flag) is ~30min, zero QA matrix, pure win for a single-user student build — ship it. The pedagogy gap is discoverability, not locking: cards are already unlocked, so a mode flip (A) or branching modal (F) is a ~6-8h build that doubles state-bleed risk (drill-card progress persisting across mode toggles is the classic two-mode bug) for a problem solvable by a single auto-scroll-to-DEFINITION + a one-line "Start here if new" badge on the DEFINITION card (~45min, no new mode). B (LLM time estimates) is med-cost (~2-3h prompt + cache + UI) but actively harmful — LLM estimates on novel material are wrong, anchor expectations badly, and erode trust the first time "~45min" takes 3h; reject. C (hover glossary) needs a term extraction pass + tooltip component + LLM definitions per term — that's ~6-10h and breaks on mobile (no hover); defer until post-finals. E (prereqs + output per task) is cheap to display (~1h) but expensive to populate accurately — the LLM has to actually KNOW the prereq graph, which it doesn't reliably; defer.
KEY CONCERN: The critique is solving a discoverability bug with architecture (modes, branches, LLM metadata pipelines). A single "Start here →" affordance on the DEFINITION card + auto-scroll-to-top on task open solves 80% of the zero-knowledge problem at ~5% of the proposed cost, with zero new QA surface. Don't build modes for a single user when a CSS scroll + one badge does the job before Jun 1.
