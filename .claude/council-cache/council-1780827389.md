# Council review — 1780827389

**Problem (the REAL one — a dev-PROCESS issue):** Loop = build → AI adversarial bug-find → fix findings → claim "fixed/closed" → a LATER separate bug-find surfaces a bug in the same area "supposedly fixed." VERIFIED instance: prior fix closed "both AIs call a definition wrong → still passes" + tested exactly that; handoff said "false-faithful holes closed" (broad); the fix was instance-narrow; a later review found the SIBLING ("only ONE AI calls it wrong → still passes"). NOT a regression — a SCOPE MISMATCH (narrow fix, broad claim, next review finds the adjacent sibling). Fixer & re-reviewer are different memoryless AI instances; "fixed" is free-text in commits/handoff.
**Proposed fix (evaluated):** treat every surfaced bug as a CLASS: (a) fix enumerates + closes + regression-tests ALL siblings; OR (b) honestly scope the "done" claim ("closed instance X; siblings unverified").
**Timestamp:** 2026-06-07 (epoch 1780827389)

## 🔴 Devil's Advocate
STANCE: CONDITIONAL
REASONING: The fix targets the wrong layer — it asks the FIXER (an AI) to self-police scope, but the failure was a TRUST gap: a broad claim entered the handoff/commit log and nothing checked it. Self-policing by the same process that already over-claimed is unenforceable; option (b) becomes a moral-hazard escape hatch ("siblings unverified" defaults to a rubber-stamp because it's the cheaper branch), so the open hole just migrates from the commit message into a caveat nobody re-reads. The real fix is structural: the REVIEWER (not the fixer) emits the class definition + sibling set as machine-checkable artifacts, and an external GATE fails the "closed" claim until a regression test exists per named sibling.
KEY CONCERN: Self-attested scoping by the AI that over-claimed is not a control; the class boundary + sibling set must be produced by the adversarial reviewer and enforced by an external gate, not promised by the fixer.

## 📚 Domain Expert
STANCE: CONDITIONAL
REASONING: Clause (b) honest-scoping is truth-in-labeling on the definition-of-done — good but the weak half. Clause (a) "enumerate + regression-test all siblings" is symptom-fixing one input at a time — exactly the instance-narrow trap one level up (it's how "both-wrong fixed, one-wrong missed" happened). Best practice: escalate to ROOT CAUSE (5 Whys) until you reach a single CLASS-KILLING INVARIANT ("verification passes ONLY IF every AI's definition-call matches the reference"); assert it once so the whole equivalence class (both-wrong/one-wrong/none-wrong) collapses to one guarded condition and siblings cannot exist — paired with RED-GREEN (write the failing sibling test FIRST). Hand the AI reviewer the bug-CLASS and demand an exhaustive equivalence-partition of the truth-table BEFORE "done."
KEY CONCERN: Clause (a) is still per-instance symptom work and leaks the next un-enumerated sibling; mandate root-cause invariant-first (collapse the class), sibling-enumeration only as fallback when no single invariant exists.

## ⚙️ Pragmatist
STANCE: CONDITIONAL
REASONING: Option (b) is the cheap 90% win (one honest sentence per fix, within a burst budget); "enumerate ALL siblings every time" is unbounded work the dev provably can't complete (can't list siblings you don't know exist), so mandating it just manufactures a NEW false claim ("all siblings closed"). The real damage is the poisoned handoff — a broad "class closed" makes the next session + the AI re-reviewer trust a falsehood, burning a re-review cycle; precise scoping turns that wasted re-review into a productive one. Enumeration earns its cost only when cheap-and-bounded (a quick grep over the same call-site pattern) — opt-in, never a per-fix mandate.
KEY CONCERN: Discipline-the-claim only pays off if MECHANICALLY ENFORCED — make the commit/handoff template DEFAULT to "closed instance X; siblings unverified"; you must DELETE the disclaimer to claim broader, so honest scope is the path of least resistance.

## 🧱 First Principles
STANCE: CONDITIONAL
REASONING: The goal is not "make fixes wider" (unbounded — every bug has infinite adjacent siblings; "all" is undecidable). It's to make "fixed" a durable, CHECKABLE artifact so a closed bug cannot silently resurface and scope is visible by construction. A permanent RED-GREEN regression test per finding IS the machine-checkable proof: it pins the precise scope actually closed and survives the amnesiac fixer/reviewer instances that free-text claims cannot. Under that regime a later reviewer finding a sibling is automatically HONEST ("no test asserts that sibling yet") rather than a broken promise. So the proposal is half-right — (b) scope-honesty is the real win — but (a) enumerate-all-siblings is the wrong primitive; test-as-proof is.
KEY CONCERN: A test only pins what it asserts — the contract must require the test's NAME/scope to literally equal the handoff claim's scope (test title = the "fixed" sentence), else you've just moved the free-text scope mismatch into the test name.

## ⚠️ Risk Analyst
STANCE: CONDITIONAL
REASONING: The worst failure mode is clause (a): "class enumerated + closed + tested" is treated by a different memoryless AI as a sealed-class guarantee, so the adjacent sibling it missed now hides behind a STRONGER badge than the free-text claim it replaced — false-FAITHFUL-of-the-process, converting an unverified gap into an asserted-closed one. Option (b) is cheap+correct but unenforced (relies on recall that already failed). Both example-based defenses share the root flaw: they trust a SAMPLING review to certify a POPULATION — logically unsound; no count of closed instances ever proves a class closed. The only mitigation: stop certifying by enumeration; encode the class as a standing EXECUTABLE INVARIANT — a property/metamorphic test on the trust gate ("for all generated content, REFUTE/contradiction ⇒ faithful badge MUST be false") run every build over randomized + adversarial inputs — catching present AND future siblings nobody enumerated, converting a prose claim into a CI-gating artifact.
KEY CONCERN: CRITICAL — clause (a)'s false-completeness. "Enumerated the class" is an unfalsifiable over-claim from a sampling review. Ship only if "fixed/closed" is REDEFINED to mean "a CI-enforced property invariant covers this class"; any fix lacking one must use (b) honest-scoping verbatim — the invariant being the enforcement (b) otherwise lacks.
