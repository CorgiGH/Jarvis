## 🔴 Devil's Advocate

```
AGENT: Devil's Advocate
STANCE: REJECT

REASONING: The whole design is built on the assumption that the PC activity logger is a benign 5-min POST that "happens concurrently" — but you already conceded the in-process write-mutex won't help cross-process, and you're about to add a second writer (conversations.jsonl) plus a third (archival.db) plus a fourth (core_memory.md) on the same disk under a 512MB JVM with no cross-process lock story. The mutex you "already shipped" is a comfort blanket — it only protects you from yourself inside one JVM. The migrator that walks wiki.md "## conversation (...)" sections runs against a live file the running server is appending to, and the only proof you have it works is that nothing exploded last time you batch-touched 988 .md files — except something did explode, you reverted, and the .bak is still sitting there as evidence you don't actually know what your write surface looks like. You're solving the wrong problem in the wrong order: you're refactoring storage layout before you've defined a single durable invariant about who is allowed to write what, when, from which process.

KEY CONCERN: There is no cross-process write contract. Every "shipped" safety claim in the brief is single-JVM. Adding three more writable surfaces multiplies the unprotected boundary instead of fixing it.

RECOMMENDATIONS:
A) FTS5. Embeddings infra "already working" for wiki entries is exactly the trap — it works at N=tens, not N=316 growing. $0.20 is irrelevant; the real cost is that an embeddings-only search has no debuggable failure mode when it returns garbage, and you have no eval set. Lexical first, embeddings later as a rerank layer if FTS5 actually proves insufficient (it won't, for one user, 316 docs).
B) Freeze writes during migration. Hard stop: take the server down, run migrator, bring it up pointed at the new file. The PC logger queues locally for the ~30s window. Anything else is theatre — and write a single SqliteWriteCoordinator with a file-lock (FileChannel.tryLock on a sentinel file) before you add a second process to the equation. No coordinator, no Step B.
C) Do NOT auto-import. core_memory.md is hand-curated only, and matricol / address / ADHD self-report do not belong in every model call shipped to OpenRouter — that's a third-party egress of identifying data on every turn for zero retrieval benefit. Seed it with: language pref, finals-window date guard, sidetrack-refusal rule. That's it. Identity facts stay in MEMORY.md retrieval-only.
D) Add "kind" day 1. Renaming a JSONL field later means rewriting 1000 entries under the same cross-process write hazard you haven't solved. Cost of adding it now: one enum. Cost of adding it in 3 weeks: another migration, another .bak file you don't trust.
E) /api/search returns ranked {path, snippet, score, mtime} — pure read endpoint, no side effects, no chat injection. The "search subsystem" is the indexer + ranker behind it; they are not two surfaces, stop pretending they are. If you can't describe them in one sentence each you haven't designed them.
F) Cut everything that isn't the write-coordinator + conversations.jsonl + FIFO truncate. Ship that. Watch it for a week. Then archival. Then core_memory. You are about to ship four subsystems in one step on a stack that has a documented retrieval-collapse incident from the last batch change.

WHAT NOT TO SHIP IN v1:
- Archival FTS5 indexer (Step C)
- core_memory.md auto-injection (Step D, hand-curated only when it ships)
- Any auto-import from MEMORY.md
- buildChatContext rewire that removes wiki.md (keep wiki.md as read fallback for at least one week post-conversations.jsonl)
- Mtime-based incremental indexing (premature; full rescan is fine at 316 docs)
- The "Alternative: embeddings" path — pick one, delete the other from the design doc
- Deletion of wiki.md.bak.1778090717 (it stays until conversations.jsonl has survived 7 days of live writes)
```

## 📚 Domain Expert

```
AGENT: Domain Expert
STANCE: CONDITIONAL
REASONING: The four-layer split (pinned core + recency window + archival retrieval + activity context) is exactly the Letta/MemGPT canonical architecture — `core_memory` blocks prepended to system prompt, `recall_memory` (chat history), `archival_memory` (vector or FTS), with retrieval gated through tool calls rather than auto-injection. Khoj uses the same separation (chat session vs `/api/search` against an embedded index). Where the proposal diverges: choosing FTS5 over the already-working VectorStore reinvents a wheel you've already built — Khoj defaults to embeddings (sentence-transformers / OpenAI), Mem0 uses Qdrant+embeddings, Letta uses pgvector, AnythingLLM uses LanceDB. Nobody serious in this category ships FTS-only in 2026; FTS is a fallback for offline/no-API-key mode (Khoj has it as `search_type=word` alongside `search_type=text` semantic). The conversations.jsonl + JSON-per-line format is fine for single-user scale (Khoj's `conversation.json`, Letta's early SQLite `messages` table both started this way), and a flat role/content schema today with a forward-compatible `kind` field is the right call.
KEY CONCERN: Picking FTS5 "because it's already free" when you have a working embedding stack is a regression to 2022-era RAG and will hurt recall on personal-knowledge queries (paraphrase, synonym, "what did I say about my prof") which is exactly where this class of system lives or dies.

RECOMMENDATIONS:
A) Reuse VectorStore + EmbeddingsClient. $0.20 one-off is nothing and you've already paid the integration cost. This is what Khoj (sentence-transformers default), Letta (pgvector), Mem0 (Qdrant), AnythingLLM (LanceDB) all do. FTS5 only makes sense as a *secondary* lexical channel for hybrid retrieval (Khoj's `search_type=word` mode, or the BM25+dense combo Vespa/Weaviate ship). If you want hybrid later, add SQLite FTS5 as channel 2 and RRF-merge — but ship dense-only v1.
B) Online dual-write with a generation marker, not downtime. Standard pattern from Stripe/GitHub online-migration playbooks: (1) deploy code that writes to BOTH old wiki.md and new conversations.jsonl, (2) backfill historical wiki.md → jsonl with a `migrated_from_wiki=true` flag, (3) flip readers to new store, (4) stop dual-write. For your single-user scale this collapses to: ship writer-side dual-write in one commit, run a one-shot migrator script idempotently (keyed by msg_id hash), then a follow-up commit removes the wiki write path. Cross-process safety comes from `FileChannel.tryLock()` on the jsonl during append (Letta uses Postgres row locks; you don't have that luxury, so file-level advisory lock is the JVM-on-Windows equivalent).
C) Explicit consent + redaction layer + visible "memory pinned" indicator. OpenAI's ChatGPT Memory shows a toast every time it writes/reads a memory and exposes a manage-memories panel. Letta requires explicit `core_memory_append` tool calls — the model decides what to pin, not the harness. Mem0 has a `memory.add()` API that's user-invoked. The dangerous pattern is auto-importing personal facts you didn't curate; ship core_memory.md as a *manually-edited* file in v1 (you write it, you commit it), with NO auto-extraction. Auto-extraction (the Mem0/MemGPT "self-edit" pattern) is a v2+ feature behind an opt-in flag.
D) Flat role/content/ts/msg_id NOW with a nullable `kind` field defaulting to "chat" added day 1. OpenAI Threads schema has `role` + `content` (typed array) + `metadata` (free-form dict) + `run_id`/`assistant_id`. Anthropic Messages API is `role` + `content` (string or block array). The forward-compatible move is: add `kind: String = "chat"` and `metadata: Map<String,String>? = null` columns from commit 1, even if unused — JSONL append-only means schema additions are free, but retrofitting a discriminator across 1000 historical lines is annoying. This is exactly Letta's evolution: `MessageType` enum (`user`/`assistant`/`tool_call`/`tool_return`/`system`) was added in their v0.3 refactor and they regretted not having it day 1.
E) `/api/search` returns ranked chunks with source pointers (file path + line range + score), no LLM in the loop — Khoj's `/api/search` is exactly this, returns `entries[]` with `additional.file`, `score`, `corpus_id`. The "search" *subsystem* is the chat-side tool that calls `/api/search`, formats results, and lets the model decide what to quote. Keep them separate: `/api/search` is a deterministic retrieval endpoint usable from CLI/curl/Android directly; the chat tool is a thin wrapper. Khoj enforces this split and it's why their search works standalone. Critical: do NOT auto-inject search results into chat context — the proposal already says this, hold the line.
F) Do NOT ship: (1) auto-extraction of facts into core_memory (Mem0/MemGPT pattern, v2+), (2) reflection/summarization passes (Letta's `summarizer` agent — wait until conversations.jsonl actually hits the 1000 cap and you feel the pain), (3) multi-agent or sub-task message kinds (build the discriminator, leave the values empty), (4) hybrid retrieval / reranking (ship dense-only, add FTS5+RRF when you measure recall problems), (5) cross-conversation threads / sessions (Letta added `agent_id` partitioning in v0.4 — single-user single-agent doesn't need it), (6) memory editing UI (OpenAI shipped this 18 months after launch).

WHAT NOT TO SHIP IN v1:
- Auto fact-extraction into core_memory.md (manual curation only)
- Conversation summarization / reflection passes
- Hybrid lexical+dense retrieval (dense-only, FTS5 deferred)
- Reranker (cross-encoder, Cohere rerank, etc.)
- Multi-agent / sub-task message kinds populated (schema slot only)
- Session/thread partitioning
- Memory management UI on Android client
- Auto-injection of archival results into chat context
- Embedding model swap / upgrade path (lock to text-embedding-3-small, deal with migration when you change)
```

## 🧱 First Principles

```
AGENT: First Principles
STANCE: CONDITIONAL
REASONING: Strip the framing: the raw problem is "one human writes things, one LLM reads them, the LLM forgets between turns." Everything else — JSONL, FTS5, core_memory.md, "subsystems," "API surface" — is scaffolding around that. From scratch with no prior art, you'd build exactly two things: an append-only log of what happened (chat + activity, undifferentiated, time-ordered), and a way for the LLM to look at older parts of that log when it wants to. The proposal is mostly aligned with that, but it's inventing four storage shapes (JSONL, FTS index, core_memory, activity log) where the underlying reality is one shape (events in time) with two access patterns (recent, and "find me older stuff like X"). The 988-file disaster wasn't a "split needed" problem — it was a "you mixed two conceptually different things into one ordered stream" problem, and the proposal correctly separates them but then over-separates.

KEY CONCERN: The framing treats "core memory" as a third storage tier. It isn't. The system prompt IS the LLM's only persistent memory across turns — there is no other. Calling a 4KB file "core_memory" and prepending it to the system prompt is just "you have a system prompt and you edit it sometimes." Naming it a tier invents a category that doesn't exist in the substrate.

RECOMMENDATIONS:
A) Wrong framing because "316 markdown files" isn't a corpus, it's a backlog of stuff the user wrote down once and mostly won't reread. The from-scratch answer: don't index anything yet. Let the LLM grep. A single ripgrep call over 316 files is sub-100ms on a 7.8GB VPS and the LLM can issue it as a tool call when it has a reason to. Build FTS5 the day grep gets too slow, which on this dataset is "never." Embeddings are wrong because the retrieval need is "find the note where I wrote X," not "find semantically similar concepts" — the user is the author, they remember their own keywords.
B) Wrong framing because migration is a sunk-cost question dressed up as an engineering question. The wiki.md ingest was the bug. Starting fresh isn't "migration ordering," it's "delete the bad thing." Conversations.jsonl starts empty on day 1. The 316 archival files already exist on disk independent of wiki.md — they don't need migrating, they need to stop being auto-injected.
C) Wrong framing because "≤4 KB pinned" is the wrong shape. The right shape is: the system prompt is a file, the LLM occasionally proposes edits to it, the human approves. No size limit beyond what fits in the context window comfortably. Don't call it "core_memory.md" — call it "system_prompt.md" because that's literally what it is. The 4KB cap is cargo-culted from systems with hundreds of users; you have one user.
D) Wrong framing because futureproofing a single-user system is anti-work. The constraint that matters is "can I rewrite this in an afternoon when I learn what I actually need?" Append-only JSONL passes that test. Anything with a schema migration story fails it. Pick the format that loses the least information (full JSON event objects with a `type` field, no normalization) and stop there.
E) Tool the LLM calls. Retrieval as a separate API surface is a layer that exists to serve UIs, and there is no UI here other than the LLM itself. Give the LLM `search(query)` and `read_file(path)` as tools, let it decide. The "/api/search" endpoint is ceremony — the consumer of search results is the same process that's running the LLM.
F) Load-bearing: append-only log of conversation turns, append-only log of activity, system prompt as an editable file, a way for the LLM to look at older entries on demand. Ceremony: subsystem boundaries, FIFO 1000 cap (just keep everything — text is cheap, 7.8GB holds millions of turns), separate "search subsystem," API surface, "schema futureproofing," distinguishing FTS5 vs embeddings before you've measured grep.

WHAT NOT TO SHIP IN v1:
- Any index (FTS5 or embeddings) — ship grep-as-tool, measure, then decide
- FIFO 1000 cap on conversations.jsonl — keep everything, it's a single user
- A separate /api/search HTTP surface — make it a tool call
- The name "core_memory.md" — call it system_prompt.md
- Any migration of wiki.md content — leave wiki.md on disk as a dead artifact, ignore it in retrieval
- A "search subsystem" abstraction — it's a function, not a subsystem
- 4KB cap on the pinned file — pick a number when you hit the actual problem
- Distinct storage formats for chat vs activity — both are just timestamped events with a `kind` field
```

## ⚠️ Risk Analyst

```
AGENT: Risk Analyst
STANCE: CONDITIONAL
REASONING: Two concrete failure modes dominate. (1) The cross-process migrator is a corruption vector — mutex is in-JVM only, so concurrent appends from the live server during migration race on byte boundaries in conversations.jsonl. We've already had ONE data incident (988-file collapse, saved by .bak); a half-written JSONL line during FIFO rewrite is the second. (2) core_memory.md prepended to every chat call ships matricol + ADHD self-report + finals dates to OpenRouter/Copilot CLI on every turn, and provider TOS on training/logging is opaque — this is a one-way leak, no rollback. FTS5 native-lib brittleness and JSON vector-store unbounded growth are real but recoverable; the migration race and the privacy ratchet are not.
KEY CONCERN: Cross-process write race on conversations.jsonl during migration — CRITICAL. In-process mutex provides ZERO guarantee against a separately-launched migrator JVM, and the failure mode is silent file corruption that retrieval will eat without complaining.

RECOMMENDATIONS:
A) FTS5. Loser=embeddings, severity HIGH for v1: OpenRouter outage takes search offline (and you already depend on it for chat), API-key rotation invalidates cached vectors silently if you don't version them, and JSON-on-disk vector store grew you to the 988-file collapse last time — 316 files of ~10KB chunks at 1536-dim float32 ≈ 18+ MB JSON loaded fully in-memory on a 512m heap, every restart. FTS5 risks (sqlite-jdbc native unpack to /tmp on Ubuntu, noexec /tmp, file-locking on shared volumes) are deterministic and fail loud at startup, not silent at query time. Mitigation for FTS5: pin sqlite-jdbc version, set org.sqlite.tmpdir to a known-writable path inside the app dir, single-writer connection, WAL mode off (single-process anyway).
B) CRITICAL. Do NOT run migrator as separate JVM against a live server. Two safe options: (1) migrator runs IN-PROCESS as a one-shot startup task behind the same mutex, server doesn't accept /api/* until it returns; (2) take the server down for the ~seconds-to-minutes the migration takes. The "separate JVM + in-process mutex" combination is the worst of both worlds — looks safe, isn't. Mitigation if you insist on out-of-process: OS-level advisory lock (FileChannel.tryLock) on a sentinel file that BOTH the server and migrator acquire, plus copy-then-rename writes (write to .tmp, fsync, rename), plus a pre-flight backup of wiki.md AND any pre-existing conversations.jsonl.
C) HIGH and irreversible. Pick: do NOT put matricol or finals dates in core_memory. ADHD self-report is borderline — it's a permanent training-set contribution if any provider in the chain logs. Leak surface: OpenRouter request logs, Copilot CLI telemetry, any future provider you swap in inheriting the same prompt. Mitigation: core_memory holds preferences/style/anti-patterns ("English default", "no plan-tweaking sidetracks", "council pattern"), NOT identifiers or medical/academic specifics. Identifiers/dates belong in a separate gated context that's only injected when the user explicitly asks something that needs them, and is logged client-side so you can audit what left the box.
D) MEDIUM. Add `kind` (or `type`/`v`) field from day 1 — JSONL is append-only by design, so retrofitting means either a one-shot rewrite (which re-introduces the migration race from B) or a forever-branched parser. The cost of adding it now is one field; the cost of not having it is every future feature pays a tax. Also add a schema version int at the top of the file or per-line — without it, a future format change forces another migration.
E) MEDIUM, payload-bounded. /api/search returns hits with: path, score, snippet (≤512 chars window around match), mtime. NOT full content. Cap response at top-K (default 10, max 50). Separate /api/read?path=... endpoint for full content, with a path allowlist (must resolve under archival root, no traversal). This avoids both the "100KB JSON response" failure and the "agent does 50 read-file calls" context-bloat failure — caller decides what to expand.
F) Top unaccounted hazards:
   1. FIFO 1000 cap implementation — naive "rewrite whole file when over 1000" is O(n) per overflow append AND a corruption window if interrupted. Use a rolling segment file or in-place truncate-from-head, NOT full rewrite. CRITICAL if not designed.
   2. mtime-based incremental indexing breaks on filesystem operations that preserve mtime (cp -p, git checkout, rsync without -t inverted). Add (path, size, sha256-of-first-1KB) tuple as the dedup key, not mtime alone. MEDIUM.
   3. No backpressure on PC logger — if /api/activity blocks (mutex contention during migration or FTS index rebuild), the logger retries every 5min and you get a thundering herd of duplicate 5-min-old events. MEDIUM. Logger should send last-event-id; server idempotent on it.
   4. JVM -Xmx512m + VectorStore in-memory + SQLite page cache + Ktor + 316 .md → if you keep BOTH FTS5 and the existing VectorStore loaded "for now," you OOM. Decide one, kill the other in the same commit. HIGH.
   5. No dry-run mode for the migrator — first run against real wiki.md is the test run. Add --dry-run that writes to conversations.jsonl.preview and reports counts/diffs. MEDIUM.

WHAT NOT TO SHIP IN v1:
- Out-of-process migrator JVM (CRITICAL — fold into in-process startup task or take server down)
- Identifiers in core_memory.md (matricol, finals dates, course codes) — preferences only
- Embeddings/VectorStore for archival search — pick FTS5, retire the JSON vector store in the same PR
- Full-content /api/search responses — snippets + paths only
- Naive "rewrite whole JSONL on overflow" FIFO — needs rolling/truncate design
- Schema without `kind` + version field
- Migrator without a pre-flight wiki.md + conversations.jsonl backup and a --dry-run mode
```
