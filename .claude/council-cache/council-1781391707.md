KEY CONCERN: The plan has NO machine-enforced quality gate — it offloads "lectie-grade polish" onto orchestrator self-discipline, which is the same unenforceable soft-gate failure mode G4 exists to eliminate. CONDITION to lift to viable: define a family-level visual-fidelity gate (pinned reference screenshot per family with a perceptual/pixel-diff threshold, plus a token/theme conformance check on the rendered SVG) that fails CI the same way trace-match does — so polish is verified-once-per-family exactly as correctness is, not trusted-per-run.

## 📚 Domain Expert
AGENT: Domain Expert
STANCE: APPROVE
REASONING: This matches the dominant industry pattern: constrained generation + verification beats bespoke output at scale, and the quality gap is recovered through tokens/theming/motion layered on a verified primitive set — exactly how shadcn/ui, Material 3, and Radix+Tailwind ship polished UIs without hand-coding each screen. The selection-sort figure looking "crude" is the well-known failure mode of un-themed templates: Vega-Lite/D3 charts also look generic until you apply a design-token theme, a typography scale, and easing curves (D3's default ordinal palette and linear tweens vs. a tuned d3.easeCubic/Observable Plot theme are the canonical example). Reopening G4 to hand-draw would be the classic regression every mature team rejects — it's why charting libraries moved to grammar-of-graphics (Wilkinson) and why game studios use engines with art-direction layers rather than per-level bespoke renderers: you fix the theme and motion system, not the generation model. The root cause you named — no pinned visual reference + first-gate-pass acceptance under token pressure — is a process gap (missing a visual-regression baseline), and the fix is the standard remedy: a golden-master/visual-regression snapshot (Storybook + Chromatic / Playwright toHaveScreenshot) and a shared design-token + motion-token layer, not a new architecture.
KEY CONCERN: Your gates verify correctness and no-clip but not aesthetic quality — the missing primitive is a pinned visual baseline. Without a per-family golden-master screenshot reviewed against the lectie.html bar (Chromatic-style approval or Playwright toHaveScreenshot with a curated reference), "polish the render" stays subjective taste that will regress again the moment token-frugality pressure returns; promote the visual reference to a machine-enforced gate, the same status as no-clip and trace-match.

## 🧱 First Principles
AGENT: First Principles
STANCE: CONDITIONAL
REASONING: The actual goal is "the figure makes the concept click and isn't misleading" — which decomposes into two orthogonal axes that G4 conflates: semantic correctness (the reflection isn't a 2.39x stretch) and perceptual quality (it's beautiful and legible). The G4 failure was a correctness failure, but "family templates vs bespoke" is the wrong axis to fix it on — a bespoke figure can be correct and a family template can be wrong; what actually caught nothing was that the gates checked lint/CSS but never checked the geometry, so the real missing primitive is a correctness oracle (assert the rendered drawing preserves the invariants the math claims), not a ban on hand-coding. Templates are a fine engineering convenience for amortizing that verification, but they're being asked to carry a guarantee they don't structurally provide; meanwhile visual quality is a third, separate gate that no template polish will systematically enforce either. From scratch I'd build two perceptual/semantic gates (invariant-assertion correctness check + a perceptual quality bar, both machine-or-render-checked) and treat "family vs bespoke" as a free implementation choice underneath them — so I approve "recover quality by polishing + themed surface" but reject the implicit claim that keeping G4 locked is what guarantees correctness.
KEY CONCERN: G4 mistakes a verification convenience (templates amortize review) for a verification guarantee — the 2.39x-stretch bug would pass a verified-once family just as silently unless there's an explicit geometric-invariant oracle, so the locked decision is treating the wrong variable as the safety mechanism and will give false confidence while the actual correctness gate is still missing.

## ⚠️ Risk Analyst
AGENT: Risk Analyst
STANCE: CONDITIONAL
REASONING: The core risk calculus favors the approach: a beautiful figure that lies has unbounded, silent blast radius — it teaches a wrong mental model the student carries into the exam, the exact 2.39x-stretch failure that motivated G4, undetectable because it passed every gate. An ugly-but-correct figure's failure mode is disengagement, which is bounded, observable, and recoverable (the student can supplement, re-read, or push through). For a finals student, internalizing a falsehood is catastrophic and one-way; mild disengagement is degrading-but-survivable. So keeping G4 locked is the correct risk posture — BUT only if the "polish the family render" path is actually funded and gated, because a generic family render that nobody invests taste into converts a manageable risk (disengagement) into a real one (student abandons the tool entirely and reverts to the ask-AI-read-backwards loop, losing the correctness guarantee anyway by routing around it).
KEY CONCERN: [HIGH] Disengagement-induced abandonment is the load-bearing risk, not figure aesthetics per se. If "recover lectie-grade quality via family polish" stays a stated intention without a concrete acceptance gate (themed surface + interaction/gating parity with lectie.html, eyeballed on a real render), the family figures stay crude, the student routes around the verified tool, and G4's correctness guarantee protects nothing because it guards a surface no one uses. Mitigation: make lectie-parity a blocking visual/interaction acceptance criterion on the family template itself — one verified-once polish pass amortized across all instances — so the correctness guarantee and the engagement bar are satisfied by the same locked template, not traded against each other.
