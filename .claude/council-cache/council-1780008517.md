# Council review — 1780008517

**Problem:** Is the proposed CONCRETE PDF-extraction stack the right tool choice for jarvis's material (Romanian STEM lecture PDFs + past exams) + constraints ($0, solo dev, Windows, offline batch)?

**Proposed approach (concrete stack):** (1) Backbone = Marker primary, MinerU fallback. (2) RO diacritic fixup regex. (3) Vision/semantic pass = Claude via Max sub reading page images + Marker output (diagrams, hard math, concept-shape classification, KC drafting). (4) Grounding = jarvis's existing verbatim-source validator. (5) Diagrams routed to a real viz component.

**Project context:** Romanian university STEM (algorithms/proofs/pseudocode, linear algebra, probability, C++ OOP, OS/networks). $0 budget (open-source local + Claude Max only). Solo dev, Windows, offline batch authoring. Timing out of scope.

**Timestamp:** 2026-05-29

## 🔴 Devil's Advocate
STANCE: CONDITIONAL
REASONING: Marker on Windows-CPU is the load-bearing risk — it ships PyTorch + Surya models; no-GPU = multi-minute/page + real CUDA/torch wheel + Surya install friction. Worse, a silent-failure the grounding won't catch: the verbatim-substring validator proves a claim's text exists in Marker's output, but if Marker drops/garbles a column, boxed formula, or table (known weak spot on dense slide decks + LaTeX-heavy ALO/PS/PA), the source text is wrong-but-self-consistent — every concept validates against a corrupted substring, nothing flags it. The RO diacritic regex is under-scoped: cedilla-vs-comma is the easy 10%; the real damage is ligature/hyphenation breakage ("algorit‐mul", collapsed spaces) that silently defeats substring grounding. "Marker primary, MinerU fallback" is backwards — you can't know extraction is "poor" without a per-subject fidelity gate, so the fallback never triggers on the failures that matter.
KEY CONCERN: Garbage-but-consistent extraction passes verbatim grounding undetected. The vision pass MUST diff extracted-text vs the page image and reject/re-route pages where math/tables/columns don't match — not just "describe diagrams" — or the $0 stack authors KCs from silently corrupted source on exactly the math-dense subjects where it's weakest.

## 📚 Domain Expert
STANCE: CONDITIONAL
REASONING: Shape is right, but the primary pick is wrong for these constraints. Marker (Surya-based) has the best OSS layout reading-order + dense prose, but its math is weaker than MinerU's, Surya's models are license-gated (free only under a revenue threshold), and it's a fragile Windows install (Torch + CUDA/flash-attn). MinerU 2.5 (1.2B VLM) leads OmniDocBench on formula + table fidelity, is right for the algebra/probability/algorithms math, and is ALREADY the standardized backbone in the repo's own spec (§7.6, line 17, the 8-stage curate-tutor pipeline at line 882) — so Marker-primary contradicts the design + the RAG path. Docling (IBM) is the genuinely better Windows-first default (pip-clean, strong tables, code-aware for C++) and belongs as fallback over Marker. Romanian is a non-issue at the OCR layer (Latin script); the diacritic regex is correct standard cleanup. The Claude-vision pass is COMPLEMENTARY, not redundant: extractors emit LaTeX/Markdown but can't verify a recurrence is mathematically correct, classify a diagram into your viz components, or recover dropped slide diagrams — that's the vision-LLM's job, paired with the verbatim validator.
KEY CONCERN: Primary/fallback is backwards AND contradicts the repo's settled backbone — make MinerU 2.5 primary (math fidelity + already wired to the RAG path + spec), Docling the Windows-clean fallback, Marker optional-tertiary; Marker-primary risks a multi-day Torch/Surya Windows fight to land a tool with weaker math than the one already chosen.

## 🧱 First Principles
STANCE: CONDITIONAL
REASONING: From the raw goal (faithful, math-correct, verbatim-groundable text, ~5 subjects, one-time, solo, $0), the simplest correct pipeline is Claude-vision reading page images → markdown+LaTeX + concept classification, then the verbatim validator. The local ML extractor is justified ONLY as a cheap deterministic baseline the validator grounds AGAINST — because that's the load-bearing problem: a vision-LLM can silently hallucinate a plausible-but-wrong equation, and "verbatim-grounded" requires a NON-LLM source of truth to ground TO. If you ground Claude's output against Claude's own re-reading of the image, you have no independent anchor — a model checking itself. So the shape is inverted from the proposal: Marker isn't the "backbone" doing the understanding (Claude is); it's the dumb, reproducible text layer that makes the grounding step MEAN something.
KEY CONCERN: The grounding source of truth. If "verbatim-checked" compares Claude-output against Claude-vision-output, the validator is theater — the extractor earns its place precisely (and only) as the independent text the claims are checked against. If the PDFs have a real text layer, plain pdftotext/PyMuPDF may fill that role and make even Marker redundant.

Detection must come from extractor-vs-extractor (or extractor-vs-vision) DISAGREEMENT on math/code spans → human review; the verbatim check is structurally incapable of catching it.
