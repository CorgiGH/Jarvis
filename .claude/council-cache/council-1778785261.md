# Council review — 1778785261

**Problem:** Surface Y student stand-in — a Playwright + LLM-persona harness meant to drive a naive-student simulation through the tutor's drill workspace to (a) surface UX friction and (b) produce real synthetic `tutor_events`. Acceptance criterion: the persona emits a `submit` action → controller clicks CHECK ANSWER → a real `drill_grade` event lands in the backend log. Across ~10 runs and 3 `:free` models this session (gpt-oss-120b, minimax-m2.5, deepseek-v4-flash), 3 distinct failure modes, 0 submits, 0 completed drills.

**Proposed approach:** A 5-fork menu — (A) accept Surface Y V1 as-is + V2-ticket the drill-completion; (B) wire Groq as a provider sibling; (C) controller-side force-submit scaffolding; (D) re-architect the persona loop; (E) keep model-roulette. Council asked which fork is right, or whether the framing itself is wrong.

**Project context:**
- Domain: Personal AI tutor + life-OS web app ("jarvis-kotlin"). Single user (Alex, university AI-bachelor's). Surface X/Y/Z is an automated quality/UX-eval suite.
- Stack: Kotlin backend (Exposed, SQLite), React/TS frontend (Vite), Node ESM tooling (`tools/*.mjs`), Playwright, OpenRouter `:free`-tier LLMs.
- Constraints: No paid APIs (`:free` only, ~50/day shared pool; Groq `:free` ok). Single-maintainer. Tutor UI brutalist-mono, verified working, not up for redesign. Work lands on `main`. Standing rule: no continued prompt-tuning past a confirmed model-capability wall.
- Prior decisions: Surface X/Y/Z shipped. Prior council rejected an earlier "A/B/C" framing as a false trichotomy → "instrument + decouple + fix-as-architecture". A second prior council APPROVED the semantic `submit` action (shipped, 63/63 unit tests).

## 🔴 Devil's Advocate

AGENT: Devil's Advocate
STANCE: REJECT
REASONING: All five forks share an unexamined load-bearing assumption: that a real `drill_grade` event must be produced *by an unsupervised free-tier LLM persona*. It does not. The event schema already carries an `is_synthetic` flag, `event-log-reader.mjs` already filters on it, and a `bootstrap-traces` golden file already exists — the project has *already built* the decoupled path and is now spending councils trying to make the expensive, unreliable path work anyway. Three free models failing three different ways at a multi-step plan-and-act loop is not a prompt problem or a provider problem; it is the known free-tier capability floor (the standing rule literally says stop tuning once a capability wall is confirmed — it is confirmed), so B/D/E are relitigating a closed question and C destroys the only thing Surface Y's persona uniquely provides (naive-student authenticity) the moment the controller force-submits. Fork A is closest to honest but still mis-files the problem as a "V2 ticket" instead of admitting the acceptance criterion was mis-specified from the start.
KEY CONCERN: The X-fixture's hard dependency is on *real `tutor_events` existing in the log*, not on *Surface Y's persona having authored them*. Conflating those two is the actual error — Alex can drive one genuine drill manually (or via a deterministic non-persona Playwright script) in five minutes, land a real `drill_grade`, unblock Surface X today, and let Surface Y's persona be honestly scoped to UX-friction discovery where free models actually succeed. Picking any of A–E without first severing that conflation locks the whole pipeline behind the single least-reliable component in the system.

## 📚 Domain Expert

AGENT: Domain Expert
STANCE: CONDITIONAL
REASONING: The established pattern here is the **scripted-skeleton-with-LLM-judgment hybrid** (Sierra's TAU-bench user simulator, web-bench/WebArena oracle harnesses, and every synthetic-user QA framework worth citing): the *control flow* of a known happy-path task is deterministic; the LLM is consulted only at genuine decision/perception points. Surface Y already half-discovered this — the prior council's `submit` action *is* a hybrid macro-action, and it was the right move — but run 1 proves the diagnosis was incomplete: gpt-oss-120b saw CHECK ANSWER enabled, saw `[current value:...]`, was prompt-told to emit `submit`, and still chose `type`. That is the confirmed capability floor the spec itself flagged at line 78, and the standing rule says stop prompt-tuning past it. So **fork D (re-architect the persona loop) is wrong** (ReAct/scratchpad scaffolding doesn't fix a model that won't fire a terminal action — browser-use and WebVoyager papers show terminal-action reliability is a model-scale property, not a prompt property), **fork E (model roulette) is wrong** (3/3 distinct failures is a pattern, not bad luck), and **fork C as written is wrong** (a controller that force-submits silently destroys the naive-student signal). The correct path is **C reframed as the hybrid-harness standard, plus B as the capability hedge, plus unblocking the downstream dependency independently of A**: (1) make the controller's terminal step an *event-driven macro* — when the code field is non-empty AND CHECK ANSWER is enabled for N consecutive steps, the controller emits an explicit `submit` and **logs it as `executor: controller-forced` in the transcript** so the naive-student friction finding ("persona never recognized it was done") is *preserved as data*, not erased — this is exactly how TAU-bench separates env-driven transitions from agent decisions; (2) wire Groq (fork B) because gpt-oss-120b on OpenRouter `:free` is heavily quantized and Groq's copy genuinely may clear the bar — cheap to try, durable quota hedge regardless; (3) **do not let the X-fixture stay blocked on Surface Y** — real `tutor_events` for the golden fixture should be generated by a thin deterministic Playwright script (no LLM) that drives one known-good drill to `drill_grade`, which is a 30-minute job and is the *standard* way every team seeds fixtures (you script the oracle path; you don't make a stochastic agent your fixture factory).
KEY CONCERN: The acceptance criterion conflates two things that established practice keeps separate — "surface UX friction" (legitimately needs a stochastic LLM persona) and "produce a real `drill_grade` event" (a fixture-seeding task that should never depend on a stochastic agent at all). Until those are decoupled, every `:free`-model failure looks like it blocks downstream work when it shouldn't; fix the coupling first and the "is `:free` Surface Y viable?" question stops being load-bearing.

Forks C, D, and E are all disproportionate: C corrupts the naive-student premise the tool exists to test (a controller that force-submits is no longer measuring what a student does), D is a multi-day persona-loop rewrite betting big on a single-user side-tool, and E is just more roulette after 3 models already burned. The juice is worth one squeeze, not five.
KEY CONCERN: If Groq's gpt-oss-120b behaves identically to the OpenRouter copy (likely — same base weights, the quantization delta is a thin hope), then the acceptance criterion "persona completes a drill on :free models" is empirically dead, and the council must not let B quietly become an open-ended model hunt. Set the stop condition NOW: B is one fast model attempt; if it fails, the X-fixture gets ONE real `tutor_event` via a manual or controller-assisted seed run (explicitly labeled non-persona, fixture-only — not folded into Surface Y's UX claims), and "persona drives drill to submit" ships as the V2 ticket. That keeps the standing no-prompt-tuning-past-a-wall rule intact and unblocks downstream without pretending the wall isn't there.

## 🧱 First Principles

STANCE: REJECT
REASONING: The framing fuses two goals that have no shared mechanism. Goal (a) — surface UX friction — is genuinely served by an LLM driving the real DOM, and the 3 runs already produced that signal (a model that types-but-won't-submit, a slow one, an incoherent one are themselves UX-adjacent persona findings; the harness is sound). You are gating a deterministic data-generation task behind a stochastic free-tier LLM clearing a reasoning bar 3 models have now failed — that is solving the wrong problem. From scratch: keep Surface Y's persona loop for friction discovery (fork A's reframe), and generate the X-fixture's `drill_grade` events with a tiny deterministic seeder that POSTs known-good and known-bad attempts directly — that is the right primitive for "synthetic tutor_events," and it unblocks downstream X work today regardless of whether any `:free` model ever makes the type→submit jump. Forks B/C/D/E all chase the false premise that the persona must be the event source; C ("controller force-submits") even concedes the persona isn't doing the work, so just remove it from that path entirely.
KEY CONCERN: A completed-by-persona drill and a synthetic `drill_grade` event were wrongly welded into one acceptance criterion — decouple them, and the only hard blocker (downstream X-fixture) dissolves without depending on free-model capability at all.

## ⚠️ Risk Analyst

AGENT: Risk Analyst
STANCE: CONDITIONAL
REASONING: Data-poisoning risk is MEDIUM, not critical — verified `is_synthetic=true` tagging (TutorRoutes.kt:1602/1686) plus `filterEvents` defaulting `include_synthetic=false` means force-submit (fork C) events are *partitioned*, not poisoned; they only reach the X-fixture through deliberate hand-labelling, and a controller-driven `drill_grade` is still a real backend event with a real grader verdict — fine as a fixture trace, just not as UX-friction signal. The CRITICAL risk is the dependency-chain framing: fork A ("V2 ticket it") leaves X-fixture-fattening permanently blocked on a bottleneck — free-tier persona action-selection — that has no credible near-term fix, so "blocked" silently means "abandoned." The HIGH risk is treating free-tier as an always-on foundation: 3/3 models failed differently, the `:free` pool is a shared ~50/day quota you don't control, and minimax was knocked out by an arbitrary hardcoded `maxDurationMin=10` — that timeout is a config bug masquerading as a capability wall and must be excluded before any "free-tier is walled" conclusion is logged under the no-continued-tuning rule. Correct path is C+B together, scoped honestly: fork C (controller force-submit) is the *only* fork that unblocks the X-fixture this week, and its naive-student-simulation cost is real but bounded — gate it behind an explicit `--assist-submit` flag and tag those runs so the transcript distinguishes "persona reached submit" from "controller rescued submit," preserving the friction signal for the runs that earn it; run fork B in parallel purely as a quota hedge with zero expectation that Groq's gpt-oss-120b copy clears the wall.
KEY CONCERN: Fork A is a disguised abandonment of the X-fixture dependency — and before anyone concludes ":free is walled," re-run minimax with the 10-minute cap lifted, because a hardcoded timeout knocking out the one capable model is an infrastructure failure, not a model-capability wall, and logging it as the latter permanently forecloses the cheapest viable path under the standing no-retuning rule.
