## 🔴 Devil's Advocate

AGENT: Devil's Advocate

   min 20-min cooldown, max 12 wakeups/day cap. NO cron.

   (b) no signal in past 2h, (c) NOT quiet-hours, (d) THEN spend LLM.

   after success rewrite-or-finalize with status=emitted. Crash mid-LLM

   "Defer to 2.2" during finals = the failure mode.

   3 consecutive failures → loop self-disables 6h, writes circuit_open.

- Two-phase write (pending→emitted) with input-hash dedup

## SYNTHESIS / VERDICT

**STATUS: CONDITIONAL — ship the design below. Default-OFF behind an env flag (`PROACTIVE_LOOP_ENABLED=true` to enable) so the user authorizes activation explicitly on first deploy.**

### Convergences
- Q1: All 3 reject pure 30-min cron. Devil + Pragmatist say event-triggered (on activity write). Domain says hybrid (poll + threshold). **Pick: event-triggered on /api/activity write.** Zero idle cost; matches the 2-of-3 vote and Pragmatist's "no consumer yet" cliff.
- Q4: Devil + Pragmatist + Domain all say SINGLE subsystem per tick. Pragmatist + Devil say ctx-model only for v1. **Pick: ctx-model only.**
- Q7: All 3 say swallow failures, never crash loop. Devil + Pragmatist say subprocess timeout 60-90s (NOT 300s default). **Pick: withTimeout(60_000), Dispatchers.IO.limitedParallelism(1), error row to signals.jsonl on timeout/exception.**

### Decisions where stances split
- Q5 restart safety: Domain's deterministic hash ID (sha256(triggerActivityTs + ts_bucket)) wins on cleanest+simplest. Devil's two-phase pending→emitted is over-engineered for v1. Pragmatist's "nothing" is too thin. **Pick: deterministic hash ID + append-once + dedup-on-read.**
- Q6 quiet hours: Devil + Pragmatist HARDCODE for finals; Domain DEFER. 2-1 toward HARDCODE. **Pick: hardcode 23:00–07:00 Europe/Bucharest now.**
- Q3 storage: Pragmatist's minimal {ts, kind, importance, snippet, rationale, id} fits v1 best. Add `status` field for error rows. **Pick: minimal schema with id + status.**

### Pragmatist's load-bearing observation
> "Without a Phase 3 push channel, Phase 2.1 is instrumentation."

Honored by **default-off env flag** + minimal scope. Ship Phase 2.1 as a dry-run logger this session. Defer Phase 2.2 + 2.3 logic. Phase 3 next session re-evaluates whether a push channel before more loop polish is the right order.

### Final ProactiveSignal schema
```kotlin
@Serializable
data class ProactiveSignal(
    val id: String,            // sha256(triggerActivityTs + hourBucket)
    val ts: String,            // emit time
    val kind: String,          // "ctx_model_summary" | "error"
    val importance: Float,     // triggering importance
    val sourceTs: String,      // activity row that triggered
    val snippet: String,       // ctx-model output capped 200ch
    val rationale: String,     // why this fired (e.g. "imp=0.85 + 30min cooldown elapsed")
    val status: String = "emitted",
    val v: Int = 1,
)
```

### Hook point
Server-side only. `WebMain.kt /api/activity` handler — after `Activity.append(entry)` succeeds, fire `ProactiveLoop.consider(entry)` on the dedicated dispatcher. Don't touch `Activity.append` itself (it's used by tests + CLI logger fallback that doesn't get a signal anyway).
