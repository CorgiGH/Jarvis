## 🔴 Devil's Advocate
STANCE: CONDITIONAL
REASONING: The exhaustive-table-mirrored-to-code-and-tests is the correct structural move — but as proposed it hard-codes a false premise: that the unit of decision is (claim-kind × leg-outcome). The latest hole isn't an "undecided cell" — it's an inconsistency in the AGGREGATION FUNCTION that collapses N legs into one outcome (consensus-veto vs any-veto), and that policy is invisible in a flat (kind × outcome) grid. Enumerating over an already-aggregated axis ratifies a table that silently bakes in whichever aggregation each cell's author used; six months later the review still finds a hole — now blessed by a green table-test, which is strictly worse (manufactured false confidence).
KEY CONCERN: The table's axes are wrong/incomplete. Decompose into (a) per-leg verdict, (b) the explicit multi-leg AGGREGATION policy (any-refuted vs consensus, and why verbatim-DEFINITIONs may legitimately differ) as a first-class single-sourced ratified rule — not an emergent property smeared across cells.

## 📚 Domain Expert
STANCE: CONDITIONAL
REASONING: The 3-step plan is the correct backbone (exhaustive decision table + table-driven/parameterized tests = how mature rules engines stay honest). But "code and test mirror the table" is the weakest option — a hand-mirrored `when{}` and hand-mirrored test co-drift (same author copies the same wrong cell into both). Stronger known patterns: (a) single source of truth via codegen — keep the table as DATA the code INDEXES into, not three copies; (b) sealed product input type so the `when` is a TOTAL function with compiler-enforced exhaustiveness (unhandled cell = compile error, not runtime false-faithful); (c) property-based invariants (`any REFUTED ⇒ ¬faithful`, `any threw ⇒ ¬faithful`, `all UNCLEAR ⇒ uncertain`) as a cross-cut net independent of per-cell verdicts.
KEY CONCERN: A hand-mirrored table+test co-drifts; make the ratified table the SINGLE source of truth that `decideOutcome` reads (data-driven dispatch over a sealed/exhaustive space) + REFUTED/threw ⇒ ¬faithful property invariants as an independent floor.

## ⚙️ Pragmatist
STANCE: CONDITIONAL
REASONING: For a solo ADHD dev in bursts, ratifying hundreds of cells by hand is the wrong cost profile — a one-shot attention sink that goes stale on the next claim-kind/leg added. Cheaper path = ~90% value: make DENY-by-default structural (any cell not on the explicit allow-list is failed/uncertain, never faithful), encode 3-4 FAIL-LOUD invariants as property tests, and enumerate only the SMALL allow-list of faithful cells (tens, not hundreds) — that set is worth council eyes. Inverts the work: review only the few things that can say "trust me"; property tests + default-deny mechanically guarantee every unenumerated cell is safe. Resumable: a new burst reads 3 invariants + one short allow-list, not a 400-row matrix.
KEY CONCERN: The allow-list must be the ONLY path to faithful and default-deny enforced STRUCTURALLY (a single chokepoint, not a convention) — else property tests pass while a stray `when{}` branch hand-returns faithful and the hole reopens.

## 🧱 First Principles
STANCE: CONDITIONAL
REASONING: Stripped to the constraint — false-FAITHFUL is the only catastrophe, "uncertain" is always free — the correct shape is provably a CLOSED whitelist: `faithful` default-DENY, emitted ONLY when an explicitly-justified ALLOW pattern fires, single catch-all `else → uncertain`. The whack-a-mole is the signature of an OPEN blacklist (faithful is opt-out): every newly-found cell was silently defaulting to allow. So the giant table treats the symptom (missing cells), not the cause (faithful is opt-in nowhere). Build the inversion FIRST (flip default, add catch-all sink), THEN enumerate only the ALLOW patterns — a tiny finite set you can actually ratify.
KEY CONCERN: A table ratified cell-by-cell with an implicit default is itself unsafe — the catch-all must be STRUCTURAL (type-system or guaranteed default branch), not "row N+1 later"; verify no path reaches faithful without passing an explicit ALLOW predicate.
