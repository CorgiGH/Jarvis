## 🔴 Devil's Advocate

AGENT: Devil's Advocate
STANCE: CONDITIONAL

REASONING: The document's single most dangerous hidden assumption is that `npx impeccable detect` actually works as advertised — 41 deterministic rules, no API key, CI-able, zero false positives on SVG-heavy educational content. This is an unverified external CLI being promoted to a hard CI gate before a single `npx impeccable detect --json tutor-web/src/` has been run against the real codebase. The plan commits to "assert zero failures on its JSON output" but the tool's rules were authored for generic web products, not for animated pedagogical SVG with step-controlled motion and deliberate high-contrast brutalist styling that may look like "bad contrast" or "animation anti-pattern" to rules written against normal UX conventions. If the tool fires false positives on legitimate intentional design choices — and with 41 opinionated rules it will — the team has three bad options: suppress the rules (defeats the gate), hand-tune the DESIGN.md until suppression is baked in (the gate becomes a formality), or block CI on noise (everyone learns to ignore the red). The Playwright no-clip gate is DOM-query-precise and project-authored; Impeccable is a third-party opinion engine being installed in the same circuit without calibration. Additionally, #3's visual-regression baseline commitment (`toHaveScreenshot` committed to `tutor-web/e2e/snapshots/`) is a maintenance trap on a pipeline that is explicitly designed to emit new lesson pages continuously — every new agent-generated lesson requires a new approved baseline, turning the screenshot-diff gate from a safety net into a manual approval queue that recreates the very iteration burden the pipeline is meant to eliminate.

FLIPS:
- **#1 adopt→conditional**: Do NOT wire `impeccable detect` into the CI gate before running it against the real codebase and cataloguing every rule that fires. Instrument first (one `npx impeccable detect --json tutor-web/src/` run, read the output), suppress or configure rules that conflict with intentional design choices, then gate. Promoting an uncalibrated third-party linter to a hard CI assertion on day one is how gates become ignored noise.
- **#3 partial flip on visual baselines**: `toHaveScreenshot` baselines are the wrong tool for a pipeline whose job is to generate novel pages continuously. They are correct for the fixed shell (AlgoStepperShell, theme surfaces, known routes) but must NOT be required for agent-generated lesson content — require structural DOM assertions (no-clip, beat-count, required `data-testid` selectors) there instead. Committing baselines for generated content means every new lesson blocks on a human screenshot approval, which is the manual-iteration failure mode this whole proposal exists to kill.

KEY CONCERN: Impeccable is being installed as a hard CI gate before anyone has confirmed its 41 rules don't conflict with deliberately unconventional brutalist/pedagogical design choices — the tool is an unverified opinion engine in a circuit that is supposed to be machine-objective, and the first false-positive cascade will either corrupt the gate's authority or create a DESIGN.md suppression file that makes the gate a rubber stamp.

## 📚 Domain Expert

AGENT: Domain Expert
STANCE: APPROVE

REASONING: The SVG-only policy for machine-verifiable auto-generated educational viz is the correct and well-established pattern. SVG's DOM-native identity is precisely what makes `getBoundingClientRect()`, accessibility-tree reads, and attribute-level correctness assertions possible — canvas collapses all of that to opaque pixels, which is why Playwright's `toHaveScreenshot` is the ceiling of canvas testability, not the floor. Manim's architecture confirms the lesson: its gold standard is offline-render-to-video (Cairo/OpenGL → MP4), not browser-interactive DOM; when the community ported it to the browser it reached for WebGL, which immediately lost the structured assertability this project's gates depend on. Impeccable's `npx impeccable detect` is verified: 41 deterministic rules, zero API key, JSON output with exit codes — this is exactly the pattern established by stylelint and eslint for CI gates, and it adds the CSS/design-quality layer that those tools deliberately skip. Self-hosting Google Fonts woff2 is unambiguously the 2025 industry standard: the Munich Regional Court GDPR ruling (Jan 2022) settled the legal question, google-webfonts-helper is the established tooling path, woff2 has 96%+ browser support making WOFF fallback unnecessary, and the font-determinism argument for Playwright baselines is the same reason Playwright's own docs recommend `animations: 'disabled'` — you want zero environmental variance in visual regression runs. The Playwright extensions (toHaveScreenshot baselines committed to version control, trace/video retain-on-failure, assertNoOverflow wired into the actual `@playwright/test` suite rather than ad-hoc shoot scripts) all match the current `playwright.dev` documented best practice, with the baseline-in-VCS pattern specifically called out as the standard for team + CI consistency.

FLIPS: Decision 4 (Taste Skill — adopt-later) warrants a harder gate condition than stated. The finding correctly identifies that top-level v2 bans Fraunces and defaults to GSAP, but "after Impeccable's gate is running" is not a sufficient unlock condition. The unlock should require explicitly scoping to only the brutalist variant's SKILL.md and running a diff against the Impeccable DESIGN.md to confirm no token collisions before install — otherwise an agent loading both skill files will receive conflicting motion and font directives in the same context window, and the council-locked system will silently degrade. This is not a flip to REJECT, but the unlock condition needs that specificity.

KEY CONCERN: The `assertNoOverflow` gate is described as wired "into `new-surfaces-smoke.spec.ts` for all 7 routes" — but the finding also notes the current no-clip gate lives only in the ad-hoc `shoot.qrqa2.mjs` script, not in the `@playwright/test` suite. This means the gate the proposal calls "machine-checked" does not currently exist in CI. The Playwright extension items are correctly identified as gaps, but the proposal's own §3 language implies the gate is operational when it is not. The actual risk is that agent-generated lesson pages run against the Impeccable detect gate and the toHaveScreenshot baselines before the DOM-query no-clip gate is live in `@playwright/test`, creating a window where the two established gates pass but the core no-clip invariant — the one the project's design spine treats as definitive — is still only checked manually. Wire `assertNoOverflow` into the `@playwright/test` suite first, before establishing toHaveScreenshot baselines, because a baseline of a clipped layout is worse than no baseline.

## ⚙️ Pragmatist

AGENT: Pragmatist
STANCE: CONDITIONAL

REASONING: The three "adopt now" items have different risk profiles. Self-hosting fonts is unambiguously correct — the playwright.config.ts has no font loading config, the current CDN fetches will silently fall back to system fonts during headless runs, and the fix is a one-time download with zero ongoing maintenance cost. The Playwright extensions (toHaveScreenshot, trace/video retention, no-clip wired into the spec suite) are also clearly net-positive — the playwright.config.ts currently has a bare `use:` block with no trace/video config, the assertNoOverflow claim is in ad-hoc shoot.qrqa2.mjs not in the spec suite, and the vitest-axe fix is a single import line. The problematic item is Impeccable: it adds a THIRD lint layer (eslint + Playwright assertions + now Impeccable) with a DESIGN.md that must be kept in sync with index.css tokens — two sources of truth for the same constraints, with no automated sync mechanism. For a one-person autonomous pipeline, the gate failure mode matters: Impeccable blocking a generation run because it disagrees with a local CSS override will be harder to triage than a Playwright geometry failure, and "41 deterministic rules" from an external tool will include rules that don't apply to this design system, producing false positives that the pipeline has no way to self-resolve without human intervention. The adopt-later items (Taste brutalist variant, VoltAgent, Google Stitch) are scope noise in their current form — they lack concrete unlock conditions that are tied to actual roadmap milestones, and experience with adopt-later queues is that they accumulate. The SVG-only policy is correct and its maintenance cost is bounded; the one genuine hidden cost is that it forces agents to write SVG particle simulations or fluid visualizations that would be cleaner as canvas, but looking at the actual viz inventory (tree/graph layout, matrix trace, geometric projection) there is no concept in the queue that genuinely requires this, so the policy cost is currently zero.

FLIPS: (1) Impeccable: CONDITIONAL rather than adopt-now — adopt only if DESIGN.md is auto-generated from index.css tokens (not hand-maintained) and only the contrast/spacing rules subset is enabled (not all 41 rules); otherwise the two-source-of-truth drift and false-positive rate in an autonomous pipeline outweigh the gate value. (4/5/6) Adopt-later items should each have a concrete named unlock event tied to a specific roadmap milestone (e.g., "Taste: after G4 viz fork is resolved" not "after Impeccable is running") or be cut from the list entirely to avoid scope-noise accumulation.

KEY CONCERN: Screenshot baselines (toHaveScreenshot) introduce per-machine anti-aliasing and subpixel rendering variance that is not neutralized by the existing playwright.config.ts setup — there is no `--update-snapshots` workflow defined, no CI machine spec, and the config sets `reducedMotion: 'reduce'` but no font hinting or device-scale-factor normalization. On Windows with ClearType vs Linux CI with grayscale AA, a 2% maxDiffPixelRatio threshold on text-heavy SVG will produce either chronic false failures or a threshold so loose the gate adds no signal. The document does not address who re-approves baselines when a theme tweak (color token change, font-weight change) invalidates 20 snapshots at once — on a one-person autonomous pipeline that is a full manual re-approval session every time visual polish happens. The Playwright extensions are the right direction, but the snapshot baseline strategy needs an explicit policy: either lock baselines to a single reference machine/container spec and never run them on developer machines, or drop toHaveScreenshot for the agent-generated lesson gate and keep it only for the two locked theme reference pages.

## 🧱 First Principles

AGENT: First Principles
STANCE: CONDITIONAL

REASONING: The proposal correctly identifies that machine-verifiability requires a constrained output format (SVG-in-AlgoStepperShell), and the WebGL block is right for exactly the right reason — gates degrade to opaque pixels. However, the proposal conflates two distinct problems and solves only one of them. The QR failure was not a clip/contrast/anti-pattern failure — it was a semantic failure: the figure was geometrically valid, lint-passing, and visually legible, yet it contradicted the invariant it was supposed to teach. No deterministic CSS rule (Impeccable's 41 checks) can catch "this Gram-Schmidt step is drawn in the wrong order" or "this QR decomposition figure inverts upper/lower triangular." The real bottleneck is not output-format constraint (SVG) nor CSS-layer lint — it is the absence of a parameterized viz-family architecture where correctness is verified once per family and instances are data injected into a verified template. None of the listed tools provide this. The proposal therefore solves the rendering-layer problem (clip, font fallback, contrast) correctly, but leaves the semantic-correctness problem (the actual QR iteration loop) completely unaddressed. The "adopt now" cluster is net-positive and should proceed, but it must not be presented as closing the one-pass quality loop — it closes only the geometry/CSS slice of it.

FLIPS: None on the verdicts themselves. One structural flip: Impeccable + Playwright extensions are necessary but must be explicitly scoped as "geometry/CSS gate only" in the DESIGN.md and active-constraints.md — the current framing implies they collectively constitute the one-pass quality loop, which is false and will be cited later as justification for skipping the parameterized family architecture.

KEY CONCERN: The proposal has no mechanism for catching semantic/pedagogical incorrectness in generated figures (wrong algorithm step ordering, inverted invariant depiction, missing labels that make a step unintelligible) — the exact failure class that caused the QR manual iteration loop. Constraining the output format to SVG is necessary but not sufficient; the sufficient condition is a parameterized component-family architecture where an agent fills typed slots (nodes, edges, labels, step-deltas) and the family's own render logic enforces the invariant. That architecture is the correct lever. Without it, the one-pass goal is a policy aspiration, not an enforced property.

## ⚠️ Risk Analyst

AGENT: Risk Analyst
STANCE: CONDITIONAL

REASONING: The plan is structurally sound and the three adopt-now decisions are directionally correct. However, decision (1) — Impeccable as a gate — carries a supply-chain risk that becomes CRITICAL specifically because it is proposed as an autonomous pipeline gate, not a developer advisory tool. Impeccable at v2.3.2 (10k stars, active) is not abandonware, but `npx impeccable detect` without a pinned version means every pipeline run silently pulls whatever the latest published version is. At 41 rules running against a locked brutalist aesthetic (high-contrast borders, heavy typographic density), a rule-version bump can flip the gate from green to red on a component that has not changed — and in an autonomous agent loop, that false-red triggers an "agent fix" rather than a human pause. The blast radius is wrong-direction aesthetic drift auto-committed, which is non-trivially recoverable but requires a human to notice it happened. The screenshot-baseline brittleness is a REAL but lower-ranked concern: single Windows box + system GPU AA + DPI means initial baselines committed today will produce false-red on any environment change (driver update, display scaling change, headless vs headed Playwright mode). In an autonomous pipeline, false-red baselines cause the agent to "fix" lessons that were correct — same wrong-direction drift problem, different trigger. The Stitch adopt-later risk is confirmed manageable (Google Labs experimental, no SLA, but the unofficial MCP wrapper has 0 stars / 1 commit — the wrapper is the real fragility, not Stitch itself; the plan's "adopt-later, stable pipeline first" timing is appropriate). Google Fonts OFL self-hosting is clean — confirmed legally permitted, no condition violated.

FLIPS:
- Decision (1): ADOPT-NOW flips to ADOPT-NOW-WITH-PINNING. Add `npx impeccable@2.3.2 detect` (explicit semver pin, not floating `npx impeccable`) in the gate invocation. Upgrade is a deliberate human act with a gate-rule diff review, not a silent pull. Until the pin is in place, the gate must be fail-open (log warnings, do not block the pipeline) so a rule-bump cannot trigger autonomous aesthetic correction.
- Decision (3) toHaveScreenshot baselines: ADOPT-NOW flips to ADOPT-NOW-WITH-ENVIRONMENT-LOCK. Baselines must be committed from headless Playwright only (not headed), with `--workers=1` and explicit `deviceScaleFactor: 1` in `playwright.config.ts`. Add a baseline-lock comment naming the exact Playwright version and Node version the baselines were generated on. If those drift, a human re-commits baselines — the pipeline never re-commits them autonomously.

KEY CONCERN: Unpinned `npx` tool in an autonomous gate = the gate itself is a remote-code-execution surface that can autonomously corrupt correct output. Pin the version; the rest of the plan holds.
