# Council review — 1778233802

**Problem:** Jarvis bot dead — Copilot CLI returning 402 quota-exhausted. Need failover so chat keeps working when Copilot premium-request quota is gone (resets monthly) or when Anthropic provider isn't available on VPS.

**Proposed approach:**
1. Tailscale on PC + VPS, mesh private 100.x network.
2. PC runs HTTP server (~30-40 LOC), bound to Tailscale IP, POST /complete → spawn `claude --print` subprocess → return stdout. Bearer token auth.
3. RelayLlm.kt on VPS implements Llm interface, POSTs to PC.
4. FallbackLlm.kt wraps two Llm instances. 2s TCP probe to primary's host:port, cache result 30s. Reachable → primary (relay); else → fallback (copilot).
5. .env: `JARVIS_LLM=fallback`, `JARVIS_PRIMARY_LLM=relay`, `JARVIS_FALLBACK_LLM=copilot`, `JARVIS_RELAY_URL`, `JARVIS_RELAY_TOKEN`.
6. Per-turn provider tag (in `Pair<String, String>` second element) so user can see which path served each turn.

**Project context:**
- Domain: personal life-OS bot (jarvis-kotlin), single-user (Android APK only).
- Stack: Kotlin + Gradle, ProcessBuilder LLM providers, `Llm` interface returning `Pair<reply, providerTag>` with 300s timeout default.
- Existing providers: ClaudeMaxLlm (`claude --print`), CopilotLlm (`copilot -p`), AnthropicLlm (stub).
- VPS at corgflix.duckdns.org runs bot 24/7. PC = Windows daily driver, sleeps when away. `claude` CLI authed on PC; NOT authed on VPS (interactive flow gated as user-confirm item per resume note).
- User has Claude Max subscription on PC.
- Anti-features (per `docs/notes/2026-05-08-autonomous-resume.md`): no public exposure of new ports; no money-spend on new APIs without OK; minimal `/opt/jarvis/.env` edits without user OK; no remote git push.

**Timestamp:** 2026-05-08 (UTC)

## 🔴 Devil's Advocate

AGENT: Devil's Advocate
STANCE: REJECT
REASONING: The entire architecture is built on the assumption that a Windows daily-driver PC is a viable backend for a 24/7 VPS service — it is not. The PC sleeps when the user is away, which is the explicit common case stated in the context, meaning the "primary" path is offline most of the time and the fallback (Copilot, currently 402'd and quota-bound until monthly reset) becomes the actual primary. You've built a Rube-Goldberg failover (Tailscale + HTTP server + bearer token + TCP probe + 30s cache + new Llm impl + wrapper Llm) to get "free Claude Max" in a window where the bot is least likely to be queried — the user is at the PC, not chatting with the phone APK. Worse, the 2s TCP probe lies: a sleeping Windows host with Tailscale's userspace networking can answer SYN at the kernel/agent level for a beat after wake-on-LAN-style nudges, or fail mid-request after the probe passed, so you'll get cache-poisoned 30s windows where every turn 300s-timeouts on a dead socket. And `claude --print` on a logged-in interactive PC session has no documented contract for headless-spawn-from-HTTP-handler stability — session token refresh, TOS rate limits, and Anthropic's right to nuke Max subs used as an API backend are all unaddressed. You are smuggling a paid consumer subscription across a network boundary to power a server, which is precisely what Anthropic's Max ToS forbids.
KEY CONCERN: PC-sleeps-when-away inverts the availability model — the "primary" is offline during the exact hours the phone-only client is most likely to be used, so you've added 4 moving parts (Tailscale, HTTP relay, FallbackLlm, probe cache) to optimize the rare case while the common case still hits the broken Copilot path; ship a direct AnthropicLlm with a $5 API key instead and delete this entire design.

## 📚 Domain Expert

AGENT: Domain Expert
STANCE: CONDITIONAL
REASONING: This is a textbook "split-brain LLM gateway" pattern — Tailscale + tiny HTTP shim + circuit-breaker fallback is exactly how people bridge interactive-auth CLIs (Claude Code, Cursor, Aider) from a residential workstation to a headless VPS. It mirrors what the `claude-relay`, `cursor-tunnel`, and various `claude-code-router` GitHub projects do, and it's the same shape as Ollama-over-Tailscale setups that have been running for two years. Tailscale ACLs (tagged devices) + bearer token is belt-and-suspenders but matches Hashicorp/Tailscale's own published patterns for "expose internal service to one peer." Two divergences from established practice worth fixing: (a) standard fallback implementations use a real circuit breaker (Resilience4j / Polly semantics — closed/open/half-open with failure-count thresholds), not a binary 2s-TCP-probe-with-30s-cache, because TCP-reachable ≠ `claude --print` healthy (PC awake but Claude CLI session expired returns 401 fast, not unreachable); (b) spawning `claude --print` per request is the known-bad path — every production relay I've seen ends up streaming or pooling because subprocess spawn + Node startup is 1-3s of dead latency per turn, and concurrent requests serialize on stdin/stdout if you're not careful. Also: PC sleep is the actual failure mode here, not network — Wake-on-LAN or just disabling sleep on the gateway machine is the standard fix, and every relay-from-laptop project hits this within a week.
KEY CONCERN: The 2s-TCP-probe health check will produce false positives — PC awake + port open + Claude CLI auth expired = primary returns error, fallback never engages. Health check needs to be an actual `/health` endpoint that runs `claude --print "ping"` (or caches last successful real call within N seconds) before returning 200, otherwise you'll get the same "bot dead" incident class you're trying to prevent, just shifted upstream.

## ⚙️ Pragmatist

AGENT: Pragmatist
STANCE: CONDITIONAL
REASONING: The proposed approach is roughly 4 moving parts (Tailscale, PC HTTP server, RelayLlm, FallbackLlm) plus 5 new env vars, to solve a problem at ~tens of turns/day for a single user. That's not unreasonable infra-wise — Tailscale is genuinely 10 minutes, and the PC server is ~40 LOC — but FallbackLlm with TCP probe + 30s cache + provider chain is the part that earns its keep only if the PC actually goes offline often (sleeps, reboots, Tailscale hiccups). Given the PC sleeps when the user is away, that's not hypothetical — fallback is justified. The real cost isn't writing this; it's the *3-months-from-now* debug session when the bot says something weird and you have to figure out which of 3 providers answered, why the probe lied, and whether the cache is stale — the per-turn provider tag (item 6) is what makes that tractable, so don't drop it. The simpler alternatives all have real problems: `claude login` on VPS is one keystroke today but creates a recurring "go SSH in and re-auth" chore the user has explicitly gated; wait-for-quota leaves the bot dead monthly; push-notify-and-wake-PC turns every chat turn into a 2-minute manual ritual, which kills the life-OS premise. So the relay design is proportionate — but only if you cut scope where you can.
KEY CONCERN: Drop the 30s TCP-probe cache entirely on first cut — just try RelayLlm, catch the connect/timeout exception, fall through to Copilot. One fewer state machine, one fewer "why is it stuck on fallback for 30s after PC came back" debug session, ~zero cost at tens-of-turns/day. Add the cache back only if you measure probe overhead actually mattering, which at this load you won't.

## 🧱 First Principles

AGENT: First Principles
STANCE: CONDITIONAL
REASONING: Strip the framing: the real problem isn't "PC online vs offline" — it's "VPS has no auth'd LLM and user won't pay/expose for one." The proposed architecture solves that by turning the always-on PC into a free auth proxy, which is fine, but it inverts the natural topology: the LLM-bearing machine (PC) is treated as a flaky satellite of a dumb relay (VPS), when from-scratch reasoning says the *bot itself* should live where the LLM lives. If the PC is the only Claude-capable node and the phone is the only client, the VPS is just a NAT-traversal endpoint — every chat hop pays VPS→PC→Claude→PC→VPS→phone latency to avoid running the bot on the PC directly. The Copilot fallback also papers over a quota that resets in hours, so the "fallback chain" is solving a failure mode that doesn't really exist on the timescale of a personal bot.
KEY CONCERN: Wrong machine is hosting the bot. If you're already putting Tailscale on both nodes, the simpler from-scratch design is: bot runs on PC (where Claude is auth'd, free, fast), VPS becomes a thin always-on HTTPS ingress that forwards phone APK requests over Tailscale to the PC; PC-asleep state is handled by Wake-on-LAN from the VPS or a "PC offline → 503 retry later" response, not by a second LLM provider. You only need the relay+fallback architecture if you've already decided the bot must live on the VPS — and that decision hasn't been justified against first principles.

## ⚠️ Risk Analyst

AGENT: Risk Analyst
STANCE: CONDITIONAL
REASONING: Top risk is CRITICAL: Claude Max ToS / abuse-flag exposure. Anthropic's Claude Max consumer terms prohibit programmatic/automated use; routing a 24/7 VPS bot through `claude --print` on the user's logged-in PC is exactly the pattern their abuse heuristics target (sustained off-hours traffic, non-IDE shell invocations, shared rate-limit pool with the user's actual Claude Code work). Blast radius is account-level: a flag could cost the user their Max sub mid-finals (priority-block window), and it's not easily recoverable. Second risk is HIGH: the "PC sleeps / interactive re-auth prompt / cached probe lies" cluster — a 2s TCP probe with 30s cache cannot detect a hung subprocess waiting on stdin for re-auth, so the LLM call blocks the full 300s timeout and the user sees a dead bot with no useful error. Mitigations required before approval: (a) hard cap relay QPS + nightly quiet hours to look human-shaped, (b) replace cached TCP probe with end-to-end `/healthz` that actually round-trips a 1-token completion under a short timeout (~5s) and treats "subprocess waiting on tty" as unreachable, (c) detect re-auth prompt pattern in stdout/stderr and fail-fast to fallback, (d) when both primary AND fallback fail, surface a clear "all providers down — check VPS logs" message to the phone client instead of a 300s spinner, (e) confirm with user that Max-account-loss risk is acceptable before shipping.
KEY CONCERN: Claude Max ToS violation risk — running a 24/7 automated relay through a consumer-tier logged-in CLI could get the user's Max account flagged/banned, which is catastrophic and account-level-recoverable-only.

CORE FINDING:
The relay+fallback shape is reasonable for a single-user life-OS bot — Domain Expert confirms it matches the "split-brain LLM gateway" pattern that production projects (claude-relay, cursor-tunnel, Ollama-over-Tailscale) ship with. But three concrete defects must be fixed before implementing: (1) the 2s TCP probe with 30s cache will lie under the exact failure modes you're trying to defend against (PC waking, Claude CLI session expired, subprocess hung on tty), (2) Claude Max ToS exposure is an unaddressed CRITICAL risk that could cost the user their subscription mid-finals window, and (3) the immediate-term value is small because Copilot fallback is currently 402'd until the monthly reset, so PC-offline = bot-dead today regardless. First Principles' "move bot to PC" topology challenge is real but defeated by the existing 24/7 ingress/deploy.sh tooling on VPS — keep the bot on VPS.

AGENT CONSENSUS: 1 REJECT, 4 CONDITIONAL — 0 flagged.

KEY ISSUES:
- **Health-check shape is wrong** (DE + Prag + RA converge): TCP probe ≠ Claude-CLI-healthy. Replace with either (a) a real `/health` endpoint on PC that round-trips a 1-token `claude --print "ping"` under ~5s, OR (b) Pragmatist's simpler cut — drop the cache entirely, just try RelayLlm and catch ConnectException/SocketTimeoutException to fall through. Pick (b) for v1.
- **Claude Max ToS / abuse-flag exposure is CRITICAL** (RA + DA): 24/7 server-side use of a consumer-tier `claude --print` on the user's logged-in account could get the Max sub flagged. Account-level loss, not easily recoverable, hits during finals priority window. Must confirm with user the risk is acceptable before shipping. Mitigations if proceeding: cap relay QPS, nightly quiet hours, log every relay call for audit.
- **PC-sleep is the actual primary failure mode** (DA + DE + FP): the design treats "PC offline" as occasional but it's the common case. Either disable sleep on PC (Windows power plan: never sleep on AC), enable Wake-on-LAN trigger from VPS, OR explicitly accept "PC offline → Copilot fallback (when quota exists)" as the contract.

RECOMMENDED PATH:
**Confirm ToS risk with user FIRST.** If user accepts the Max-account-flag risk, proceed with the design but apply these fixes: (1) drop the 30s TCP probe cache — try-and-fail with short HTTP read timeout (~10s connect, ~120s read) is sufficient at tens-of-turns/day; (2) add a real `/health` check returning 200 only after a successful 1-token `claude --print` round-trip — call it from a daily cron, not per-request; (3) detect "session expired" / re-auth prompt patterns in subprocess stderr and fail-fast to fallback rather than timing out; (4) when both providers fail, surface a clear error string ("all providers down — VPS logs") to the phone client, NOT a 300s spinner; (5) keep the per-turn provider tag — Pragmatist is right that this is the diff between this being debuggable and not in 3 months; (6) disable PC sleep on AC power as the standard mitigation. If user rejects the ToS risk → either pay for Anthropic API key (violates no-money-spend anti-feature, needs explicit OK) or `claude login` on VPS (interactive blocker, also user-gated). Do NOT relocate the bot to the PC — VPS hosting is correctly justified by deploy.sh + 24/7 ingress.
