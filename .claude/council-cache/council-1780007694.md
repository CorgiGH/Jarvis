# Council review — 1780007694

**Problem:** Was the `os-study-guide` project's content/visual APPROACH actually good/best — and is it the right fit to adopt for the `jarvis-kotlin` teaching engine?

**Proposed approach (os-study-guide's, under evaluation):** Course content = canonical JSON per course → ordered "steps" → typed "blocks" (learn/definition/equation/code/quiz/image/diagram) + a block→React-component renderer registry. A 5-stage LLM curation pipeline (Gemini→Claude) converts PDFs → that JSON; diagrams stored as text DESCRIPTIONS + a `canBeReproducedAsSVG` flag; PDF pages rasterized to PNGs referenced by `image` blocks; interactive widgets (SVG graph animator, Judge0 code-run, v86 terminal, KaTeX). Observed outcome: sparse — 4 images total, diagram renderer shipped as a permanent placeholder, mostly text-only.

**Project context:** jarvis-kotlin = Kotlin/Ktor + SQLite + React. Content currently = git-tracked YAML knowledge-concepts (KCs) + prereq graph + misconceptions + verbatim-source validation. ~20 hand-built interactive viz components, FSRS spaced-repetition, LLM drill-grader. A DRILLING/MASTERY tutor, not a course-page reader. Timing/deadlines ruled out of scope; judge approach quality + fit only. $0 budget, solo dev.

"Visual-prone KC ⇒ a rendered, paint-verified visual" must be a hard ContentValidator error (KC carries visual_kind + bound viz_id; missing ⇒ validateContent fails) AND a Playwright first-paint assertion — else the visual layer stays aspirational exactly as in the cited project.

## 📚 Domain Expert
AGENT: Domain Expert
STANCE: REJECT
REASONING: os-study-guide's approach is a textbook structured-content / portable-text pattern (canonical JSON → blocks → renderer registry) — exactly Sanity Portable Text, Contentful Rich Text, Notion's block model, MDX-as-data. Sound for a READING/authoring surface (a CMS-backed course reader). But that's the wrong model for jarvis, which is an intelligent tutoring system (ITS), not a course CMS. ITS content has its own prior art — the knowledge-component model (Koedinger/Aleven CTAT, ASSISTments, KLI framework), parameterized problem-templates + grader rules, prerequisite graphs, misconception/bug catalogs (Brown & Burton BUGGY, Ohlsson constraint-based modeling) — and jarvis already implements precisely that on FSRS + PFA mastery. A block-document renderer has no slot for a learner model, scheduling state, item generation, or per-attempt grading. The os-study-guide outcome also confirms it's a poor VISUAL fit — "diagram = text description + canBeReproducedAsSVG flag" is a deferred-rendering escape hatch that predictably produced 4 images; jarvis's "visual-prone REQUIRES a visual" with 19+ already-built interactive V1-V20 SVG primitives is the correct discipline (dual-coding) and already shipped.
KEY CONCERN: Adopting the block/renderer model forces jarvis to re-encode mastery, scheduling, item-generation and grading as out-of-band metadata bolted onto a document tree — fighting the schema instead of using one (KC + template + misconception + prereq-graph) that natively expresses them.

## ⚙️ Pragmatist

STANCE: CONDITIONAL

The two live at different layers: jarvis's YAML KCs are a semantic MASTERY layer (bloom + prereqs + misconceptions + verbatim-source) feeding drill→grade→FSRS; os-study-guide's JSON-blocks are a PRESENTATION layer for linear reading. KEY CONCERN: Don't let "adopt the approach" creep into swapping the content representation — block-renderer and KC model are not substitutes (presentation vs mastery); adopting the former guts the drill/FSRS engine. Cherry-pick the PDF-extraction pipeline + diagram-description convention only.

## 🧱 First Principles

STANCE: REJECT
REASONING: jarvis's real goal is mastery — present a concept, drill it, grade the attempt, schedule it via FSRS until owned; content exists to GENERATE + GRADE, never to be read. os-study-guide's real goal is the opposite: faithfully render authored lesson pages for a human to read. os-study-guide is built entirely around the page-reader unit (JSON → steps → display blocks → renderer), exactly the abstraction jarvis must NOT inherit, because jarvis's atomic unit is a gradeable, schedulable concept (bloom_level, difficulty, exam_weight, prereq edges, misconceptions, verbatim-source) — none of which is a "block" you paint. KEY CONCERN: Adopting the JSON-blocks→renderer model forces jarvis to model concepts as renderable pages rather than gradeable/schedulable mastery units, structurally weakening the drill-generation, grading, and FSRS signals that are the entire point.

## ⚠️ Risk Analyst
AGENT: Risk Analyst
STANCE: CONDITIONAL
REASONING: The misframing is treating os-study-guide's content model and its visual failure as one package — they aren't. The block→component model is fine; what under-delivered was the visual generation/ENFORCEMENT mechanism (diagrams as text descriptions behind a flag nobody acted on → renderer shipped as a placeholder). jarvis is structurally inoculated against THAT failure: its 24 viz components (verified in tutor-web/src/components/viz/) are concrete hand-coded React/SVG, each with a logic test AND a palette test. But HIGH risk: jarvis re-imports the failure through a different door — the visuals exist but aren't WIRED to concepts. Verified: ContentSchema.kt / ContentValidator.kt have ZERO viz/visual field, and the 24 components are mounted ONLY in the /viz-demo gallery (main.tsx:23), referenced by no concept surface — the project's own ghost-component / built-but-not-mounted pattern. MEDIUM: a YAML-only "require a visual" check verifies only the YAML half — requires_visual:true + viz_id:foo passes green while foo renders nothing.
KEY CONCERN: The "REQUIRE a visual" rule only prevents the failure if enforcement spans BOTH halves (none exists today). Mandatory: (1) add viz_id to the KC schema; (2) create a vizId→component registry (none exists); (3) extend ContentValidator to fail when requires_visual && viz_id missing OR not in a generated registry-manifest; (4) a Playwright assertion that each visual-prone KC's surface paints the registered component, zero 4xx, no placeholder/error text. With all four it genuinely prevents recurrence; with only the flag it reproduces the placeholder trap.
