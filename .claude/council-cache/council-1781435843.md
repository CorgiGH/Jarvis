**Problem:** A solo-built AI tutor keeps shipping PARTIAL artifacts claimed "done" but aren't (a viz "family" = a picture, not a teaching lesson; subagents reported "zero text overlap" when a manual sweep found 9). Owner wants a PIPELINE with "100% compliance to the final spec on everything" — every artifact for the whole corpus provably complete-to-spec, every defect caught by MACHINE gates (never the user's eyes), proven by the §10.2 11-artifact acceptance run. Curing subjective "done."

**Proposed approach:**
1. **Compliance harness** — decompose the entire final spec into ONE machine-checkable checklist every artifact must pass → a single GREEN/RED "fully spec-compliant?" verdict. Wires existing gates (trace-match, no-clip×2 viewports, legibility, interaction-smoke, RO-language, impeccable lint, visual-baseline, NLI trust-net faithful-check) PLUS 2 newly-ruled unbuilt gates: a machine VISUAL/overlap gate (per-frame real-viewport, zero text overlap, golden-master diff) + a DEPTH-artifact gate (separate non-figural artifact: invariant + complexity DERIVATION + correctness + edge cases + line-synced code, own correctness gate). Makes "100%" objective.
2. **Gated build workflow** — PM-orchestrated subagent workflows build each remaining unit → run the harness → block until GREEN; PM re-verifies every result himself at the real viewport. Drives locked plan sequence V (viz families) → deploy → 5 (digestion) → 7 (tracking + proof run); W (warm theme) after V.

**Project context:** Single-user AI tutor (jarvis-kotlin), only active vertical of a personal life-OS. Kotlin/Ktor + SQLite backend; React/TS + vitest + Playwright frontend; SVG-only viz; FSRS; NLI "trust-net" faithful-check. No paid LLM APIs. User (ADHD) cannot vet subject content → ALL truth machine-gated ("corespunde cursului," never "correct"). ## 🔴 Devil's Advocate
AGENT: Devil's Advocate
STANCE: REJECT
REASONING: The pipeline encodes a category error: it equates "passes every machine gate" with "complete-to-spec," but the two original failures were both *missing-gate* failures, not failing-gate failures — the viz "family" passed because no gate asserted "teaches," and the overlap slipped because the gate that should have caught it didn't exist or wasn't wired. A harness can only ever prove compliance to the checklist you decomposed; it is structurally blind to the requirement you forgot to decompose, which is precisely the class of defect that has burned this project twice. Worse, the plan hands the *authoring of that decomposition* and the *building of the two net-new gates* to the same subagent-workflow layer that already self-reported "zero overlap" against reality 9 times — so the gate that certifies the gates is the untrusted component, and "PM re-verifies at the real viewport" silently smuggles the human eyeball back in as the true final authority, contradicting the entire premise that no defect is ever caught by the user's eyes. The fixed §10.2 acceptance run makes this catastrophic rather than merely incomplete: it converts "11 exemplars are GREEN" into "100% of 140 sources are done," so a single missing checklist item becomes a *silent corpus-wide false-GREEN* that ships garbage to a user who by definition cannot detect it.
KEY CONCERN: The harness can only falsify against the decomposition it was given; the decomposition itself has no gate. Before building anything, the missing-requirement class must be made machine-detectable — e.g., a gate-coverage invariant that fails RED whenever a spec clause maps to zero gates, and a standing adversarial-red-team gate whose sole job each cycle is to find one spec-true defect the current checklist passes (a "same defect-CLASS escapes twice = pipeline design bug" tripwire). Until the decomposition is itself adversarially gated and the two new gates are built+verified by something other than the workflow they police, GREEN means "compliant to what we remembered," not "100% to spec."

## 📚 Domain Expert
AGENT: Domain Expert
STANCE: CONDITIONAL
REASONING: The core architecture is sound and maps cleanly onto proven patterns: decomposing the spec into machine-checkable assertions is **acceptance-criteria-as-code / executable specifications** (the ATDD/Fitnesse/Cucumber lineage), the per-artifact gate suite is a **SonarQube-style quality gate**, the block-until-GREEN-before-merge orchestration is a **merge queue / required-checks** model, and the figure-render-once-per-family plus the new VISUAL gate are textbook **golden-master/approval testing (ApprovalTests) and visual regression (Percy/Chromatic/Playwright snapshots)**. Constraining viz to typed FAMILIES and validating `family_id`+typed data is **schema/contract validation**, exactly how mature EdTech content pipelines (QTI/Open-assessment, Khan-style item banks) keep auto-graded items provably well-formed. Where it diverges from what works is the headline framing — "ONE machine-checkable checklist → a single GREEN/RED verdict" is the classic **single-aggregated-pass/fail anti-pattern**: SonarQube, OPA/Conftest, and merge-queue systems deliberately keep **per-dimension conditions with named, independently-reported sub-results** under the aggregate. Keep the aggregate as a derived rollup, never as the primitive.
KEY CONCERN: The two named **fail-open / environment-skew defects (impeccable-lint FAIL-OPEN; visual baselines captured on Windows while CI is Linux) structurally invalidate the "100% compliance" claim before the new gates are even built.** A gate that fails-open is not a gate — it is the "zero overlap when a sweep found 9" failure re-encoded into CI; Windows-vs-Linux snapshot skew is the canonical reason teams move golden-master capture into a **pinned container (the Playwright Docker image / `--platform linux` snapshots, the way Chromatic pins rendering infra)**. Until every gate is **fail-CLOSED and runs in the same containerized environment that produces the baselines**, the harness hands a GREEN over an unverified artifact. Fix the meta-gate (fail-closed + env-parity, enforced by a "gate-self-test" that feeds each gate a known-bad fixture and asserts RED) before sequencing V→deploy→5→7.

KEY CONCERN: An aggregated single verdict built on top of a fail-open gate + lying baselines + a 1-in-2 flake is worse than no harness — it manufactures false GREENs and trains you to ignore RED; fix the trust of the underlying gates before you give them one authoritative voice.

## 🧱 First Principles
AGENT: First Principles
STANCE: CONDITIONAL
REASONING: From scratch, the goal is not "100% spec-compliance proven by machine" — that is logically impossible and the proposal half-admits it (§10 keeps a human approval step while the stated goal says "never the user's eyes"). The true goal is: minimize the rate of FALSELY-confident "done" claims. A machine gate partitions artifact properties into two disjoint classes — DECIDABLE predicates (cardinality, monotonicity, trace-match-vs-oracle, 4xx-free, lint, pixel-overlap-by-bbox) where a green is a PROOF, and SEMANTIC/aesthetic judgments ("genuine depth," "faithful," "beautiful," "pedagogically good") where any "gate" is only an ESTIMATOR with false-negatives. The documented failure — subagents reporting "zero overlap" while 9 existed — was a green from the SEMANTIC class misread as a green from the DECIDABLE class. So the right design: (a) build the harness, but tag every check `PROOF` or `ESTIMATOR`; an all-green verdict reads "all PROOF checks pass; N ESTIMATOR checks are advisory" — it can NEVER print "100% spec-compliant." (b) Drive estimators toward proof by converting fuzzy properties into decidable ones at the SOURCE: "no overlap" becomes a computed bounding-box intersection assertion (proof), not an agent's eyeballed report (estimator); "faithful" stays an estimator (NLI admits/refuses) and is permanently fenced. The proposal aligns with build-the-harness and convert-overlap-to-geometry, and its DEPTH gate correctly externalizes depth into a separate decidable-shaped artifact — but it MUST drop the "100% / never the user's eyes" framing, which collides with its own §10.
KEY CONCERN: The single verdict bit erases the PROOF-vs-ESTIMATOR distinction — and that exact erasure (a true estimator-green misread as a proof-green) is the precise mechanism of the failure this effort exists to kill. The verdict must structurally separate "proven" from "estimated, human-gated."

## ⚠️ Risk Analyst
AGENT: Risk Analyst
STANCE: CONDITIONAL
REASONING: The single aggregated GREEN/RED verdict is the most dangerous choice: it fuses ~8-10 gates of wildly different reliability into one bit, so a FAIL-OPEN lint gate, a cross-platform visual-baseline that diffs Linux against Windows-captured goldens, and a not-yet-buildable DEPTH gate all silently report "pass-or-skip" and the harness still shows GREEN — recreating the false-green incident the pipeline exists to kill, now laundered through a machine verdict the PM trusts MORE. The DEPTH gate is the structural impossibility: "is this complexity derivation / correctness argument correct" is not machine-decidable the way pixel-overlap is, so either it stays a perpetual "neverificat" flag (100%-compliance never reachable by construction) or someone weakens it to a proxy (lint/keyword/NLI-on-prose) that passes garbage — and the spec's "fail twice = pipeline DESIGN bug, no third regen" rule means a too-weak DEPTH gate doesn't just leak, it permanently bricks artifacts. Secondarily, "run against the REAL corpus" + StateCache ~1-in-2 flake means the acceptance run itself is non-deterministic (flaky RED re-run to GREEN = retry-until-green), and a live-DB harness without the spec-mandated off-box backup risks corrupting the only real trust-net DB.
KEY CONCERN: CRITICAL — a single aggregated verdict that treats FAIL-OPEN/SKIP/cross-platform-unverified gates as GREEN reincarnates the false-green incident with higher PM trust; compounded by the DEPTH gate being non-machine-decidable yet bound by the no-third-regeneration rule. APPROVE only if the verdict is a non-collapsing per-gate matrix where SKIP/FAIL-OPEN/cross-platform-unverified each render RED, depth ships as an explicit "neverificat" human-checkpoint lane, and the acceptance run is pinned (serialized, backup-first, flake-quarantined-not-retried).
