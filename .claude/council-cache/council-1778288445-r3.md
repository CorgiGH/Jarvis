## 🔴 Devil's Advocate

STANCE: NEITHER (both = procrastination, but if forced to pick, neither plan ships PS-tutoring before May 21)

KEY CONCERN: Plan A's "React Router introduced mid-stream at Layer B" is a self-inflicted wound. Adding Router after components already use raw URLs means rewriting every navigation site, every test that asserts on location, every link. This isn't a "small refactor" — it's a category change. Either commit Router day 1 or never. Mid-stream is the worst option and Plan A picks it.

Other findings:
- Both estimates ("~3-5 hours") are vibes-numbers with no basis
- Option 2's empty-scaffold premise contradicts the only thing R2 actually validated — schemas/contracts/auth correct day 1. Empty MarkerRenderer with all types "registered" but nothing rendering is exactly the kind of fake-correct surface that hides integration bugs until Layer C, when there's less time to fix them.
- 3-pane workspace scaffolded in Layer A before 2-pane is dogfooded = locking UX before learning UX
- "Multi-tenant by userId on a single-user app" is dead weight in both plans
- The "PS HW deadline" is doing zero work in either plan

## ⚙️ Pragmatist

STANCE: PLAN_A

KEY CONCERN: Front-loading 600-800 LOC of scaffolding before validating UX shape locks in design decisions that will get re-decided anyway, while the actual deadline-critical artifact (PS HW solver) is identical under both plans.

Concrete cost analysis:
1. The deadline doesn't care which plan wins (~0h delta to PS HW target). Option 2 burns ~4-6h on scaffolding that doesn't move the deadline-critical needle.
2. Refactor cost overstated, greenfield-fill cost understated (~1-2h actual delta). Layer B refactor = adding `<BrowserRouter>` wrapper + 1 new `<Route>` + grid-template-columns CSS swap + a badge component = ~30min, not 1-2h. Layer C "refactor" = mostly NEW component code; the swap is `<MarkerRenderer />` instead of `<pre>{text}</pre>` in one file = 10min mechanical.
3. Mid-execution plan switch has its own cost (~1-2h). Tasks 1-18 already done with Plan A in mind. Switching to Option 2 means revising plan doc + re-scoping Tasks 19-26.
4. Mental-hold argument cuts the OTHER way for solo dev mid-finals. Plan A's intermediate states are uglier shape-wise but each layer is self-contained. Option 2 mid-build = "what was I supposed to do with /trust again? did I finish /audit's table or just stub it?"
5. Dead-route risk in Option 2 (~30min debug per incident, 1-3 incidents likely)
6. Test rot on placeholders is the silent killer (~1-2h cleanup later)

**Verdict: Plan A. Refactor cost is a paper tiger; scaffold cost is real and front-loaded into the deadline window.**

## 🧱 First Principles

STANCE: PLAN_A (with sharp reframe of what "rebuild" means)

KEY INSIGHT: The user is conflating "modifying code" with "rebuilding it." In Plan A, nothing that's expensive to change ever changes. The 3-5 hour refactor tail is *additive composition*, not rebuild. Option 2 doesn't eliminate that work — it just frontloads it as scaffolding-without-payload, which is strictly worse because you're committing to UI shape decisions before you have the information needed to make them well.

Findings:
1. Is the instinct correct? No — it's misframed. "Rebuild = waste" is true when rebuild means *throwing away load-bearing decisions and redoing them*. That's not what's happening here. The load-bearing layer is schemas + contracts + auth + WAL + ULID — and that's already done and final day 1.
2. The real distinction: Plan A = build minimal → use it → learn what the actual shape needs to be → add the next layer with that knowledge. Option 2 = predict the full shape upfront → scaffold it empty → fill it in later. Same total work; different *information ordering*.
3. Which principle: YAGNI, decisively. Design-first applies when shape is hard to change later (DB schemas, API contracts, auth). All those are *already designed-first* in Plan A. UI structure is the opposite category — cheap to change, right shape only visible once you're using the thing.
4. Third option ("skip Layer A UI, jump to Layer B/C") deletes the feedback loop. Worse than either.
5. The real third option: continue Plan A, timebox the refactor tail. If at Layer C the "3-5 hours" balloons past 8, that's the signal Plan A miscalibrated — pause and reshape. Until then, the user's already-validated 6/26 progress is the most reliable evidence.

**Verdict: User's instinct is reasonable in general, misapplied here. Plan A is correct. Don't pause to over-design what you'll learn cheaper by building.**
