# Council review — 1779311876

**Problem:** Gate 3 ("Content authoring foundation") of the jarvis-kotlin tutor redesign has an internal spec contradiction: §14 makes Gate 3 deliver the `/curate-tutor` skill stages 1-7, but §13 defines those stages as a Python pipeline (MinerU + 4× DSPy + PaperQA2) that §14 schedules for Gate 5. A gate cannot depend on a later gate's deliverable. The Gate 3 architecture must be resolved before an implementation plan is written.

**Proposed approach:** Re-scope Gate 3 so it does NOT depend on the Python sidecar — Q1: build `/curate-tutor` as a Claude Code skill (markdown), not DSPy programs; Q2: content = git-tracked YAML files under `content/{subject}/...`, not DB tables; Q3: `subjects.yaml` manifest + per-artifact YAML + `edges.yaml` + auto-generated `edges.mmd`; Q4: Kotlin DAG validator (cycle/orphan/exam-weight/bilingual/attribution); Q5: `/api/v1/curator/*` read+validate+write routes (write = optimistic version lock + atomic git commit); Q6: `/curate-tutor` as a repo-tracked skill file. Acceptance: run `/curate-tutor` end-to-end on real PA chapter 1.

**Project context:**
- Domain: jarvis-kotlin — Socratic CS tutor (web + mobile) for a CS student + small friend cohort; mid 11-gate dependency-ordered redesign.
- Stack: Kotlin/Ktor backend (SQLite + Exposed, routes in one TutorRoutes.kt, magic-link auth shipped Gate 2); React/TS frontend; single VPS, manual deploy.sh; Python ML sidecar planned, not built.
- Constraints: $0 LLM budget (OpenRouter free-tier / `claude -p` only); per-gate merge-to-main shippable increments; spec §4/§5 stale (describe a JS Auth.js/Drizzle/Postgres stack that cannot run on the JVM — actual code is Kotlin-native SQLite).
- Prior decisions: Gate 1 + Gate 2 shipped + deployed. Existing code: SubjectCorpus.kt, KnowledgeGraph.kt; no YAML lib, no DAG code, no content schema.

## 🔴 Devil's Advocate

AGENT: Devil's Advocate
STANCE: CONDITIONAL
REASONING: The re-scope correctly fixes the dependency inversion, but it smuggles in a fatal equivocation: "build `/curate-tutor` as a skill, not DSPy programs" treats the *mechanism* swap as cost-free when it silently deletes stage 7 — PaperQA2 groundedness verification. Spec §13 makes attribution (stage 6) and groundedness (stage 7) the gate that stops hallucinated knowledge-concepts and misconceptions from entering the corpus; an LLM doing "misconception mining" with no automated groundedness check is *exactly* the failure mode where a free OpenRouter model invents a plausible-sounding prerequisite edge or a misconception that PA chapter 1 never states, and your Kotlin validator happily passes it because "non-empty source attribution" only checks the field is non-empty, not that the cited span actually supports the claim. Gate 3 then ships poisoned content as the git-tracked *source of truth*, and every downstream pedagogy/drill gate builds on it — the corruption is discovered three gates later when a student is taught a wrong prerequisite. The proposal also assumes Claude reading the PDF "directly" is equivalent to MinerU 2.5 structured extraction; for an Algorithm Design chapter that is dense with pseudocode, recurrence math, and figures, raw multimodal ingestion silently drops or mangles exactly the structured content that KC discovery depends on, and nothing in the acceptance criteria would catch it.
KEY CONCERN: Dropping stage 7 (groundedness verification) with no replacement means Gate 3 can ship LLM-hallucinated KCs/edges/misconceptions that pass the validator and become the permanent source of truth — the gate MUST add a deterministic claim-to-source-span check (every KC/misconception field must quote a verbatim substring of the extracted PDF text, validated by the Kotlin DAG validator) before acceptance, or the re-scope is just trading a scheduling bug for a silent data-integrity bug.

## 📚 Domain Expert

AGENT: Domain Expert
STANCE: APPROVE
REASONING: This is the textbook fix for a circular dependency in a layered build plan: split offline content authoring from runtime ML productionization, which are genuinely different concerns — the former is a batch ETL/ingestion job, the latter is a serving pipeline. Treating content as git-tracked YAML with a Mermaid diff mirror is the established "docs-as-code" / GitOps pattern (cf. Backstage software catalog, OpenAPI specs, dbt's YAML+`ref()` model) and is the correct source-of-truth choice for an artifact a human curator approves before it ever hits a DB. Cycle detection via three-color DFS is the standard algorithm, optimistic `version` locking returning HTTP 409 is the canonical concurrency-control idiom (HTTP RFC, CouchDB, JIRA all do exactly this), and gating the schema-targeting pipeline as a versioned repo artifact is the right call — a skill that drifts from its schema is the classic broken-contract failure. The one real divergence from established practice is benign: replacing DSPy with an LLM-guided skill trades DSPy's compiled, eval-optimized prompts for an unoptimized one-shot extraction, which is acceptable for offline authoring precisely because a human swipe-approves every KC — the curator IS the eval harness.
KEY CONCERN: The `git commit`-per-edit inside an HTTP handler is the weak seam — shelling `git` from a request thread is not atomic under concurrent writes (index lock contention, partial commits, dirty working tree from other processes), and the standard pattern is a serialized write queue or a real content-API/headless-CMS (Git Gateway, Strapi-style) rather than racing the shell; for a single-VPS single-curator deployment this is survivable, but mark it explicitly as a Gate-3 simplification to revisit, not a permanent design.

## ⚙️ Pragmatist

AGENT: Pragmatist
STANCE: CONDITIONAL
REASONING: The re-scope is correct and removes a real dependency contradiction — building `/curate-tutor` as a Claude skill matches the existing `curate`/`adding-subject` pattern (zero Python, already proven) and is far cheaper to build and hand off than standing up a DSPy/MinerU sidecar two gates early. But the proposal smuggles in three non-trivial sub-projects that aren't load-bearing for "content authoring foundation": git-commit-per-edit from inside an HTTP route, optimistic version locking with 409s, and a CI-integrated bilingual/exam-weight validator. The DAG validator and YAML schema are the real Gate 3; build those well and ship `/curate-tutor` against PA chapter 1, but treat the curator write-endpoints as the thinnest possible thing — at this stage YAML is hand/skill-edited and reviewed in a normal PR, so an HTTP upsert with server-side `git commit` is solving a problem one developer doesn't have yet.
KEY CONCERN: Q5's write path. A Ktor route shelling out to `git add`+`commit` per accepted edit is a known maintenance trap — concurrent writes, partial commits, dirty working tree on the VPS, commits authored by the server instead of a person, and a debugging story that's miserable when it half-fails. Gate 4 builds the curator SPA anyway, so Gate 3 only needs read/validate endpoints plus a validator; defer the upsert+commit+version-lock machinery to Gate 4 where it has a real consumer, or replace it with "skill writes the file, human commits via PR." Shipping the write path now is ~60% of the build cost for a feature nothing uses for a whole gate.

## 🧱 First Principles

AGENT: First Principles
STANCE: CONDITIONAL
REASONING: The proposal correctly diagnoses the surface contradiction — a gate cannot consume a later gate's deliverable — and resolves it the only logically valid way: §13's 8-stage Python pipeline is a *runtime* RAG/tutoring system, while Gate 3's actual goal is a one-time *offline content extraction*, and conflating them is the spec's error, not a real dependency. But the proposal still inherits one piece of the proposer's framing without challenge: it accepts that Gate 3 must "run /curate-tutor end-to-end on real PA chapter 1" as an acceptance criterion, when the stripped goal is only "produce a validated content corpus + the machinery to validate/serve it." Whether the YAML was hand-seeded, LLM-extracted, or both is irrelevant to whether Gate 3 is a shippable increment — coupling the gate's done-state to a non-deterministic LLM extraction run makes acceptance flaky for no architectural reason. Also, building both the Kotlin DAG validator AND the §13 PaperQA2 groundedness checker means two attribution/verification systems will exist; first principles says pick one home for "is this content valid," and Gate 3's Kotlin validator should own the structural invariants permanently while Gate 5 owns only semantic groundedness.
KEY CONCERN: Acceptance is tied to a non-deterministic LLM run ("/curate-tutor run end-to-end"). Make the gate's pass/fail depend only on deterministic artifacts — schema, validator, routes, and a checked-in validated corpus — and treat the extraction skill as a tool that produced the corpus, not as a test that must pass.

## ⚠️ Risk Analyst

AGENT: Risk Analyst
STANCE: CONDITIONAL
REASONING: The re-scope correctly removes the impossible Gate-3-depends-on-Gate-5 cycle, and YAML-as-source-of-truth with a Kotlin DAG validator is sound and recoverable. But two failure modes are underweighted. First, the curator write-route shelling `git add`+`commit` from inside the long-lived `jarvis` JVM process: concurrent writes race on the index lock, a crash mid-commit leaves a dirty/partially-staged tree, and an unbounded local commit history will silently diverge from `origin` until a human pushes — the route owns a stateful resource (the working tree) it cannot safely arbitrate. Second, with no PaperQA2 groundedness check in Gate 3, `/curate-tutor` can emit confident, schema-valid, hallucinated KCs and fabricated prerequisite edges: the DAG validator proves the graph is acyclic and weights sum to 1.0, but proves nothing about whether the content is true — so a clean validator pass becomes false assurance that poisons every downstream pedagogy/drill gate.
KEY CONCERN: Mitigate the git-from-JVM risk before shipping: serialize all curator writes through a single-threaded queue (or a process-level file lock) so only one commit runs at a time, make the write atomic (tmp-file + fsync + rename, then stage only that path), fail the request loudly on any `git` non-zero exit instead of leaving a dirty tree, and treat commit-but-no-push as expected state with an explicit operator push step. Separately, gate the Gate-3 acceptance on human curator swipe-approval (stage 8) being mandatory and on the validator output explicitly labeling "structural checks only — content not groundedness-verified," so an unverified hallucination can never be mistaken for vetted content.

CORE FINDING:
The central re-scope — split offline content authoring from the runtime ML sidecar, treat YAML as git-tracked source of truth, validate with a Kotlin DAG checker — is unanimously sound and correctly resolves the spec's dependency contradiction. But the proposal as written has three fixable defects that must be corrected before it becomes a plan: it silently drops §13 stage 7 (groundedness verification) leaving the validator's "non-empty attribution" check unable to stop hallucinated content from entering the permanent source of truth; the Q5 curator write-path has no consumer until Gate 4 and is a known git-from-JVM hazard; and acceptance is tied to a non-deterministic LLM run.

AGENT CONSENSUS: 1 APPROVE, 4 CONDITIONAL — 0 agents flagged.

KEY ISSUES:
1. Groundedness gap (Devil's Advocate, Risk Analyst). The Kotlin validator must verify each source-attribution field is a VERBATIM substring of the extracted source text — not merely non-empty — and its output must self-label "structural checks only — content not groundedness-verified." Human swipe-approval is Gate 3's real semantic gate (the curator is the eval harness).
2. Write-path scoping (Pragmatist, Domain Expert, Risk Analyst — 3 of 5). Q5's upsert + `git commit` + version-lock has no consumer until Gate 4's SPA. Defer the write path to Gate 4. Gate 3 ships read + validate routes plus the validator only. If any write capability is kept it must be serialized (single-threaded queue / file lock), atomic (tmp+fsync+rename), and fail loudly on any `git` non-zero exit.
3. Deterministic acceptance (First Principles). Gate 3 pass/fail must depend only on deterministic artifacts: the schema, the validator (with tests), the read routes, and a checked-in PA-chapter-1 corpus that passes the validator. `/curate-tutor` is the tool that produced the corpus; running it is not itself the gate test.

RECOMMENDED PATH:
Adopt the re-scope with three amendments. (1) Cut Q5's write path from Gate 3 — ship read + validate routes only; move upsert/commit/version-lock to Gate 4 alongside the SPA that consumes it. (2) Extend Q4's validator with a verbatim-source-quote check plus a "structural-only" self-disclaimer. (3) Redefine Gate 3 acceptance as deterministic artifacts: schema + validator-with-tests + read routes + a committed validator-passing PA-chapter-1 corpus; treat `/curate-tutor` as the authoring tool run once to produce that corpus. Keep Q1, Q2, Q3, Q6 as proposed.
