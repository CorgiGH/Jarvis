## 📚 Domain Expert

```
AGENT: Domain Expert
STANCE: CONDITIONAL
REASONING: The proposed shape is a defensible mashup of three proven primitives, but the ordering inverts the strongest one. (a) Step 2's "predict before reveal" is precisely Slamecka & Graf 1978's generation effect, and tightened further by Richland, Kornell & Kao 2009 ("pretesting effect" / "errorful generation") — even wrong guesses before instruction outperform passive reading. That part is well-grounded. (b) Step 3's "LLM gets prediction in system prompt + told to confirm what's right, name what's missing, don't hand the answer" is exactly Khanmigo's published guardrail lineage (Khan Academy's 2023–24 Khanmigo system-prompt forbids direct-answer; uses Socratic counter-questions) and matches the Cognitive Tutor / ASSISTments "hint-on-demand AFTER attempt" pattern (Heffernan & Heffernan 2014 on ASSISTments; Anderson, Corbett, Koedinger, Pelletier 1995 on Cognitive Tutors). (c) HOWEVER: Step 1 ("show 3-5 corpus hits, no LLM yet") BEFORE Step 2's prediction gate is the confused part. Roediger & Karpicke 2008 (Science 319:5865) operationalized the testing-effect condition as **closed-book free recall** — putting corpus excerpts on screen before the user commits a prediction collapses the condition toward open-book study (Agarwal, Karpicke et al. 2008 showed open-book produces materially smaller long-term gains). Anki/Quizlet enforce this strictly: card front alone on screen, back gated behind self-rated "I tried." Brilliant.org's "I tried" reveal is the same primitive. The fix is small: swap Step 1 and Step 2. Predict FIRST (closed-book, only selected text + drill question visible — no corpus pane), THEN reveal corpus hits + sidekick answer together as the feedback phase. Current ordering is a Hypothes.is-style "browse-then-think" shape wearing testing-effect clothing.
KEY CONCERN: Step 1's "show 3-5 corpus hits BEFORE the predict gate" violates the closed-book condition the testing effect is operationalized over. Reverse the order: predict first with no corpus visible, then reveal retrieval hits + LLM response together as feedback — that ordering is Anki/Brilliant/Khanmigo lineage.
```

## 🧱 First Principles

```
AGENT: First Principles
STANCE: REJECT
REASONING: The proposal solves the wrong problem. The chip is attached at `inlineAsk.ts:39-46` to ANY `.card-body` descendant, which means today's design treats DRILL, WORKED, DEFINITION, and CHECK as identical clarification surfaces (`DrillCard.tsx:103`) — that is the actual bug. The right primitive separation is: chip stays on the read-along cards (WORKED/DEFINITION/CHECK) where clarifying a definition is legitimate; the chip should NOT mount inside the DRILL card body at all, because a drill IS the closed-book test and Slice 3 already shipped a real grader (`DrillStack.tsx:86-107`) that gates progression. Rebuilding the chip into a 3-step "show corpus → predict → confirm" flow is pedagogically self-defeating: showing retrieval hits BEFORE prediction destroys the retrieval-effort precondition (Roediger/Karpicke) and a textarea that paraphrases the excerpts above it isn't the generation effect (Slamecka 1978) — it's transcription. If the testing-effect goal is real, it belongs as a "predict in plain language" textarea ABOVE the existing code editor inside `DrillStack.tsx`, gated by the existing grader, not as a sidekick chip variant. "Your-notes-first" is marketing for a mechanism the implementation contradicts; the precise operation is "force closed-book retrieval before reveal," which `card-body` chip scope cannot enforce.
KEY CONCERN: The chip-flow does not belong on the DRILL card body at all. Scope `attachSelectionListener` to non-DRILL `.card-body` elements (or skip when the nearest `[data-testid="drill-card"]` has `cardType="DRILL"` and `state="open"`), and if a predict-before-attempt step is wanted, add it inline to `DrillStack` above the `drill-attempt-input` textarea — not into the sidekick chip. Anything else rebuilds the wrong primitive.
```
