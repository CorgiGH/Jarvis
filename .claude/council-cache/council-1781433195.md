# Council review — 1781433195

**Problem:** A tutor lesson should default to a simplified INTUITION layer and let the student drill into the ACTUAL TECHNICAL DETAIL. Claim made: family 2 (selection sort) "proves intuition + detail" via two render skins of the same figure-family + data — (A) bars + plain callouts (intuition), (B) index trace + one "O(n²)" line ("detail"). User rejected the claim: "that's not detail — all the detail was the index trace + O(n²), that's it."

**Proposed approach (under review):** The detail/depth layer of any lesson = the SAME figure rendered with a more technical skin + the trace + a complexity line, toggled via "show details"; presented as the template for all future lessons.

**Project context:** RO university-exam tutor; visual-learner student (ADHD) in finals; figures = LOCKED "family" architecture (family_id + typed data, render verified once, SVG in a gated step-through shell); lessons follow I-do→we-do→you-do; real lessons to be generated by an LLM pipeline (Plan 5, not yet built).
**Timestamp:** 2026-06-14 (session clock)

## 🔴 Devil's Advocate
STANCE: REJECT
REASONING: Category error — equates "detail" with a lower-level visual annotation of the same trace. Intuition skin and "detail" skin are both animations of ONE execution on ONE input — same thing at two zoom levels, not two KINDS of knowledge. Genuine depth (invariant + why it holds, complexity derivation, correctness, edge cases, stability, code, when-to-use) is largely NON-figural and INPUT-GENERAL — cannot be a render skin over per-element typed trace data, so the architecture structurally cannot represent it. Blessing this as the template hard-codes the shortfall: every subject inherits "detail = same picture, more labels."
KEY CONCERN: "Detail" modeled as a second RENDER of one figure (zoom), when the requirement demands a second LAYER of KNOWLEDGE (invariant, derivation, correctness, edge cases, code, trade-offs) that is non-figural — the family/typed-data architecture has no slot for it, guaranteeing shallow "detail" forever.

## 📚 Domain Expert
STANCE: REJECT
REASONING: Conflates a UI affordance (render-skin toggle) with a pedagogical depth layer. In the prior art, depth lives in OTHER modalities beside the figure: VisuAlgo pairs animation with a synced pseudo-code panel (highlighted executing line) + an explicit complexity-analysis section; 3Blue1Brown's visual is deliberately incomplete, always backed by the symbolic derivation; CLRS delivers depth via the loop-invariant (init/maintenance/termination), summation-based runtime derivation, and a correctness argument — none a "skin" of an array picture. Explorable explanations (Bret Victor "Up and Down the Ladder of Abstraction", Nicky Case) and worked-examples + faded-guidance research (Sweller/Renkl/Atkinson) build the intuition→rigor ladder from progressively-faded WORKED STEPS + explanatory text — genuinely different content per rung, not the same data re-styled. Selection sort's real depth (invariant "prefix [0..i) sorted and holds the i smallest", Σ(n−i)=n(n−1)/2, the unconditional-swap edge case, the pseudocode) is prose/proof/code by nature.
KEY CONCERN: The locked figure-family is the right substrate for the INTUITION layer (and the trace animation) but structurally cannot host the depth layer — depth needs first-class prose/proof, readable pseudocode with line-sync, and a complexity derivation as content types. Approving this as the template hard-codes a ceiling where "drill into technical detail" = a fancier picture; instead mandate a distinct DETAIL panel (code + invariant + derivation + edge cases) that the figure annotates, not replaces.

## ⚙️ Pragmatist
STANCE: CONDITIONAL
REASONING: The render-skin toggle is the cheap reusable 90% for the WRONG half and a false economy for the right half. A skin amortizes over instances of a family (every sort animates the same way), but genuine depth — invariant + why, complexity derivation, correctness, edge cases, tradeoffs — is concept-specific and does NOT ride the family; it's authored per concept regardless. Worse, it's exactly the content the Plan 5 pipeline is least able to GENERATE trustworthily: an LLM emitting a "proof"/"O(n²) derivation" as decorative text beside a verified figure inherits NONE of the figure's verification guarantee → confident-looking but unchecked math at scale, the highest-cost failure for exam prep. Keep the two-skin toggle for the intuition↔trace VISUAL layer (real, nearly free) but stop calling it "detail"; the depth layer is a separate concept-keyed content artifact with its own authoring budget + correctness gate.
KEY CONCERN: Conflating "visual detail skin" with "technical depth" makes depth look free when it's the most expensive + correctness-risky thing to author per concept — adopt the skin toggle, but budget and GATE a real depth artifact rather than let "show details = index trace + O(n²)" masquerade as the template.

## 🧱 First Principles
STANCE: REJECT
REASONING: Depth is not a render property — it's a different representation: the loop invariant + justification, why-it-terminates/why-correct, the cost derivation (why comparisons sum to n(n−1)/2 → O(n²)), the actual code, edge cases, why-this-not-that. "Intuition vs detail" is not one axis with two skins; it's ≥2 orthogonal axes — representation/modality (picture vs symbol vs code vs prose-argument) AND rigor (hand-wave vs proof) — collapsed into a false single slider. A richer skin + index trace + one O(n²) label gives a richer picture and a RESULT, never the DERIVATION or the ARGUMENT — exactly the exam-rigorous content the student must reproduce. For an ADHD visual learner in finals this is the worst failure: it FEELS like depth was delivered, so he stops, never meeting the invariant/proof he's tested on.
KEY CONCERN: Conflates "showing more of the figure" with "explaining why correct and why it costs what it costs"; the exam-rigorous layer (invariant + correctness + complexity derivation + code) is categorically absent, and a skin-toggle can never contain it because it's argument + symbolic derivation, not a picture.

## ⚠️ Risk Analyst
STANCE: CONDITIONAL
REASONING: Asymmetric failure: an honestly-thin lesson the student knows to supplement is recoverable; a lesson that LOOKS complete via "show details" but ships a shallow/possibly-WRONG depth layer manufactures false mastery — discovered in the exam room, unrecoverable. The figure-family verifies the FIGURE (pixels), zero guarantee over the prose/invariant/derivation/code in the "detail" skin → at pipeline scale, unverified depth claims propagate (wrong O-bound, mis-stated invariant, buggy trace) with no gate. Salvageable only if "detail" is redefined as verified technical content, not a cosmetic toggle, and the unverifiable claim never wears the verified figure's authority.
KEY CONCERN (CRITICAL): Verification-authority mismatch — a "show details" toggle visually certifies completeness/correctness the system cannot guarantee (family verifies pixels, not proofs/code). Mitigations: (1) block this as the universal template until "detail" = what a rigorous exam question demands per concept (real invariant + derivation + correct complexity argument), not a fixed index-trace+O(n²) garnish; (2) any auto-generated depth claim (complexity, invariant, code) MUST pass a verification gate distinct from figure-rendering, or be visibly flagged "unverified / check source"; (3) at scale, no lesson ships its detail layer as authoritative without that gate — default honest-thin-with-pointer over looks-complete-but-unverified.

CORE FINDING:
"Detail = a second render skin of the same figure + an O(n²) line" solves the wrong problem. The user is right: that is the same picture at two zoom levels, not two kinds of knowledge. Genuine technical depth — the loop INVARIANT and why it holds, the complexity DERIVATION (not the answer), the correctness argument, edge cases, the actual code, tradeoffs — is non-figural, input-general, and prose/proof/code by nature. The LOCKED figure-family architecture (typed data → verified SVG) structurally has no slot for it. Blessing two-skins as the template would hard-code "detail = fancier picture" across every subject.

AGENT CONSENSUS: 3 REJECT, 2 CONDITIONAL — 0 flagged. Unanimous that two-skins is NOT depth; the CONDITIONALs agree it's salvageable only by adding a separate, gated depth artifact.

KEY ISSUES:
- Category error (Devil's Advocate, First Principles): intuition↔trace is a VISUAL zoom (one axis); depth is a different modality+rigor (invariant/derivation/correctness/code). A skin toggle cannot contain argument or symbolic derivation.
- Depth lives in OTHER modalities (Domain Expert): the prior art (VisuAlgo, 3B1B, CLRS, explorable explanations, faded worked-examples) carries rigor in a synced code/pseudocode panel + complexity derivation + invariant + correctness text BESIDE the figure — never as a re-render of it.
- Verification-authority mismatch — CRITICAL (Risk Analyst): the family verifies PIXELS, not proofs/code. An auto-generated (Plan 5) depth layer inherits no guarantee; a "show details" toggle that looks authoritative but is shallow/wrong = false mastery, catastrophic for a finals student. Honest-thin-with-pointer beats looks-complete-but-unverified.
- Depth doesn't amortize (Pragmatist): unlike the intuition skin, depth is concept-specific, the most expensive + correctness-risky content to author per concept — must have its own budget + gate.

RECOMMENDED PATH:
1. KEEP the two-skin toggle — but rename it for what it is: an intuition↔TRACE visual zoom. It is NOT "detail." (The bars + trace work stands as a good VISUAL layer.)
2. Define the DEPTH layer as a SEPARATE, concept-keyed content artifact, first-class in the lesson model: loop invariant + justification · complexity DERIVATION · correctness argument · edge cases (duplicates, sorted, n=0/1, stability) · readable pseudocode/code (ideally line-synced to the figure, VisuAlgo-style) · tradeoffs / when-to-use.
3. Give the depth layer its OWN correctness gate, distinct from figure pixel-verification (e.g. complexity/invariant/code checks, or human/authored-source grounding). Until gated, mark depth content "neverificat / verifică sursa" — never let it wear the figure's verified authority. Default honest-thin-with-pointer over looks-complete.
4. This is a Plan-5 (lesson-generation) design requirement + a correction to the lesson-form model. Do NOT adopt two-skins as the universal "intuition+detail" template.
