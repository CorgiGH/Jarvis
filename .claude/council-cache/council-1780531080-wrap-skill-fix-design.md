## 🔴 Devil's Advocate — REJECT
The fail-loud post-condition is EMPIRICALLY broken, not fragile: the prior HEAD SHA legitimately lives in BRIDGE-HEAD.md's own Git-state line (cites the just-pushed range), the RECOVERY comment, the append-only BRIDGE-LOG (every historical SHA forever, by design), and MEMORY.md's session-stamped pointer — so grepping the prior SHA false-FAILS every wrap; and `resolved:` appears as instructional prose in wrap.md's own guard (lines 37/38/44/54) + active-constraints.md's header (line 3), so that grep false-FAILS too. Scoping tight enough to dodge all these is exactly tight enough to miss the one real stale line — trading a tolerable silent-skip for an always-red gate a tired low-experience Alex learns to override (killing the gate's credibility). Worse, "/wrap REWRITES active-constraints" is unsafe at root: that file holds HUMAN-stated hard lines NOT in BRIDGE-HEAD (PROPOSAL-GATE, the NEVER-rules, north-star) — a regenerate-to-match step silently destroys constraints Alex set by hand. And as a GLOBAL skill, a step assuming active-constraints + a git SHA + the `resolved:` convention breaks/no-ops noisily in projects that have none.
KEY CONCERN: The grep post-condition false-FAILS on every wrap (prior SHA is load-bearing in BRIDGE-HEAD's git-state/recovery + immortal in BRIDGE-LOG; `resolved:` is a literal token in the guards), and the paired "rewrite active-constraints to match" risks deleting the human-only hard lines that have no BRIDGE-HEAD source. Better served by a project wrap-probe.md + a SURGICAL "rewrite only the RESUME-RECIPE/Git-state/LOCKED blocks, never the human hard-lines below the `---`" rule.

## 📚 Domain Expert — CONDITIONAL
The three pieces are individually sound patterns: (1) sync-on-write = reconciliation/convergence; (2) the fail-loud grep = fail-closed assertion / poka-yoke, correctly placed since /wrap is the only writer of the derived region (same shape as Step 5's existing SHA self-verify); (3) routing non-checkable captures to wrap-probe.md = the "shared tool, per-project config" pattern, and the skill ALREADY loads wrap-probe.md at Step 1 (zero new mechanism). BUT the file is ALREADY physically partitioned (human hard-lines above the `---` at L30; BRIDGE-HEAD-derived RESUME-RECIPE/LOCKED/SHA below) — so the strictly-better design is achievable WITHOUT losing the human lines: make the derived region a delimited GENERATED BLOCK (`<!-- BEGIN/END derived-from-BRIDGE-HEAD -->`) that /wrap regenerates wholesale, while the human region above the marker is NEVER auto-touched. That's the canonical "single source of truth + idempotent regeneration of a marker-fenced section" pattern (Ansible managed blocks, gofmt, terraform fmt) — eliminates drift BY CONSTRUCTION, strictly dominant over hand-sync+grep for the derived part. APPROVE the wrap-probe routing + the human/derived split; CONDITIONAL on upgrading piece (1) from "rewrite to match" to "regenerate the marked block," at which point the grep gate is a fine belt-and-suspenders backstop.
KEY CONCERN: Idempotency + the regeneration boundary. "Rewrite to match" is under-specified about WHAT gets rewritten — whole-file clobbers the human lines; surgical edit is non-idempotent + makes the grep load-bearing forever. Pin the contract to a MARKER-FENCED derived block (whole-block regenerate, idempotent, provably can't touch the human region). Global-skill caveat: if a project's active-constraints has no markers, the skill must fall back to "don't auto-edit, just warn," never blind-overwrite.

The grep-and-paste, not the framing, closes the loop.

## 🧱 First Principles — CONDITIONAL
The cause is DUPLICATION: the RESUME-RECIPE/LOCKED/git-SHA block in active-constraints is a hand-maintained 2nd copy of BRIDGE-HEAD's Git-state + Hot-work + Open-items; the genuinely-separate content (north-star, numbered hard-lines, PROPOSAL-GATE) never drifts because nothing else holds it. The principled fix is to STOP duplicating the volatile state: split active-constraints into a stable human-hard-lines region the wrap never rewrites + a single delimited `<!-- AUTO:STATE -->…<!-- /AUTO -->` block /wrap regenerates as a literal slice of BRIDGE-HEAD (one writer, one source) — so "in sync" is true BY CONSTRUCTION, not by a re-sync step performed correctly each time. The proposed fix re-syncs two free-form copies + bolts on a grep to detect sloppiness = treating the symptom (two copies) not the cause (the recipe duplicates state at all); the fail-loud check is a smoke alarm, not a non-flammable design.
KEY CONCERN: A "rewrite to match" step still relies on the very model diligence that failed — the only correctness guarantee is making the synced region a mechanically-regenerated slice of the one canonical file (delete the duplicated state prose from the recipe), not asking a no-diligence model to re-type it + grep for the one stale token you remembered.

KEY CONCERN: The global-skill edit firing on the 10-of-11 no-active-constraints / 8-of-11 non-git projects — gate every new step on existence + git-repo FIRST, else a one-project resume-poison bug becomes an all-projects broken-wrap regression. Net-safer = YES with the gate, NO without.
