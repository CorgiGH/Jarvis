## üî¥ Devil's Advocate
AGENT: Devil's Advocate
STANCE: REJECT
REASONING: The boundary you drew is fiction: a worked example and a practice problem are the same epistemic object ‚Äî both assert facts (this is a valid instance, this property holds, this step is correct, this answer is right). Grounding the explanation while letting Llama improvise practice means the FALSE content moves from the explanation tier (where you fixed it) into the practice tier (where it's now load-bearing and ungrounded), and a wrong worked-example-disguised-as-practice teaches the omitted "halts in finite time" just as silently ‚Äî except now it wears the trust halo of sitting next to grounded prose. That breaks your one locked differentiator: the learner cannot tell which pixels are "matches your lecture" and which are an LLM guess, so the trust-net's promise is now a lie by adjacency, which is worse than honest improv-everywhere. And the hidden assumption ‚Äî that "explanation" and "example" are a small, author-once set while "practice" is the infinite-variety part ‚Äî is backwards: a curriculum's explanations ARE the long tail (every KC needs one), so Hybrid quietly commits the solo builder to authoring + faithful-checking grounded prose for the entire corpus, i.e. the exact content-treadmill that never ships, while Option 2's only "savings" was the part you just declared mandatory.
KEY CONCERN: Generated practice problems assert ungrounded facts under the trust-net's halo ‚Äî the explanation/practice boundary does not contain trust, so Hybrid ships a two-tier UX the learner can't distinguish, violating the locked "every served fact traces to the lecture" invariant.

## üìö Domain Expert
AGENT: Domain Expert
STANCE: APPROVE
REASONING: The hybrid split ‚Äî author + faithful-check the canonical explanation and worked example once, generate only the extra practice items ‚Äî is the dominant, battle-tested pattern in real ITS and courseware, not a compromise. Cognitive Tutor / Carnegie Learning and ASSISTments both human-author the instructional text and the canonical worked example (the "example-tracing tutor" model and CTAT authoring are built on hand-authored worked examples, per Renkl/Atkinson worked-example research), while reserving generation/parameterization for problem variants; Khan Academy serves a fixed authored explainer video/article per skill and only its "Get another problem" practice is templated/randomized; Anki, SuperMemo, and RemNote treat the card front/back (the canonical content) as authored and immutable, with the scheduler ‚Äî never an LLM ‚Äî varying when, not what. Your instinct that the explanation is the one artifact that must be grounded-once is exactly right: the explanation and worked example are the load-bearing truth claims (they assert lecture properties), so they belong under the same verbatim-trace / faithful-check / FAIL-LOUD regime as definitions and invariants ‚Äî this is the citation-required-generation / "grounded summary with extractive provenance" standard from RAG practice, where the canonical answer must be attributable and re-generating it each request is a known anti-pattern (non-determinism + silent property-drop, which is precisely the failure you observed). Generating practice problems live is the correct place to keep the LLM, because a practice item is checkable against the authored grader_rules/invariant ‚Äî its faithfulness is enforced at grade time, not asserted as truth, so ungroundedness there is bounded and self-correcting in a way an ungrounded explanation never is. Option 1 (ground everything) over-constrains a surface that is already validated downstream; Option 2 (improv everything) is the status quo you're correctly abandoning.
KEY CONCERN: The authored explanation + worked example must be held to the SAME faithful-check as the formal facts, not a weaker bar ‚Äî add an `explanation`/`worked_example` field to `KnowledgeConcept` (currently absent) that carries `SourceRef` spans and flows through the existing `verification_status` / faithful-check pipeline, and apply `grounding_tier: strict` (vision-confirmed spans) for any worked example that asserts a numeric/algorithmic property. Otherwise you create a two-tier trust system where the most-read surface (the explanation the student actually internalizes) is the least verified ‚Äî inverting the trust-net's whole premise.

STANCE: CONDITIONAL
REASONING: The hybrid is the proportionate choice precisely because the expensive half is already paid for ‚Äî curate-tutor + faithful-check already author and verify grounded prose, and the PA facts exist, so grounding the explanation is a one-time author-per-KC cost (6 KCs), not a recurring treadmill; the only treadmill would be re-authoring when a lecture changes, which is rare and is exactly when you WANT a re-check. Keeping live AI for extra practice problems is right: that's the high-variance, low-stakes surface where freshness beats faithfulness.
KEY CONCERN: Don't build a bespoke "faithful-check on prose" pass ‚Äî reuse the EXISTING faithful-check that curate-tutor already runs on KCs; if prose grounding needs a meaningfully different verifier than the fact-grounding you already have, the marginal cost stops being small and the proportionality argument weakens. Constrain to: nullable field (no migration drama for the 828-card live DB), no UI fork (explanation renders in the same study-loop surface Phase 5 ships), and a fallback-to-live-AI path so a not-yet-grounded KC still serves rather than 404s.

## üß± First Principles
AGENT: First Principles
STANCE: CONDITIONAL
REASONING: Stripping the framing: the learner wants understanding, which by definition requires rephrasing and adding intuition the lecture never states verbatim ‚Äî so "faithful to the WORDS" is the wrong primitive for an explanation, and the locked promise already knows this ("matches your lecture" is about the FACTS the source asserts, not its phrasing). The system's real primitive is already correct and already built: a typed CLAIM (DEFINITION/INVARIANT/GRADER_RULE/REFUTATION/STEM) bound to a verbatim span, audited by an independent non-LLM leg, and forced through `CitationGuard` at the emit boundary so no un-cited claim ships ‚Äî and `MisconceptionPayload.fromCited` already demonstrates the right pattern: generated/authored prose whose load-bearing claim crosses the citation chokepoint while the connective tissue stays free. So the honest cut is "every CLAIM is grounded + cited; presentation (ordering, analogy, scaffolding, faded-example narration) is free to be generated" ‚Äî NOT "explanation grounded / practice improvised." The proposed hybrid grounds the wrong axis: it conflates "is it generated" with "is it trustworthy" (which are orthogonal ‚Äî a generated explanation is trustworthy iff its claims are cited; a hand-authored practice problem with an un-grounded numeric answer is UNTRUSTWORTHY), and it leaves the actual leak open, because today's improvised explanation smuggles in NEW factual claims that never pass `CitationGuard`, while a generated practice problem's STEM/answer is itself a claim that must be grounded too.
KEY CONCERN: The decomposition axis is wrong. Don't fork on "explanation vs practice"; fork on "claim vs presentation." Decompose generated teaching prose into its asserted claims, route each through the existing `CitationGuard`/claim-audit machinery (extending `ClaimKind` to cover explanation-embedded claims and generated-STEM answers), and let the non-claim scaffolding be freely generated. Approve grounding the core, but reject "AI for practice = ungrounded" ‚Äî a generated practice item's stem and worked answer are claims and must be cited/audited exactly like an explanation's, or the trust-net has a hole on the side the hybrid declared safe.

## ‚öÝÔ∏è Risk Analyst
AGENT: Risk Analyst
STANCE: CONDITIONAL
REASONING: HYBRID is the right call, but the code as-built has a concrete trust-leak. A generated practice drill is NOT just a stem: `DrillContentDto` (DrillContentDto.kt:8) ships `worked` (a worked solution) and `definition` (a one-sentence concept definition) straight to the learner via `/tasks/{id}/prep`, and that DTO has NO verification/provenance field ‚Äî only the sibling `Problem.modelTag`, which never reaches the rendered drill card. So a free-LLM-authored worked solution and definition render in `DrillStack` under the same KC heading as the grounded explanation that carries the `faithful` "matches your lecture" badge (HonestFloor.kt:15, Phase1Tables.kt:127). The learner cannot tell which content the badge covers; the generator already demonstrably drops key lecture properties ("halts in finite time"), and its only grounding gate is the free-LLM critic's self-reported `grounded:bool` (DrillGenerator.kt:38) ‚Äî fail-closed on parse, but still one non-deterministic free model vouching for another, never re-checked against the source span the way grounded content is. The generated `definition`/`worked` is the leak vector far more than the stem. On the other risks the engine is sound: generation failure is genuinely fail-loud (`generate` returns a reject list / 502 BadGateway at TutorRoutes.kt:1561, malformed JSON ‚Üí null ‚Üí reject, no garbage drill is persisted), so a learner sees no drill rather than a wrong one; and authored-grounded explanations scale strictly BETTER multi-user (write-once, served from `task_prep`, cacheable) than per-request generation against a rate-limited `:free` model.
KEY CONCERN: CRITICAL ‚Äî trust-leak via badge ambiguity. The generated `worked` + `definition` are ungrounded teaching content with no per-drill trust marker, rendered beside grounded content under one KC-level `faithful` badge. Blast radius: every KC that runs `/generate-drills`, multiplied across users at scale, with a generator that provably omits lecture properties. Mitigation before ship: (a) restrict the generated drill to the STEM only ‚Äî drop or hard-gate `worked`/`definition`/`referenceSolution` in the generated `DrillContentDto`, since the explanation layer is exactly what HYBRID grounds; (b) add an explicit `provenance`/`generated:true` field to `DrillContentDto` and render a distinct, non-lecture badge ("AI practice ‚Äî not checked against your lecture") so the `faithful` badge can never visually span generated content; (c) add a CI invariant asserting no `DrillContentDto` served under a `faithful` KC carries an un-marked `worked`/`definition`. Until (b) lands, the differentiator is actively undermined, not merely incomplete.
