# Council review — 1781561170

**Problem:** Is the proposed 3-layer figure "overseer" the BEST setup for jarvis-kotlin's actual content, spec, and already-built code — given the just-ratified scope (full system, OOP-first deploy)?

**Proposed design (critiqued):** A 3-layer figure overseer. L1 CORRECTNESS (decidable hard CI gate, mostly built): figure = deterministic fn of algorithm execution; state-delta⇒figure-moves (frame-conjunction gate); no clip/overlap (svg-overlap gate); final==end-state. L2 SYMBOLIC FIDELITY (grounded estimator, NEW = "the overseer"): family/instance declares symbol-grammar + concept declares defining-dynamic; a second independent LLM grounded against the algorithm/source verifies symbol↔concept mapping + the FITNESS (family expresses the defining dynamic, fail-closed); can't ground ⇒ flag. L3 INTUITIVENESS (human, residual #1): owner ratifies a figure-TYPE once, then L1+L2 hold every future instance. Key claim: ratify a TYPE once → gates hold instances → ends the hand-polish loop without compromising quality.

**Project context:** single-user RO tutor, no paid APIs (free/relay only); scope = full system, push OOP/POO end-to-end + deploy FIRST; POO = 0 KCs / 0 figures (`.gitkeep`); all 5 built families are algorithm-trace; 2 gates are merge-sort-hardcoded + not CI-wired; the student cannot vet content.

**Timestamp:** 2026-06-16T~ (epoch 1781561170)

**Grounding note:** all 5 advisors independently opened the spec §5, the v2 plan Phase A/B/C, the diagnosis, the built families + trace-match harness + both gates, the content dirs, and the Hermes dive — verified (e.g. `git grep` zero hits for inheritance/UML/dispatch across spec+plan; POO dirs `.gitkeep`-only; gates merge-hardcoded). ## 🔴 Devil's Advocate
STANCE: REJECT
REASONING: The whole 3-layer overseer is induced from, and validated against, algorithm-trace evidence ONLY — the diagnosis's reasoning is "bars worked / merge-boxes failed," both array-state traces. L1's load-bearing mechanism is literally `typed algorithm-STATE delta ⇒ figure region must move`; L2's "defining dynamic"/FITNESS is computed against the algorithm definition/source span. Grounded: all 6 spec families are step-trace renderers; SortMergeFamily is a pure `MergeStep[]` reference-execution machine; POO = `.gitkeep` only; `git grep` returns ZERO hits for inheritance/method-dispatch/object-diagram/class-diagram/polymorphism/UML across BOTH spec and plan. So the design has no family, no "defining dynamic," no state-delta oracle, and no fitness mapping for the one subject scope ships FIRST. An OOP concept (class hierarchy, polymorphic dispatch, exception path) is not a deterministic-state-stepping trace, so L1 is inapplicable, L2 has no dynamic to ground against, and you fall back to L3 (human) for EVERY POO figure — the endless hand-polish loop this design exists to kill.
KEY CONCERN: "Ratify a TYPE once → gates hold every instance forever" is validated only for trace-shaped types, but POO-first means the very first shipped figures are NON-trace types with zero ratified precedent and no applicable L1/L2 machine gate — so for POO the design degrades to pure per-figure human ratification while wearing a "gated" label. Latent: tests green on PA/ALO traces, fails only when POO arrives (the ghost-class / deploy-gap failure mode).
Q1: Mis-scoped for POO-first; sharper split adds a 4th axis — figure ONTOLOGY (is the concept a deterministic state-trace at all?) decided BEFORE L1–3; routing a class/dispatch diagram through a trace family is merge-boxes one level up.
Q2: Partially groundable (symbol↔concept mapping is source-decidable), partially circular ("expresses the defining dynamic" only grounds when a dynamic exists; for static OOP it collapses to the unprovable-without-a-learner thing → model-judging-model, the autophagy risk).
Q3: LEAKS — L1/L2 are blind to "correct-but-unintuitive" by construction (residual #1); a ratified bars-type can spawn a 40-bar / bad-aspect-ratio instance that's trace-correct + overlap-clean yet defeats the at-a-glance read, uncaught.
Q4: Does NOT survive OOP-first — every family is a state-stepper, the concept_type enum has no object/inheritance/dispatch type, OOP visuals appear nowhere in spec/plan; OOP figures are structurally homeless (like the depth artifact was).

## 📚 Domain Expert
STANCE: CONDITIONAL
REASONING: The 3-layer split is real, known-good architecture and L1 is genuinely built, not vapor. The trace-match harness is golden/reference-oracle testing done correctly (each family re-runs the real algorithm from a seed and asserts every rendered frame == reference state — Manim object-continuity + property-based testing fused). frame-conjunction-gate is a legitimately sophisticated conjunction invariant. L3's "ratify a TYPE once" is the standard design-system component-contract / golden-template pattern, sound FOR THE FAMILIES THAT EXIST. CONDITIONAL because: (1) L2's grounded-second-LLM check is the weakest, least-established leg and the design over-trusts it; (2) the entire built corpus is algorithm-trace while deploy is OOP-first — OOP is exactly where trace-match breaks, with no family and no figures authored.
KEY CONCERN: "Ratify a type once → gates hold instances forever" is structurally true only for the trace-match leg (L1), NOT for L2 — and POO lives almost entirely in L2/L3 where the gates do NOT hold instances forever. Trace-match works because "run the real algorithm" is a decidable oracle independent of the figure; OOP/UML/inheritance has no such executable oracle. For POO every figure falls back to L2 (LLM fidelity) + L3 (human), and the headline promise silently does NOT transfer to the subject shipping first. Honest version: trace families auto-gate; non-trace families (all of POO) are estimator-bounded + per-instance human-checkpointed — the loop is bounded, not ended. The spec's FITNESS gate (fail-closed → "needs new family") must be the load-bearing mechanism for OOP, not the trace harness.
Q1: Name L1=reference-oracle(golden) testing, L2=rubric-graded generation w/ independent verifier, L3=component-contract/approve-the-template; ADD the missing pattern — grammar-of-graphics semantic-encoding declarations (Vega-Lite style): each family DECLARES its encoding channels (height↦value, color↦sorted-prefix, x↦index) as machine-checkable metadata, making "this concept needs a magnitude channel; this family lacks one" a DECIDABLE fitness check — pulling work down from L2 into L1.
Q2: Real but weakest leg; independent-verifier (never self-grading), fail-closed, grounded-against-source, second-family is the correct strong form (matches plan Phase-B/C + Hermes) — but ESTIMATOR not proof; inherit dim-14's "NOT GATE-OF-GATES" stamp + residual #1; strongest version grounds symbol↔concept against the declared encoding grammar FIRST, invokes the LLM only for the residual semantic claim.
Q3: Right pattern, leaks at the family boundary — holds inside a trace-match family (decidable per-instance oracle), leaks wherever the per-instance check is L2 (an estimator re-estimates fallibly, doesn't "hold forever"); don't let one-time L3 ratification launder an L2 instance as PROOF-gated.
Q4: Does NOT generalize — trace-match is a temporal-execution paradigm; OOP structure (class/object/inheritance/UML, association/composition) is static-relational. `state-machine/code-trace` (a spec family, unbuilt) absorbs BEHAVIORAL OOP (dispatch as code-trace), but there is NO structural/object-diagram/UML family in the 6 — a genuine spec gap for OOP-first. Treat OOP structure under FITNESS fail-closed "needs new family" and author a structural/relational family (verified by graph-isomorphism / relation-set assertions against the declared model, not temporal trace-match) BEFORE POO figures are shippable.
