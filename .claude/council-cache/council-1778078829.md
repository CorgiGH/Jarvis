## 🔴 Devil's Advocate

AGENT: Devil's Advocate
STANCE: REJECT
REASONING: Approach A treats the symptom (dominated tail) and leaves the bomb (a 7.3 MB / 190K-line wiki.md) sitting under every read path. `MemoryWiki.recent()` still has to scan the whole file to find `^## ` headers before filtering — every chat turn now pays that I/O + regex cost on a phone-tier VPS, and any other consumer (search, embeddings reindex, future `wiki.all()`, even a rogue `recent(100)`) ignores your filter and re-pollutes. Worse, you're hard-coding a title-prefix allowlist (`conversation (` / `daily reflection (`) as the load-bearing safety filter — the day you rename a section type or add `journal (`, `dream (`, `planning (`, chat silently drops them and you won't notice because the bug looks like "model forgot." And the hidden assumption: that 988 study files belong in wiki.md *at all*. They don't — they're a corpus, not a memory. Putting them behind a filter in the same file is pretending the ingest wasn't a mistake instead of reverting it.
KEY CONCERN: The 988 study files should never have been written to wiki.md — they're reference corpus, not personal memory. Revert the ingest (git checkout wiki.md or move SOURCE: sections to a separate `corpus.md` / `corpus/` tree the chat path doesn't read), then decide on retrieval. A title-prefix filter on a 7.3 MB file is a band-aid over a schema mistake.

## 📚 Domain Expert

AGENT: Domain Expert
STANCE: REJECT
REASONING: The "single growing markdown file + recency window" pattern is not what production personal-knowledge RAG systems use — it's a jarvis-specific shortcut that just hit its first scaling wall. Every reference system in this space separates *conversation memory* from *knowledge corpus* and uses an indexed store rather than `read+split` on each turn. **Obsidian Smart Connections** keeps a per-note JSON embedding cache and queries via cosine top-k, never recency-over-vault. **Khoj** runs a SQLite + sentence-transformers index per content type (markdown, org, pdf) and routes chat queries through semantic search with explicit content-type filters — exactly the conversation-vs-knowledge split jarvis is missing. **Letta (MemGPT)** is the canonical reference for the architectural pattern you actually want here: tiered memory with `core_memory` (always-in-context, small, mutable) + `recall_memory` (conversation history, FIFO/searchable) + `archival_memory` (knowledge corpus, vector-searched on demand). **mem0** does the same split with explicit user/session/agent scopes. **continue.dev** uses LanceDB locally for code+docs and keeps chat history in a separate SQLite table. **Cody local** uses sqlite-vss. Nobody does `tail -n 30` over a heterogeneous append-only file because the moment you mix content classes (chat turns + lecture notes + council transcripts + plans), recency stops being a meaningful signal — which is precisely the failure mode you just hit on ingest. Filter-by-title-prefix is a band-aid: it works until you ingest the next batch (research papers? journal entries? code snippets?) and you're back here adding another prefix or another flag. The 7.3 MB read+split per turn is also a latency/cost timebomb — at 190K lines you're already burning ~50-100ms per chat turn on file I/O + regex split + UTF-8 decode that production systems amortize to a single indexed lookup.
KEY CONCERN: Option A treats a *schema problem* (mixing conversation turns and knowledge documents in one append-only stream with a single retrieval strategy) as a *filtering problem*. Option B is closer to right but still wrong: it bolts vector search onto the same flat file instead of separating conversation memory (small, recency+FIFO, always-in-context summary) from archival knowledge (988 lecture notes, vector-searched only when relevant). The correct move is the Letta/Khoj split — `chat_log.jsonl` (conversation, recency window stays at 30, never contaminated) + `archival/` indexed via SQLite-FTS or sqlite-vss (lecture notes, retrieved on semantic match only) + a small mutable `core_memory.md` (the user's pinned identity/priority context, e.g. the finals-window rule). $0.20 of embeddings is fine; spending it on the wrong architecture is not. Ship A as a 24-hour stopgap if the chat is actively broken right now, but file the Letta-style refactor as the actual fix and don't let A become load-bearing.

## ⚙️ Pragmatist

AGENT: Pragmatist
STANCE: CONDITIONAL
REASONING: Approach A is a 10-minute change that unblocks chat tonight — that's the right move for a session-end fatigue moment, not a re-architecture. But the underlying design (one wiki.md as the universal store with prefix-string filtering as the routing mechanism) is already broken: you ingested once and it collapsed. In 3 months you'll have re-ingested twice more, the prefix list will have 4-5 entries, and somebody (you, drunk on a Saturday) will rename "conversation" to "chat" and silently break recall. Ship A as a band-aid with a TODO, but the real fix is structural: split storage by source — `conversations.jsonl` (chat-managed, append-only, the actual recent-N source) separate from `wiki/` directory (ingested reference material, search-only, never in recent()). That's ~1 hour of work and kills the class of bug permanently.
KEY CONCERN: recent(N)-by-file-order is the wrong primitive once the wiki is heterogeneous; filtering by title prefix is a load-bearing string match that will silently rot. Fix the storage split before re-ingesting again, not after the third breakage.

## 🧱 First Principles

AGENT: First Principles
STANCE: REJECT
REASONING: Stripped of framing, the real goal is "give me searchable knowledge of my own past output, surfaced when relevant." That's a corpus + retrieval problem, not a chat-context-window problem. From scratch, I'd build five distinct stores keyed by document type (conversations, reflections, study notes, council transcripts, plans) — because they have wildly different access patterns: conversations need recency, study notes need semantic match on demand, council transcripts need by-topic recall, reflections need temporal scan. The single growing wiki.md conflates all five into "append-only log being abused as chat memory, searchable corpus, and knowledge base simultaneously" — which is incoherent past ~50 entries and is exactly why the last-30-sections heuristic now returns lecture pseudocode. The proposed title-prefix filter is a symptom patch on a category error: it preserves the wrong-shape abstraction and will break again the next time a new doc-type gets dumped in.
KEY CONCERN: wiki.md is being used as four different abstractions at once (log, corpus, chat memory, KB) — split it into typed stores with type-specific retrieval (recency for conversation, on-demand semantic for study notes triggered by /quiz or explicit query, never always-on injection); the prefix-filter fix locks in the wrong shape.

## ⚠️ Risk Analyst

AGENT: Risk Analyst
STANCE: CONDITIONAL
REASONING: Top risk is HIGH-severity recurring O(file-size) work on the hot path: every chat turn reads + regex-splits 7.3 MB and allocates a 988-element list under -Xmx512m, with MC already consuming 4.2 GB of 7.8 GB RAM — two concurrent chat turns can transiently allocate ~30+ MB of strings/arrays and trigger GC stalls or OOM in the jarvis JVM. Fix A reduces the *output* slice (30→5) but does NOT fix the read+split cost, which still scales linearly with wiki.md and will degrade further as the user keeps writing notes. Second risk is MEDIUM: concurrent unlocked appends to wiki.md from /api/wiki POSTs can interleave bytes mid-section and silently corrupt the regex-split boundary, poisoning all future retrievals — the ingester just shipped 988 files so this code path is now load-tested adversarially. Title-prefix filter fragility is LOW (single string, one place to grep) and the OPENROUTER secret risk is LOW given existing bearer-auth + .env pattern.
KEY CONCERN: HIGH — MemoryWiki.recent() must stop reading the full file per turn. Mitigations required before merge: (1) cache the parsed section index in memory, invalidate on append, OR tail-read the last N KB only; (2) add a single-writer mutex / file lock around /api/wiki append path to match activity.jsonl's single-writer guarantee; (3) add one test asserting [SOURCE: filter actually excludes ingested sections so silent retrieval degradation is caught.

VERDICT: WRONG APPROACH

CORE FINDING:
All five clean agents independently converged on the same root cause: this is a SCHEMA problem, not a retrieval problem. Mixing 988 study notes, conversations, reflections, council transcripts, and plans into one append-only `wiki.md` and reading the entire file via `recent(N)` was already the wrong abstraction at ~10 entries; the ingest just exposed it. Approach A (filter-by-title-prefix + smaller N) preserves the broken shape. The correct move is the Letta/MemGPT/Khoj pattern that every production personal-knowledge RAG system uses: separate conversation memory from archival knowledge, with type-specific retrieval per store.

AGENT CONSENSUS: 3 REJECT, 2 CONDITIONAL — 0 agents flagged. The two CONDITIONALs (Pragmatist, Risk Analyst) accept A only as a sub-day stopgap behind a "real fix is structural" requirement. No clean APPROVE.

KEY ISSUES:
- **Schema mismatch (Domain Expert, First Principles, Devil's Advocate):** wiki.md is doing 4-5 incompatible jobs at once (chat memory, append log, searchable corpus, knowledge base). recent(N) by file order stops being a meaningful signal once content classes are mixed.
- **HIGH-severity hot-path cost (Risk Analyst):** 7.3 MB read + regex-split per chat turn under -Xmx512m JVM with MC eating 4.2/7.8 GB — concurrent turns can OOM or GC-stall. Approach A doesn't fix this.
- **Filter fragility (Devil's Advocate, Pragmatist):** load-bearing string match on `[SOURCE:` / `conversation (` / `daily reflection (`. Silent rot the next time a section type is renamed or a new doc-type is ingested.
- **Concurrent-append corruption (Risk Analyst):** /api/wiki has no write lock; bytes can interleave under load and break section regex permanently.

RECOMMENDED PATH:
1. **Today (immediate, ~30 min):** Revert the wiki.md ingest. Move the 988 SOURCE-tagged sections OUT of wiki.md into a separate tree on the VPS (e.g. `/opt/jarvis/data/archival/<source>/<rel-path>.md`) so the chat path stops seeing them. Keep the original markdown files on disk for later indexing. Restore wiki.md to its pre-ingest state (which had the 7-10 real conversation/reflection entries) so chat works again immediately.
2. **This or next session (~2-4 hours):** Implement the Letta-style split:
   - `conversations.jsonl` — append-only, chat-managed, the only thing `recent(N)` reads. Cap entries (FIFO truncate at e.g. 1000) to bound retrieval cost.
   - `archival/` — directory tree of ingested markdown, indexed via SQLite FTS5 or via the existing embeddings.json (after `runReindex` on VPS). Retrieved only on explicit semantic match or via a `/sub search <query>` subsystem call. Never auto-injected into chat context.
   - `core_memory.md` — small (≤4 KB) mutable file for pinned user context (finals window, identity, current priorities). Always in chat system prompt.
   - Add a write-mutex around any append path that touches the file/store.
3. **Before re-ingesting anything:** Run a regression check that recent() returns conversations only.

Approach A as proposed should NOT ship even as a stopgap, because it locks in the wrong primitive AND leaves the HIGH-severity I/O cost unmitigated. If chat is broken right now, the faster fix is to revert wiki.md (step 1) — that takes ~5 minutes and produces a clean state — rather than apply a filter that masks the problem.
