## 0. The thing the old playbook never did: SCORE WHAT ALREADY SHIPPED (Track A, step zero)

**This is the single most important move and it comes before every gate below.** Every other section prescribes gates for *future* viz. But "never wow + endless minor fixes" is a claim about the **24 components already on disk** — and the old playbook never once looked at them. You cannot fix a quality verdict you haven't measured.

**Do this first, as a one-time retrospective pass:**

1. **Inventory + mount-audit.** List all 24 components in `src/components/viz/`. Cross-reference `vizRegistry.ts`: today it maps **1** id. That means **up to 23 components are ghosts** — built, possibly tested, never reachable from the route a student hits. This is the *exact* failure the project has scar tissue for (global `CLAUDE.md`: 2026-05-11 Slice-1 ghost-component + Slice-1.5 PDF-404). **Before scoring aesthetics, answer: which of the 24 does a student actually see?** For each component, find the route/id that renders it; if none, it is a ghost — flag it `GHOST` and decide keep-and-wire vs. delete. A gorgeous component nobody mounts is worth zero.
2. **Retrospective taste + correctness scoring.** Run the taste rubric (§4) AND the pedagogical-correctness check (§3.6 — the most important gate) against each *mounted* component via the self-seeing loop (§3). Produce a ranked table: `component | mounted? | taste avg | correctness pass? | top fix`. This table is the actual backlog. It replaces guessing about "wow."
3. **Calibrate the taste threshold from this table, not from a magic number.** Pick your 3 best-looking current components, score them, set the gate floor *just below the worst keeper*. A hardcoded "≥3.5" (old playbook) either passes everything or blocks everything because it was never anchored to this repo's real output.

Output of step 0 is a ranked backlog + a calibrated threshold. Everything downstream operates on that.

## 1. The diagnosis — name the two failure modes precisely

**Failure A — "works but never wow."** Concrete root causes, not vibes:

- **Wrong perceptual channel for the load-bearing variable.** The ok→wow gap is almost always encoding the most important data attribute with a *low-effectiveness* channel (area, hue) instead of the highest-ranked one (spatial position, length). Munzner's channel ranking — position >> length >> angle >> area >> saturation >> hue — is the single most load-bearing finding [https://www.cs.ubc.ca/~tmm/vadbook/ (Munzner, *Visualization Analysis & Design*, ch. 5 "Marks and Channels")]. An agent left to its own devices reaches for color because color is "easy JSX." That's the wow-killer.
- **Pointless animation that competes for attention.** 3Blue1Brown's discipline: every animation must have a nameable purpose or it harms comprehension [https://www.3blue1brown.com/about/]. AI agents add entrance fades, idle pulses, and easing on chrome by default — extraneous cognitive load, doubly toxic for an ADHD learner [https://www.artmiker.com/post/neurodiverse-animation-principles-for-adhd-autistic-audience].
- **No "concrete → abstract" arc.** Viz consumed passively never becomes a thinking environment; the insight lives in the *transition* between a concrete instance and the abstract pattern [https://worrydream.com/ExplorableExplanations/]. Agents build one static frame and stop. **Audit which of the 24 lack the abstract pole** (step 0) rather than asserting it abstractly.

**Failure B — "endless minor fixes" (the iteration tax).** TWO root causes, and the old playbook under-weighted the second:

- **No mechanical floor.** Nothing rejects `pl-[17px]`, `bg-[#FFDD00]`, a contrast violation, or a misaligned element *before* it reaches you. Each becomes a manual round-trip — a lint-error class masquerading as a review class [https://github.com/francoismassart/eslint-plugin-tailwindcss/blob/master/docs/rules/no-arbitrary-value.md].
- **The agent is blind.** Text-only self-refine cannot catch z-index, overflow, animation-fill-mode, broken layout, or contrast — it edits code it never looked at. Each blind edit is a coin-flip → N round-trips instead of 1.
- **(The under-weighted one) 24 bespoke components with no shared layout/animation primitives.** Each fix is *local* because there's no system — fixing spacing in `RecursionTree` does nothing for `BayesTree`. `motion-helpers.tsx` is the **seed** of a shared layer (it already abstracts Motion-SVG). The highest-leverage refactor is **consolidate the 24 toward N shared primitives so one fix propagates** — NOT inventing a brand-new spec system (Track B) before harvesting the abstraction that already exists.

**Fix order (forced):** (0) score + mount-audit what shipped → (Failure B) kill the mechanical tax (deterministic, cheap) and consolidate primitives → (Failure A) close the perception/taste gap (judgment; needs a rubric + the agent's own eyes) → and threaded through all of it, the **correctness gate** (§3.6), without which a beautiful wrong viz passes everything.

## 2. Kill the minor-mistakes tax (Track A, mechanical gates)

Mechanical, deterministic, cheap. Add before any "make it prettier" work. Wire in this order.

**2.1 — Design tokens as the only legal vocabulary (Tailwind v4 `@theme`).** You already have an `@theme` block at `src/index.css:45` — good, but it is **not enforced**. Make the three-layer hierarchy explicit: base primitives (`--color-yellow-400: oklch(...)`) → semantic (`--color-accent: var(--color-yellow-400)`, `--color-surface`) → component (`--btn-bg: var(--color-accent)`) [https://tailwindcss.com/docs/theme]. The rule that makes refactors safe: *never skip the semantic layer.* In v4 each token auto-generates a utility, so `bg-accent` exists and `bg-[#FACC15]` becomes a **detectable** violation.

**2.2 — ESLint as the mechanical lock (single highest-ROI add — you have NO ESLint today).** Add `eslint` + `eslint-plugin-tailwindcss` with `tailwindcss/no-arbitrary-value: 'error'` (error, not warn) and run it in a pre-commit hook via lint-staged [https://github.com/francoismassart/eslint-plugin-tailwindcss/blob/master/docs/rules/no-arbitrary-value.md]. Note: `eslint-plugin-tailwindcss` v4 support is still maturing — verify the installed version reads your `@theme` tokens before relying on token-awareness; if not, the `no-arbitrary-value` rule still works on its own and is the load-bearing one. This *alone* kills the "agent invented a spacing/color value" round-trip class — output is rejected before it reaches you. Also ban `rounded-*` (brutalist = `radius:0`) so shape drift is a lint error, not a review note. Pair with `eslint-plugin-jsx-a11y` for static a11y.

**2.3 — Contrast/a11y as a hard gate on the real rendered DOM.** You have `vitest-axe` and `axe.drillstack.test.tsx` / `axe.fsrsreview.test.tsx` — **extend them to the mounted viz components**. The decisive gate is axe-core's `color-contrast` rule run against the *rendered* DOM (tests the real computed yellow-on-dark value, not class names) [https://github.com/dequelabs/axe-core]. This is the only gate that catches "text is unreadable" before you see it — critical for a yellow-on-black brutalist palette living on the WCAG-AA edge. **Caveat for this stack:** KaTeX output is notoriously overflow/contrast-prone and Plotly renders to canvas/WebGL where axe's DOM contrast check does not reach — see §6c for how those two surfaces are gated differently.

**2.4 — Per-component pixel gate (Playwright `toHaveScreenshot` — you have ZERO baselines today).** Catches alignment, overlap, whitespace, and contrast shifts no unit test sees; **free, zero external service** — fits no-paid-API exactly [https://playwright.dev/docs/test-snapshots]. Specifics that matter:
- Scope to a locator, not full page: `expect(page.locator('[data-testid="viz-canvas"]')).toHaveScreenshot('bfs-step-3.png', { animations: 'disabled' })`.
- **Do NOT use `maxDiffPixels: 0`.** The dev loop runs on **Windows 11** (per env) while deterministic baselines must be generated in the official Playwright **Docker image** to kill font/AA drift — a zero-tolerance baseline will red-fail *every local Windows run* and manufacture exactly the minor-fix churn this gate exists to kill. **Rule: baselines are CI/Docker-only; set a small tolerance** (`maxDiffPixelRatio: 0.01`) so sub-perceptual AA noise doesn't trip it; treat local Windows screenshot runs as advisory, not gating [https://playwright.dev/docs/test-snapshots].
- `animations: 'disabled'` fast-forwards finite CSS animations and cancels infinite ones — but **CSS cannot pause SVG SMIL.** For any `<animate>`/`<animateTransform>`, attach a ref and call `svgEl.pauseAnimations()`. (Most viz here uses Motion/WAAPI, which `animations:'disabled'` handles; SMIL is the edge case.)
- Set `use: { reducedMotion: 'reduce' }` globally in `playwright.config.ts` — also correct accessibility behavior [https://www.joshwcomeau.com/react/prefers-reduced-motion/].

**2.5 — Storybook + auto-fan-out — ADD ONLY AFTER §2.1–2.4 + the taste system are stable (it is `partial-fit`, not top-tier).** The honest tradeoff: for 24 components with zero stories today, "write the story once" means **24 new files** — for a solo ADHD dev that is *more* to maintain, not less, and the evidence set rates Storybook **`partial-fit`: worth adding only after the token/rubric system is stable.** The payoff is real (story → free Storybook-`index.json` → Playwright loop iterates every story, `toHaveScreenshot`) [https://markus.oberlehner.net/blog/running-visual-regression-tests-with-storybook-and-playwright-for-free], and stories map naturally to algorithm states (BFS step-0/3/complete). But sequence it **last** among mechanical gates, and only commit to it if step-0 scoring shows you'll keep enough components to amortize 24 story files. A lighter interim: a single `VizDemoPage`-style route that renders each mounted component, screenshot-gated directly — you already have `VizDemoPage.tsx`.

> **Do NOT** reach for Chromatic / Percy / Applitools (paid, conflicts with no-paid-API). The free Playwright path covers the same gate.

## 3. The self-seeing agent loop — make the agent LOOK (Track A)

Collapses iterations from N to 1–2: the agent must *see its own render.* Substrate already present: Playwright MCP (`mcp__playwright__browser_*`) + Vite dev server on `:5173` + Claude vision.

> **One toolchain, not two.** Use **Playwright MCP** for the whole loop. The old playbook bolted on `@playwright/cli` for a single, unreplicated vendor-blog token number; that splits the toolchain (every other step here — conditional probe, bbox→selector, three-frame capture — assumes MCP) for no verified gain. **Cut it.** If unattended token cost ever becomes a measured problem, revisit then, with your own measurement.

**3.1 — Conditional-screenshot pattern (token discipline first).** Don't screenshot blindly. **Phase 1:** `browser_evaluate` runs a diagnostic JS snippet returning structured JSON — `scrollWidth > clientWidth`, animation running-state, SVG `viewBox` vs rendered size, `animation-fill-mode === 'forwards'`, any element clipping an in-flight animation with `overflow:hidden`. **Phase 2:** screenshot + vision critique *only if* the JSON shows anomalies [https://dev.to/ohugonnot/zero-overflow-in-10-minutes-responsive-testing-with-claude-code-and-playwright-59an]. This skips vision calls on already-correct renders — directly extends how many iterations fit inside the OpenRouter `:free` / relay budget. Keep **pass-before-advance**: the loop refuses to move to the next viewport/step until the current one returns an empty anomaly list.

**3.2 — Three-frame temporal capture for animation correctness.** A single screenshot can't catch a transition that stalls or overshoots. Capture at **t=0, t=500ms, t=1000ms**, feed all three to Claude as one vision call with a 4-dimension rubric: (1) algorithm state shown matches the step description, (2) layout not broken, (3) no overlapping elements, (4) animation reaches final state by the last frame. This is ReLook's multi-frame critique architecture, with Claude vision standing in for the paper's VLM at zero cost [https://arxiv.org/html/2510.11498v1]. *Note on transfer:* ReLook's headline lift was measured on **generic web-generation** benchmarks, not algorithm-viz on this stack — treat the *loop structure* as the transferable asset, not the percentage.

**3.3 — Forced-optimization gate (the anti-thrash rule).** The #1 thrash failure is the agent cycling between two equally-broken states, each called a "fix." ReLook's remedy: **accept a revision only if its score strictly exceeds the prior best; resample up to ~10×, then halt** and surface to you [https://arxiv.org/html/2510.11498v1]. Encode this literally.

**3.4 — Logical/spatial correctness via assertions, not vision (cheaper, zero hallucination).** Algorithm animations have *spatial* ground truth: bar at index 2 must be taller than index 3 at step 3; edge (A,B) must have `stroke-dashoffset=0` after 800ms. Write MoVer-style JS predicates, run in `browser_evaluate` after each revision, feed failures back as the critique [https://arxiv.org/abs/2502.13372]. Split the labor: **assertions gate spatio-temporal truth; vision gates aesthetics.** *Transfer caveat:* MoVer's one-shot→loop gain is on a generic SVG-animation benchmark; the *predicate-correction mechanism* is what transfers. **Crucially, these assertions only check the animation matches its OWN declared steps — they do NOT check the steps are pedagogically correct. That gap is §3.6.**

**3.5 — Spatially-grounded critique so feedback is actionable.** "The spacing is off" is unactionable — the agent can't locate the element. Require the critique to emit JSON per issue: `{issue, element_description, approximate_bbox:{x,y,w,h} as %, suggested_fix}`, then map bbox→selector via `browser_evaluate('document.elementFromPoint(x,y).className')` [https://arxiv.org/pdf/2412.16829]. Your brutalist high-contrast palette *helps* MLLM grounding — dark/light boundaries localize far better than gray gradients. The bounding-box requirement is what makes the critique map to a code change instead of a vibe.

**3.6 — THE PEDAGOGICAL-CORRECTNESS GATE (the most important gate in this document; net-new vs. the old playbook).**

Every other gate makes a viz *mechanically clean, perceptually well-encoded, aesthetically on-brand, and internally self-consistent.* **None of them checks that the visualization teaches the CORRECT thing.** For a tutor whose user (per project memory: *"Alex can't vet content or name a subject's pedagogy"*) **cannot vet subject content himself**, a polished, brutalist-yellow, WCAG-passing, pixel-locked animation of a **subtly wrong algorithm** sails through every gate above — and is *more* dangerous than an ugly one, because it looks authoritative. Wow comes from a learner *trusting and understanding*; a correct-but-plain viz earns that, an incorrect-but-gorgeous one destroys it. This is the real lever on "never wow."

**The correctness oracle (two layers):**

1. **Trace-match against a reference implementation.** For any algorithm viz (BFS frontier order, recursion tree of a recurrence, scheduler Gantt, page-table walk, TCP cwnd), write a tiny **reference implementation of the algorithm in plain TS** and have it emit the canonical step trace. Then assert the viz's step sequence **matches that reference trace** — not just that the animation is self-consistent. A BFS viz whose frontier dequeues in DFS order fails here even though every MoVer spatial assertion passes. This is the difference between "the animation does what it says" and "what it says is true."
2. **Content review against the source.** Each viz illustrates a Knowledge Concept (KC) sourced from a lecture PDF in `content/`. Add a review pass: feed the viz's claimed teaching point + the source KC/PDF excerpt to Claude and ask "does the visualization's claim match the source, and is anything pedagogically misleading?" This catches the class of error a reference implementation can't (wrong intuition, mislabeled axis, a correct algorithm explained with a false *reason*).

**Apply this BACKWARD first** (per §0): score the mounted subset of the 24 for correctness before building any forward gate. An incorrect shipped viz is the worst bug in the system and currently has zero detector.

## 4. The taste gate — encode "great" as a repeatable target (Track A)

Mechanical gates make output *correct*; the taste gate makes it *intentional*. All no-paid-API, Claude-Code-native.

**4.1 — `DESIGN.md` as the single machine-readable brand source (do this once).** Drop a `DESIGN.md` (Google Labs, Apache-2.0) at repo root: YAML front-matter holds tokens with curly-brace path refs (`{colors.accent}`) so the agent *must cite a named token or the reference is invalid*; Markdown holds Do/Don't rationale [https://github.com/google-labs-code/design.md]. Mirror its YAML 1:1 into the Tailwind v4 `@theme` block at `src/index.css`. Brutalist-yellow: `--color-accent: #FACC15`, `--color-surface: #0A0A0A`, `--color-text: #F5F5F5`, all radii `0`, monospaced display font; Markdown "Do not" bans gradients, drop-shadows, rounded cards, Inter/Roboto. Every agent prompt opens: *"Read DESIGN.md. Use only named tokens. Arbitrary values are FORBIDDEN."*

**4.2 — Two-anchor LLM-as-judge rubric.** You don't need a 5-level rubric: describing only score-1 (worst) and score-5 (best) is about as reliable as describing all five, and *criteria matter more than reference answers* [https://arxiv.org/html/2506.13639v1]. Write exactly two anchors:
- **Score 1:** "Generic AI output — Inter font, blue/purple/teal palette, `rounded-lg` cards, three equal columns, gradient backgrounds, drop shadows. Looks like every SaaS landing page."
- **Score 5:** "Swiss brutalist — monospaced everywhere, single `#FACC15` accent on `#0A0A0A`, full-width hard black borders as dividers, asymmetric grid, zero shadows, zero gradients. A 1960s Swiss poster remixed for a terminal."

Run the evaluator **3× at temperature 0.7 and average** (averaging stabilizes drift across sessions) [https://arxiv.org/html/2506.13639v1]. **Set the gate by calibration, not a magic number:** from §0, score your 3 best existing components and put the floor *just below the worst keeper* (the old playbook's hardcoded "≥3.5" was unanchored and would pass-all or block-all). Sonnet as judge is sufficient — no Opus needed.

**4.3 — Generator/evaluator separation with named dimensions (Anthropic's pattern).** Two messages, GAN-style. Generator: build with DESIGN.md tokens only. Evaluator (after Playwright screenshot): score **Design Quality / Originality / Craft / Functionality**, each 0–4, target ≥3, and **produce a detailed critique, not pass/fail**, so the generator has a targeted repair [https://www.anthropic.com/engineering/building-effective-agents]. Two concrete artifacts to make this actionable in *this* harness (not just a slogan):
- **The directional phrase** lives as the first line of the generator's system prompt in the `grounded-ui-design` skill: *"The best designs look like Swiss International Style posters, not SaaS landing pages."* Inject it before any constraint — it steers the prior.
- The evaluator reviews the **`vizSpec`/component plan before code is written**, not only the render after.

**4.4 — Binary consistency locks as a pre-flight.** Three locks the agent self-checks before claiming done: **Color lock** (only the 3 semantic tokens), **Shape lock** (`radius:0`, zero exceptions), **Theme lock** (dark-brutalist, no toggle). Tailwind v4 enforces color mechanically (only 3 tokens exist in `@theme`); shape via banning `rounded-*` in ESLint. The pre-flight is a 2-second ESLint scan + a regex for arbitrary colors.

> **Wiring note (corrected):** you have the **`grounded-ui-design`** project skill installed (`.claude/skills/grounded-ui-design/`). **`review-site` is NOT a project skill** — it is a globally-listed skill and there is no verified contract by which either accepts or exposes a taste rubric. So: **wire this rubric INTO `grounded-ui-design`'s prompts yourself** (it is yours to edit); do not assume a plug-in point that hasn't been verified.
>
> **Skip / verify-before-trusting:** third-party "taste" skills (Impeccable, Taste Skill, etc.) are sourced only to their own marketing pages — no independent confirmation they are free or that they install in this harness. Given the project's documented `gws`-hallucination scar (a non-existent npm package was trusted for 5+ commits), **do not recommend any external skill without first verifying the package/skill actually exists and installs.** The two-anchor averaged-Sonnet path above needs no external dependency and is the recommendation.

## 5. Steal-these exemplars + motion principles (concrete, for Track A and B alike)

**Pedagogy patterns to copy into component specs:**
- **Predict-then-reveal on every multi-step viz** — captures self-explanation + testing effects, beats passive viewing [https://distill.pub/2020/communicating-with-interactive-articles/]. Tiny state machine `{phase:'predict'|'reveal'}`; prediction UI = 2–3 brutalist-yellow buttons; reveal = the animation. Wrapper: `<PredictReveal options={...}>{stepViz}</PredictReveal>` — deterministic from algorithm state, no API. **This is a shared primitive — build it once, reuse across the 24, NOT per-component.**
- **Staged complexity, sandbox LAST** (Nicky Case): isolated mechanic → combined → controlled-parameter → sandbox, gated as `?phase=1..4` URL state [https://blog.ncase.me/explorable-explanations/]. Parable of the Polygons is the structural exemplar; its flat-shape/visible-grid aesthetic already matches your stack [https://ncase.me/polygons/].
- **Concreteness-first two-frame pattern** (3b1b Linear Algebra): Frame 1 animates one specific case, Frame 2 generalizes — *same layout, same vocabulary* so the model transfers [https://www.3blue1brown.com/topics/linear-algebra]. Build as `<ConcreteCase/>` then `<GeneralCase/>` sharing one state slice. **In §0, audit which of the 24 have only the concrete pole and need the abstract one added** — don't assert the arc in the abstract.
- **Pseudocode-highlight-in-sync** (VisuAlgo): `<PseudocodePanel code={lines} currentLine={step.lineNumber} />`, yellow-highlight the executing line; copy VisuAlgo's step/pause/speed control strip as the minimum control set [https://visualgo.net/en].
- **Synchronized multi-view** (Victor's Ladder of Abstraction): a concrete view + an abstract view both read the same `step` state, updating in lock-step [https://worrydream.com/LadderOfAbstraction/].
- **Prose-anchored interaction** (Distill): every interactive figure is cited in surrounding text saying what to manipulate and what to notice [https://distill.pub/2017/feature-visualization/]. **Make this a named field**, not a vibe: add `proseHook: string` to each viz's metadata (the one-line "what to notice" the surrounding tutor text must reference), and lint that it's non-empty for any mounted viz.

**Motion numbers to hard-code as Tailwind `@theme` tokens (reference tokens, not magic numbers):**
- **Durations:** `--duration-fast: 200ms` (hover/click), `--duration-base: 300ms` (state changes), `--duration-slow: 450ms` (panels). Below 200ms imperceptible; above 500ms breaks flow-of-thought [https://valhead.com/2016/05/05/how-fast-should-your-ui-animations-be/]. Never a flat 300ms everywhere.
- **Easing (Material):** `--ease-out: cubic-bezier(0,0,0.2,1)` *entering*, `--ease-in: cubic-bezier(0.4,0,1,1)` *exiting*, `--ease-standard: cubic-bezier(0.4,0,0.2,1)` point-to-point. CSS `ease`/`linear` are rarely correct for UI; exiting elements get *shorter* durations [https://m3.material.io/styles/motion/easing-and-duration].
- **Staging/stagger:** primary element first (200ms), secondary targets stagger **60–80ms** each (`staggerChildren: 0.07, delayChildren: 0.2`). Education exception: **two panels starting at the identical time = the explicit claim "these are the same event"** (tree rearrange + array swap) — start them in the same `useEffect` flush; everything else staggers.
- **Step control beats easing polish:** play/pause/**step-back**/speed (fast≈9ms, normal≈60ms, slow≈200ms per frame), current-step counter visible. A pausable ugly stepper teaches better than a beautiful auto-player that can't stop [https://hinsxd.dev/blog/sorting-visualizer]. Emil Kowalski's gate for *every* animation: "does this animate where it enriches information or indicates state change?" — if no, cut it; never animate keyboard-repeated actions [https://emilkowal.ski/ui/great-animations].
- **Architecture to steal:** ES6 generator-per-step (`function*` yields `{array, colors, highlightedIndices}`) + a `useRafLoop` delta-accumulator clock — zero deps, decouples algorithm logic from render timing [https://hinsxd.dev/blog/sorting-visualizer]. This is *also* the natural place to attach the §3.6 reference-trace check (the generator's yields ARE the trace).
- **ADHD/yellow rule:** `#FACC15` is the current-focus highlight **only**, one element at a time; everything inactive is monochrome and motionless. No idle/ambient animation, ever [https://www.artmiker.com/post/neurodiverse-animation-principles-for-adhd-autistic-audience].
- **`prefers-reduced-motion` — EXTRACT THE EXISTING CONVENTION, DO NOT REINVENT.** The repo already handles it in **8 components** with a consistent `matchMedia` pattern. The move is **not** "write a `useReducedMotion()` hook" — it is **extract the existing 8-site pattern into one shared hook and migrate all 8 + every new viz to it.** `prefers-reduced-motion` is *reduce, not eliminate*: static by default, motion inside `@media (prefers-reduced-motion: no-preference)`; reduced fallback = instant opacity/color change (opacity isn't vestibular) + a manual in-UI localStorage toggle since ADHD users may want motion off regardless of OS setting [https://www.w3.org/WAI/WCAG21/Techniques/css/C39]. (Motion's own `useReducedMotion()` already exists; the existing 8 sites predate consolidating on it — unify them.)

## 6. Motion toolchain for THIS stack (Track A — use what's installed)

**Verdict: `motion` (Motion, formerly Framer Motion) is the layer — already installed (`motion ^12`) AND already abstracted in `motion-helpers.tsx`.** Nothing to add for the common cases. It is declarative-first (diffable JSX props = agent-friendly + git-citable), ships `motion.rect/circle/path/g` with native `pathLength`/`viewBox`/`d`, runs via WAAPI up to 120fps without React re-renders, has `layoutId` FLIP for swaps, `<Reorder.Group>` for sorted-array viz, `useReducedMotion()` built in, and `useAnimate()` as an imperative escape hatch [https://motion.dev/docs/react-svg-animation, https://motion.dev/docs/react-reorder].

> **Use `motion-helpers.tsx` — it already does the edge-drawing the old playbook wanted GSAP for.** The repo's `motion-helpers.tsx` already exports `DrawLine` and `DrawPath` (the latter animates `pathLength: 0→1`), plus `TweenText`, `FadeText`, `PopIn`. **The old playbook's "add GSAP DrawSVG for animated edges" recommendation is REJECTED for this stack** — it would add a *second, imperative* animation runtime (state lives in JS closures, not diffable JSX — the agent-hostile property the evidence set itself flagged GSAP `partial-fit` for) to do something Motion already does declaratively here. Extend `DrawPath`/`DrawLine` instead.
>
> **GSAP only if a genuinely Motion-impossible need appears** — and only **MorphSVG** for *topologically-different* path morphs (array-bar ↔ tree-node) is a real candidate. Even then, prefer **Flubber** (`interpolate()` + Motion `useTransform()`, ~3.5KB MIT) in one reusable `useMorphPath(from,to)` hook, `maxSegmentLength: 0.1` — fully declarative to the consumer, no second runtime [https://motion.dev/tutorials/react-path-morphing]. Reach for GSAP only after Flubber demonstrably can't do the morph.

**Explicitly reject for the build loop:** **Rive** (binary `.riv`, Canvas renderer breaks SVG, GUI-authored, agent can't read back what it made), **Lottie** (bezier-keyframe JSON, not semantic — agent can't generate algorithm-driven Lottie), **Theatre.js** (GUI-first, poor React-DOM support). **Motion Canvas / Remotion** are *offline-video-only* — bolt on later only if you want an MP4 walkthrough that reuses the same React+SVG components; never import into the SPA.

## 6b. The declarative-spec + auto-layout system — TRACK B, NET-NEW, OPTIONAL, NOT "one dependency"

> **Reality check, corrected from the old draft:** the viz layer is **24 hand-coded components that each position their own SVG**. There is **no `vizSpec`, no node/edge JSON, no per-element `source` citation field**, and **`@visx/hierarchy` / `d3-hierarchy` are installed but used for layout NOWHERE** (grep is empty; visx is used only for `scale`/`group`/`shape` *rendering* primitives in a handful of components). So a declarative-spec system is **a from-scratch architecture**, not a bolt-on — ELK + the spec runtime + the assertion layer + a patch-op format are **all net-new code**. The old claim "ELK is the one new dependency; everything else is already in `package.json`" was **false** and is retracted.

**When to even consider Track B:** only if §0's scored backlog shows the *bespoke* approach is the bottleneck — i.e., you're repeatedly hand-nudging x/y to stop overlaps across many new graph/tree viz. For the existing 24 (each already hand-positioned), ELK does nothing. If most future viz are number-lines, bar charts, and FSMs, Track B may never pay off.

**If you do build it, the design (kept from the old draft, now correctly scoped as net-new):**
- **Spec, not coordinates.** Agent emits `{ vizSpec, assertions }` as JSON (same pattern as the drill/tutor JSON it already emits): nodes/edges/labels/animation-steps/**`source` citation per element**/spatio-temporal predicates — but **no x/y**. The hard separation, from yWorks: **agent = intent + composition; a deterministic layout engine = coordinates; the agent never writes x/y** [https://www.yworks.com/blog/democratizing-graph-visualization-with-agents]. yWorks pulled LLMs out of the layout path explicitly: *"LLMs are not appropriate for precise, high-performance layout computation."*
- **Layout engine: ELK via `elkjs`** — deterministic single async call, JSON-in/JSON-out, `layered` (Sugiyama) for tree/DAG traces, `stress`/`force` for network topology [https://github.com/kieler/elkjs]. Wire like React Flow's example: `elk.layout(graph).then(p => setNodes(...))` in `useEffect(...,[rawSpec])` [https://reactflow.dev/examples/layout/elkjs]. **Cost note (corrected):** elkjs is a **~2MB WASM-backed bundle** — a real bundle-size decision, not free. Pick ELK over `dagre` (unmaintained since 2020) and over `d3-force` (iterative, circular-collision-only — wrong shape for CS boxes; use d3-force *only* for free-form network viz).
- **Render:** positioned spec → React + SVG (visx for scales/paths) → Motion `<motion.g animate={{x,y}}>` animates each element old→new ELK position. Zero-overlap (ELK) + smooth choreography (Motion) + inspectable JSON.
- **Assertion layer (MoVer-style):** TS spatio-temporal predicates over the *positioned* spec; paint a **yellow debug overlay** on failure (matches the brutalist aesthetic and the existing interaction-smoke gate). Treat the published one-shot→loop gains as motivation for the *mechanism*, not a promised number for this stack [https://mover-dsl.github.io/].
- **Incrementally patchable spec** (Apple Keyframer's lesson: emitting full CSS blobs makes refinement expensive — design `vizSpec` so the agent emits a `patch` op: add-node / update-step / change-citation, not a full re-emit) [https://machinelearning.apple.com/research/keyframer].

**Reject as the substrate:** GUI/MCP-driven tools (Rive/Figma — binary or paid, agent can't read back its own output → fails no-paid-API + agent-inspectability); code-first runtimes as the base (Motion Canvas — imperative generators, Canvas2D loses SVG inspectability, original project went dormant). Steal their *vocabulary* (concurrent generators, signal reactivity), not their runtime.

## 6c. The four non-SVG surfaces the old playbook silently ignored (Track A — must handle)

The deps include **`plotly.js-dist-min`, `mafs`, `katex`, `react-pdf`**, and there is a `PlotlyEmbed.tsx`. These break the "SVG-first / pixel-baseline / cited-per-element" assumptions §2–§3 lean on, so they need explicit handling:
- **Plotly** renders to **canvas/WebGL** — axe's DOM `color-contrast` check (§2.3) does not see it, and `toHaveScreenshot` will be noisier across GPUs. Gate Plotly with a **full-locator screenshot in Docker + a wider tolerance**, and rely on Plotly's own config for color/contrast (pick accessible color scales explicitly). Do **not** expect per-element citation here — it's a chart, treat it as one cited unit.
- **KaTeX** output is notoriously **overflow- and contrast-prone**; it IS DOM (so axe can read it) but tends to clip in narrow containers. Add an overflow probe (`scrollWidth>clientWidth`) specifically around KaTeX spans in §3.1's diagnostic snippet.
- **`mafs`** is React+SVG (fits the SVG-inspectable model) — usable in the self-seeing loop normally.
- **`react-pdf`** is a *source-material* surface, not a teaching viz — out of scope for the taste/correctness gates, but in scope for the existing interaction-smoke gate (the Slice-1.5 PDF-404 lesson lives here).

## 7. The recommended pipeline — one end-to-end loop, numbered

The loop the agent runs **per mounted viz surface**. Front-loads the cheap deterministic floor; spends vision tokens only when something's wrong; and ends at *the user can see it on the live route*, not at a screenshot file.

0. **(Retrospective, once)** Inventory the 24 → mount-audit against `vizRegistry.ts` → score taste + correctness → ranked backlog + calibrated threshold (§0). Everything below operates on a *mounted* component.
1. **Read the contract.** Agent reads `DESIGN.md` (+ the `vizSpec` schema if Track B exists); resolves every color/spacing/radius to a named `@theme` token. No arbitrary values [https://github.com/google-labs-code/design.md].
2. **Build / edit the component** using `motion-helpers.tsx` primitives + visx scales; (Track B: emit `{vizSpec, assertions}`, no x/y → ELK assigns coordinates).
3. **Mechanical floor (no LLM).** ESLint (`no-arbitrary-value: error`, no `rounded-*`), `tsc --noEmit`, vitest-axe `color-contrast` on the rendered DOM, MoVer spatial assertions in `browser_evaluate`. **Any failure → fix and restart at step 3. Do not advance.**
4. **PEDAGOGICAL-CORRECTNESS gate (§3.6).** Reference-implementation trace-match + source/KC content review. **Fail → fix and restart at step 4. This gate is non-negotiable and has no aesthetic substitute.**
5. **Diagnostic probe (no vision yet).** `browser_evaluate` → JSON: overflow (incl. KaTeX spans), animation-running, `fill-mode`, viewBox-vs-rendered, clipping. **Empty → skip to step 8.** Non-empty → step 6 [https://dev.to/ohugonnot/zero-overflow-in-10-minutes-responsive-testing-with-claude-code-and-playwright-59an].
6. **Three-frame vision critique.** Screenshot t=0/500/1000ms; one Claude vision call scores the 4-dimension correctness rubric + emits per-issue `{bbox%, selector, fix}` [https://arxiv.org/html/2510.11498v1, https://arxiv.org/pdf/2412.16829].
7. **Forced-optimization gate.** Apply fixes; **accept only if the score strictly improves**; resample ≤10×, else halt and surface [https://arxiv.org/html/2510.11498v1].
8. **Taste gate.** Playwright screenshot → evaluator scores Design Quality / Originality / Craft / Functionality vs. the two-anchor brutalist rubric, **3× temp-0.7 averaged**; gate = the *calibrated* floor from §0 + three consistency locks pass [https://www.anthropic.com/engineering/building-effective-agents, https://arxiv.org/html/2506.13639v1].
9. **MOUNTED-AND-VISIBLE gate (the project's most expensive recurring lesson).** Confirm the component is wired into `vizRegistry.ts` (or its route) AND **paints on the live tutor URL at the route a student actually hits**, with **zero 4xx/5xx during first paint** and no on-screen `/404|HTTP \d{3}|error/i` after interacting. Bundle + green tests ≠ shipped (global `CLAUDE.md`: Slice-1 ghost / Slice-1.5 PDF-404). The pipeline does **not** end at a screenshot file.
10. **Lock the baseline.** `toHaveScreenshot` (`reducedMotion:'reduce'`, **Docker-generated, `maxDiffPixelRatio:0.01`, baselines CI-only — never gate on a local Windows run**) on every step-state; commit PNG baselines [https://playwright.dev/docs/test-snapshots].
