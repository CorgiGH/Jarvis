# Repo organization & labeling standard

**What:** the single standard for where things live, how they're named, and what is tracked — so cleanup and navigation stay trivial for every future session. · **When:** 2026-06-19 (SESSION-82) · **Status:** active

This is the **schema layer** (in the sense of Karpathy's "LLM wiki" pattern): raw sources are immutable, derived knowledge is tracked, and *this doc + `.gitignore`* define the structure. Read it before creating files or doing cleanup.

## The rule in one line

Every file has (a) a **fixed home** by kind, (b) a **dated, self-describing name**, and (c) a **tracked-or-ignored** status. Anything ephemeral goes in `.scratch/`; anything large/raw goes in `material/`; both are gitignored, so the working tree never accumulates junk.

## Directory map

| Kind | Home | Tracked? |
|---|---|---|
| Production source | `src/`, `tutor-web/src/` | ✓ |
| Design specs / plans / findings | `docs/superpowers/{specs,plans,findings}/` | ✓ |
| Session research / audits / recon | `build-review/*.md` | ✓ (the `.md` only) |
| Council transcripts | `.claude/council-cache/council-<epoch>.md` | ✓ |
| Knowledge-concept corpus (digested) | `content/<SUBJECT>/` | ✓ |
| Tools / scripts (kept) | `tools/`, `tutor-web/tools/` | ✓ |
| **Raw course material** (immutable sources) | `material/<name>/` | ✗ gitignored (large) |
| **Ephemera** (screenshots, probes, temp, throwaway scripts) | `.scratch/` | ✗ gitignored |
| Build / cache / test output | `build-iso-review/`, `.vite/`, `test-results/`, `build-review/{classes,kotlin,libs,resources,tmp,reports}/` | ✗ gitignored |
| Served frontend bundle | `src/main/resources/tutor-dist/` | special — see Protected |

## Naming

- Dated artifacts: `YYYY-MM-DD-<kind>-<topic>.<ext>` — `kind ∈ {spec, plan, finding, research, audit, recon, council, design}`. Already the norm in `build-review/` and `docs/superpowers/`; apply it everywhere.
- No context-free names. `tmp-secondbrain-scrape` → `material/raw-corpus-secondbrain`. If a name doesn't say *what it is*, rename it.

## Ephemera & material rules

- **Ephemera → `.scratch/`** (gitignored). Screenshots, probe JSON, `.tmp-*`, throwaway `shoot.*.mjs` / demo harnesses, scratch outputs. Cleanup = delete `.scratch/`; nothing of value is ever there.
- **Raw material → `material/`** (gitignored). Big subject corpus (scrapes, PDFs). Digested output goes to `content/<SUBJECT>/` (tracked). See `material/_README.md` for the current contents.

## Protected — do NOT `git clean` / `git add -A`; touch only deliberately

- `src/main/resources/tutor-dist/` — the **deployed frontend bundle** (hashed `assets/*`, `fonts/`, demo HTML baked in). Regenerated by the Vite build; resolve as a deliberate build/deploy step, never an ad-hoc clean.
- `tutor-web/src/door/{DoorBrutalist,DoorCompare}.tsx, concept.ts, figures.tsx` — door/demo source, intentionally **untracked but wanted** (mirror on `Desktop/jarvis-door-snapshot/`). **Recommended permanent fix: commit these** so they're tracked like their siblings (`DoorWarm.tsx`/`ThemePicker.tsx`/`palettes.ts`) and the "never add -A" hazard disappears.

## Enforcement

`.gitignore` carries a `Cleanup standard (2026-06-19)` block that ignores `.scratch/`, `material/`, and all regenerable output. If junk reappears in `git status`, the fix is a `.gitignore` pattern + the right home — not a one-off delete.

## Cleanup procedure (future sessions)

1. `git status` — anything untracked that isn't a keeper means a missing home or `.gitignore` pattern.
2. Move ephemera → `.scratch/`, raw material → `material/`, never delete unverified work.
3. Hard-delete only regenerable output (build dirs, caches) and confirmed garbage.
4. Never `git add -A` / `git clean` while the door files are untracked (see Protected).
