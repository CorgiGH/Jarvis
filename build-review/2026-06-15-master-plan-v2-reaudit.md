## 2. REMAINING BLOCKER / MAJOR — with the exact fix

### MAJOR (the only non-cosmetic survivor) — INV-1.1 servable-view seam is unwired

**Where:** v2 line 224 (Phase F EXIT GATE) and line 256 (Phase I HONEST-100% EXIT GATE), against spec INV-1.1 (spec line 75).

**The hole (verified this pass):**
- Spec line 75, binding, literal: `INV-1.1 … SELECT COUNT(*) FROM <servable view> WHERE verification_status != 'VERIFIED'` returns 0, **against the real DB**.
- v2 line 256 names a `verifică sursa`-flagged **SERVABLE** row that is **visible + counted** and gates the board behind a budget. A `verifică sursa`-flagged row is by definition **not** `VERIFIED`. So the moment Alex's horn-(b) policy *renders* such a row (making it servable, v2 line 32 + line 148), the literal spec query returns >0 → INV-1.1 RED → the line-256 "board prints 100% if flagged count ≤ budget" path is **unreachable whenever the budget is used.**
- v2 has **zero** carve-out language: grep for `servable view` / `flagged-servable` / `carve` / `admission-gated` over v2 returns **no matches**. v2 never scopes the INV-1.1 query's `<servable view>` to exclude flagged-rendered rows, and never adds a status-enum value to distinguish a `verifică sursa` stub from a plain-unverified row.
- v2 line 224 says INV-1.1 = "every servable row VERIFIED — **or** flagged `verifică sursa` + counted"; v2 line 256 says "INV-1.1 **re-asserted**: flagged ≠ done; a flagged row is RED-on-coverage." These are one policy at two altitudes (display honesty vs scoring), not a flat contradiction — but neither one reconciles with the *actual CI query*, and the word "servable" is reused for two different sets (the all-VERIFIED INV-1.1 set, and the flagged-rendered set) with no view boundary between them.

**Why it survived the check at MAJOR, not blocker:** the BINDING SPEC already supplies the reconciliation skeleton v2 failed to cite — §15.2 line 631 defines the assumed-knowledge stub as a *flagged placeholder carrying `neverificat / verifică sursa`* that is part of the served graph; INV-EC4.3 line 656 explicitly distinguishes "VERIFIED-servable KCs (INV-1.1)" from "unverified KCs in the target surface as `neverificat`." So in spec terms a flagged-rendered row is NOT inside INV-1.1's `<servable view>` — it is flagged content surfaced separately, excluded from mastery. The reconciliation is *implied by the spec's own model*; v2's defect is that it reuses the word "servable" for both sets and never makes the view boundary explicit, NOT that reconciliation is impossible. Hence MAJOR (one unwired sentence on the document's most load-bearing fix), not blocker (no dead end, no silent serve-as-truth).

**EXACT FIX (pick one, state it once, stop citing INV-1.1 as both relaxed at 224 and re-asserted at 256):**
- **(a) preferred — scope the view + add a counted budget invariant.** Define, in §0.2 or Phase 0's coverage registry, that the spec INV-1.1 query runs over the **admission-gated VERIFIED set only** (`<servable view>` = rows that passed faithful-check, per spec §1.2(b)); a `verifică sursa`-flagged-rendered row lives in a distinct **`flagged-servable`** surface that the INV-1.1 query EXCLUDES. Add a SEPARATE invariant: `COUNT(flagged-servable rows) ≤ displayed per-dimension budget` (target 0). Then INV-1.1 stays a true `0 == 0` decidable gate, horn-(b) rendering is legal, and the board math is well-defined. Rewrite BOTH line 224 and line 256 to this single definition.
- **(b) alternative — keep INV-1.1 literal** (flagged rows are non-servable / refused) — but then horn-(b) "render behind a warning" is impossible, contradicting Alex's ratified policy (line 32). Not recommended; (a) is the policy-faithful horn.

This is the **single remaining pass v2 needs.** Land (a) and v2 is pristine.
