## 1. HEADLINE VERDICT

**NO — has holes. The plan does NOT, as written, reach an honest 100%-to-spec.**

It is the best plan this project has produced — the oracle-first / breadth-last resequence is correct, the per-gate matrix and the P1 NO-SPEC-CUT law are the right machinery, and the refuted-findings list shows the authors anticipated most of the obvious attacks. ## 2. THE BLOCKERS (confirmed; would prevent honest 100% / silently downsize spec)

### B1 — F2: Circular dependency — A/B/C exit gates require the proof-set that Phases E/F produce downstream

- **What's wrong:** Phase B's exit gate (line 91) requires "depth gate green **on the proof-set KCs**"; Phase C's (line 105) requires "discriminating-CHECK gate green **on proof-set KCs** … the precondition for opening generation." But the depth artifacts, misconception-deltas, and discriminating-CHECK items those gates read are **generation outputs of Phase E** (line 129: "Generation emits … depth artifact (Phase B) + misconception delta + discriminating CHECK items (Phase C)") — and the GATE OF GATES (line 120) forbids Phase E until A+B+C are green. The content the gates must judge is produced downstream of the gates that require it.
- **Evidence:** Repo confirms ZERO existing content carries any depth / discriminating-CHECK / misconception-delta field (`grep -rliE 'depth|discriminat|misconception_delta|target_conception'` over `content/` = 0 files). Only ~7 real KCs exist (PA 6 + ALO 1), spanning 4 of 8 `concept_type` enum values; POO/PS/SO-RC have 0 KCs. The §10.2 proof-set deliberately spans PA/ALO/PS/POO/SORC and all concept-types — none of which exists yet. So the B/C exit gates either run **vacuously** (forbidden by the plan's own anti-vacuity law INV-9.4 / P1 "never fake-green") or require **unscheduled manual authoring** that no phase names.
- **Why blocker:** It is a contradiction in the plan's central hard-ordering rule — the GATE OF GATES the entire oracle-first thesis rests on. The two horns (vacuous gate OR silent authoring) hold even under the most charitable reading (gate only the ~7 existing KCs), because those KCs also carry zero depth/CHECK fields.
- **Exact fix:** Split each oracle phase into two gates:
  - **Early:** "oracle BUILT + self-test green on **seeded fixtures** + green on the ≤7 real KCs once their depth/CHECK bank is authored."
  - **Late (moved AFTER a first E/F slice):** "oracle green on the full §10.2 proof-set."
  - **OR** insert an explicit in-scope deliverable in Phase B/C: "hand-author / digest the depth + misconception-delta + discriminating-CHECK bank for the ~7 existing real KCs **before** the B/C exit gates fire," and relabel the current Phase E as "E-bulk." Name this authoring as in-scope effort, not a free rider on the Phase-E firehose.

### B2 — F27: "Honest 100%" is defined to accept `neverificat`-flagged servable content (honest-100 loophole)

- **What's wrong:** Phase I's exit gate (line 175) defines honest 100% as "all PROOF green · every **ESTIMATOR green-or-`neverificat`-flagged** · §10 user-checkpoint passed." Because depth (dim 13) and pedagogy (dim 14) — the two net-new load-bearing oracles the JUDGE calls the whole point — are inherently ESTIMATORs, a "100%" board can print green while those exact dimensions are mostly `neverificat`. That is a downsized spec wearing a green label — the single-aggregate laundering anti-pattern council-1781435843 named, re-created one level up. The Phase-I 100% one-liner does **not** re-state the servability coupling (it omits "and every `neverificat` artifact is provably suppressed from servable/mastery/faithful surfaces, INV-1.1/1.2 still hold"), so a reader of line 175 in isolation can read it as licensing served-but-unverified content.
- **Evidence:** Spec §1.2(b) (line 32) mandates always-on serve-side refusal of unverified content; INV-1.1 (line 75) requires 0 non-VERIFIED servable rows; a `neverificat` artifact is by definition not VERIFIED. The refutation pass downgraded the original "served-as-truth" charge (the plan DOES suppress `neverificat` depth per lines 89/91 and EC2/EC4.3), which is why this is a **definition-honesty** loophole rather than a serve-bug. But the residual is real and blocker-grade: the headline "100%" definition is the load-bearing honesty claim of the whole document, and as worded it lets the count of `neverificat` servable artifacts be non-zero while printing 100%.
- **Why blocker:** The user's literal demand is "pristine, spec NOT downsized." A 100% definition that admits flagged-but-undone estimator content on the two oracle dimensions is the precise failure the audit was commissioned to catch.
- **Exact fix:** Tighten the Phase-I exit gate to make `neverificat` and 100% mutually exclusive on servable rows. Either:
  - (a) `neverificat` artifacts are NON-SERVABLE (parked/refused, INV-1.2) — then "100%" requires **ZERO `neverificat` flags on any servable row**, and the exit gate must say so explicitly with an INV-1.1/INV-1.2 re-assertion; OR
  - (b) if `neverificat` content is ever served, state plainly that **100%-to-spec is NOT reached while any servable row is `neverificat`**, print those RED-on-coverage, and add a hard board floor (the board prints "100%" ONLY if the count of `neverificat` servable artifacts is 0, or below a tiny Alex-ratified per-dimension budget **displayed on the board**).
  - Delete the "green-or-flagged = 100%" wording for servable rows; flagged ≠ done.

### B3 — F30: Phase B suppresses only the mastery/faithful BADGE on `neverificat` depth, but still RENDERS the derivation/code (violates §1.2(b))

- **What's wrong:** Phase B (lines 89/91) protects against ungated depth only by excluding it "from every `mastery`/`faithful` UI signal" and asserting it is "suppressed from mastery/faithful UI." That is BADGE suppression, not refusal-to-render. The depth body (invariant, complexity derivation, correctness argument, code) is still **rendered as lesson content**; only the trust-badge is stripped. A finals student reading an LLM-generated complexity derivation that is wrong internalizes the wrong derivation regardless of whether a green badge sits beside it — council-1781433195's "discovered in the exam room, unrecoverable."
- **Evidence:** Spec §1.2(b) (line 32) is a serve-side REFUSAL to **render** unverified content ("Nothing unverified reaches the learner"). Phase B substitutes the strictly weaker "render it without a badge." The depth artifact is net-new (line 190, "no prior plan hosted it") and is NOT in INV-1.1's enumerated servable set (KCs, beat fields, figures, problems), so it does NOT auto-inherit §1.2(b)'s refusal — Phase B is the only place that could bind it, and it binds only the badge.
- **Why blocker:** It is the plan's OWN load-bearing Risk #1 mitigation (line 201) — the depth-carries-`neverificat`-excluded-from-mastery-UI claim — and the mitigation under-extends the spec's actual protection, leaving the authoritative-looking-wrong-proof hole open at exactly the layer the plan says carries false `mastery` straight into an exam.
- **Calibration note (do not overstate the fix):** The refutation pass showed the finding's *prescribed* fix (ZERO learner-visible derivation/code; serve only a FILES gap-record) **overshoots** the council it cites — council-1781433195 (Judge path #3, Risk-Analyst mit #2) permits rendering depth content **as long as it is visibly flagged "unverified / verifică sursa," never wearing verified authority.** So Phase B's real defect is that it specifies NEITHER the spec's refusal NOR the council's mandatory-visible-flag — only badge exclusion.
- **Exact fix:** Make Phase B do one of the two stronger things, not badge-strip:
  - (a) **literal §1.2(b):** an ungated depth artifact is NOT rendered as content — served as a FILES gap-record requesting the authored-source depth material; add a DOM test asserting a `neverificat` depth artifact produces ZERO learner-visible derivation/code text; OR
  - (b) **council-permitted:** the depth body renders only with a mandatory, prominent, non-dismissible "neverificat / verifică sursa" warning, asserted in the DOM, AND excluded from mastery — and the plan must state which of (a)/(b) is chosen.

## 3. THE MAJORS (confirmed; weaken 100% or reality-fidelity)

### F6 — `every-frame-advances` measures rendered pixels, not algorithm-state; the dead-frame class survives (→ major after check)

Phase A's `every-frame-advances` gate (line 77) is a **perceptual-hash** dead-frame detector: it proves PIXELS changed, not that the ALGORITHM'S STATE advanced. The real on-disk merge instance (`viz-pa-mergesort-runs-001.yaml`) has four `phase:"divide"` frames where the array is byte-identical (`[5,2,8,1,9,3]`) and only the partition brackets move — a perceptual hash sees bracket motion and PASSES while sorting state is frozen, exactly the "frames 2,3,4 do nothing" defect (council-1781475889 §3). `monotone-progress` (inversions non-increasing) is also satisfied by a frozen array (inversions = 7,7,7,7, constant). Trace-match (INV-5.1) does NOT save it — a correct mergesort reference genuinely has static-array divide steps, so the static frames MATCH the reference and are green-lit. The existing per-family semantic suite (`MERGE_INVARIANTS`) also passes all four divide frames. **Fix:** compute `every-frame-advances` over the **semantic algorithm-state delta** (the typed state vector the generate-from-algorithm stream emits), not a rendered hash; assert each non-hold step mutates the state vector; require structural-only frames (`phase=='divide'`) to be explicitly tagged `hold:true` subject to a **max-consecutive-holds cap**; add as an explicit Phase-A exit clause so a pixels-moved-but-state-frozen frame reds. (Major not blocker: the gate still catches a truly static figure and `final==end-state` catches never-sorts; only the pixels-moved-yet-frozen leak survives.)

### F7 — Phase C discriminating-CHECK gate is an LLM-grounded ESTIMATOR presented as a PROOF / GATE OF GATES (→ major after check)

Phase C (lines 38/100/105/120) calls the discriminating-CHECK gate "provably separate," "terminal," "the precondition for opening generation," and the GATE OF GATES — and lets a lesson ship `faithful`/`mastery` on its strength. But with no live learner, the wrong-model and right-model are themselves LLM constructs, the misconception-delta + CHECK items are LLM-authored, and the judge is an LLM: an LLM judging whether LLM-authored items discriminate against LLM-simulated models (circular self-grounding). The binding JUDGE itself titles Stage C a "**PEDAGOGICAL-QUALITY ESTIMATOR**" (line 60) and admits it "could rubber-stamp 'renders perfectly, teaches a falsehood'" (line 86); spec §A.3.3 (line 733) puts learning-gain measurement out of scope. The plan escalates the council's ESTIMATOR to the plan's PROOF — a P1 violation. **Why major not blocker:** the plan is NOT silently fake-green here — it carries the proxy admission as Risk 5 (line 209), wires CHECK-beat miss-data feedback, and the matrix supplies a `neverificat` fallback (dim-14 "else `neverificat`"). The defect is the OVER-CLAIM in the gate's authority/labeling, not a missing safety net. **Fix:** demote dim-14/Phase-C from "decidable PROOF / GATE OF GATES / provably" to an explicitly ESTIMATOR-tagged lane in the matrix; the gate may ASSERT only the decidable half (distractor↔named-misconception-id binding on THIS KC, non-paraphrase trigger, ATTEMPT-precedes-answer) and must serve `neverificat` for the undecidable "separates models" claim; require the wrong/right-model responders be grounded in mined R-ERRORS and generated by a **second independent LLM family** (no single-model self-grading); forbid `mastery`/`faithful` stamping on Phase-C-only evidence until N real-learner miss-data points confirm discrimination; stop calling Phase-green "the precondition" as if it proves teaching — it BOUNDS, not eliminates, wrong-teaching.

### F8 — Depth oracle's complexity/invariant/edge legs are LLM judgments presented as a green PROOF gate

Phase B's exit gate (line 91) conjoins four depth legs under one "green," but only **one is decidable**: "code compiles + passes tests" (rides the real Phase-G execution graders, INV-6.2) — and that proves the CODE runs, never that the prose complexity-derivation or invariant argument is right (a test-passing bubble sort can ship under a depth artifact whose prose claims O(n log n)). The other three — "complexity derivation sound," "invariant re-derivable," "edges enumerated and correct" — have NO named decidable mechanism; they are ESTIMATORs that turn the gate green with no inline `neverificat` requirement. This is exactly council-1781433195's binding ruling: depth needs its OWN correctness gate or authored-source grounding; until gated, flag `neverificat`. The SIBLING Phase C wires the estimator→`neverificat` routing inline ("else `neverificat`") — proving the plan knows the pattern and failed to apply it to Phase B's three estimator legs. **Fix:** split the depth gate into a decidable PROOF leg ("the code runs," nothing more) and an ESTIMATOR leg (complexity-soundness, invariant-rederivability, edge-completeness) that stays `neverificat` until grounded against an authored source or a second-independent-derivation cross-check; enforce **per-leg suppression** — forbid the depth artifact from contributing to mastery while ANY estimator leg is `neverificat`, not all-or-nothing green.

## 5. THE PATTERN (the meta-finding)

Three weakness CLASSES recur, and they are not random — they cluster on exactly the parts of the plan that are net-new or oracle-bearing:

**CLASS 1 — Estimator-dressed-as-proof on the two net-new oracle phases (the dominant pattern).** F7, F8, F6, F16, F30, F41, F42, F43, and the F27 honesty-loophole all share one shape: a dimension whose honest gate is an ESTIMATOR (or whose decidable leg proves a *weaker* thing than the clause demands) is written into the plan as a green PROOF / "provably" / "the precondition" / "GATE OF GATES" — and then the gate is allowed to license downstream breadth or `mastery`/`faithful` ship-state. The plan *invented* the correct routing (P1: PROOF gate OR ESTIMATOR + `neverificat` + checkpoint; the matrix's PROOF/ESTIMATOR tags) and then **failed to apply its own routing to the depth and pedagogy legs** — the exact dimensions the JUDGE calls load-bearing. Phase C even demonstrates the correct inline `neverificat` routing ("else `neverificat`"), which Phase B omits. This is the meta-finding: **the plan's honesty machinery is sound, but it is not wired through the two oracles it exists to protect, so a "100%" can print while the only two dimensions that measure *teaching* are estimates wearing proof labels.**

**CLASS 2 — Effort-downplay folded into peer-sized phase bullets on the zero-code half.** F20, F22, F23, F15, F34 (and F40's missing schema task): the entire digestion/generation half (~50% of spec, zero code) and the net-new generate-from-algorithm + OCR subsystems are compressed into peer-sized bullets, while satisfied work (F17, F24, F25) is listed as pending — so the effort picture is distorted in both directions, making the genuinely-huge generation cell look like a peer of the small foundation cells. The plan states the magnitude in prose (line 15) but does not reflect it in phase granularity or sub-gates.

**CLASS 3 — Reality-drift in the ground-truth header.** F14, F19, F37, F31, F36, F25, F17: the "Ground truth established" header and several phase notes present uncommitted/untracked work, mis-stated content census, file-based artifacts, and already-done items as established DB reality. The header is the one place the plan claims to be grounded, and it is the least verified against committed HEAD + the live DB.

The unifying root cause: **the plan applies its rigor outward (to the gates it designs) but not inward (to its own claims and to the two estimator oracles).** The fixes in §6 are almost entirely "apply the plan's own P1/matrix discipline to the plan's own oracle phases and ground-truth claims."

## 7. HONEST RESIDUALS (cannot be fully proven — must be named, not hidden, so "100%" stays honest)

These are real limits of what ANY version of this plan can machine-prove. They must be stamped on the matrix as ACCEPTED LIMITATIONS carrying `neverificat` + a human-checkpoint record, never folded silently into a green "100%":

1. **Teaching effect without a live learner is not measurable.** Spec §A.3.3 itself scopes learning-gain out. The Phase-C discriminating-CHECK gate is the closest computable proxy ("does this item separate a wrong-model from a right-model responder"), but with no real student population it judges LLM-simulated models — it BOUNDS wrong-teaching, it does not eliminate it. This must be labeled the load-bearing residual, exactly as the JUDGE (line 86) holds it.

2. **Depth-derivation soundness (complexity, invariant) is undecidable without authored-source grounding.** "Code compiles" is decidable; "this O(n log n) derivation is correct prose" is not, absent an authored reference. Until a second-independent-derivation or authored-source cross-check exists, the depth estimator legs are `neverificat`, not proof.

3. **Free-tier generator capability against the new oracles is unknown until measured.** Whether the available free/relay model can produce content that clears the depth+pedagogy bars is an empirical fact the capability spike will reveal; until then, the breadth volume Phase F can honestly produce is unknown, and "100% coverage" may legitimately resolve to "100% of what cleared the oracle, the rest `neverificat`."

4. **Synthetic-tagged exam coverage is not real-paper coverage.** Spec §11.1: ALO/PS/RC have zero past papers anywhere. Dimension-16 "all 5 subjects" will be partly synthetic; the synthetic fraction per subject must be displayed, not absorbed into an all-real reading.

5. **The human checkpoint judges experience, not truth — and the checkpoint-holder cannot vet content.** "§10 user-checkpoint passed" is a real compliance leg (spec §1.4/§10.3, binding), but it is an EXPERIENCE approval, not a teaching-truth oracle. "100%" must state this is "every machine gate green + experience-approved," explicitly NOT "no human ever sees a defect" (line 175 already says this — keep it, and do not let any future edit quietly upgrade the checkpoint into a quality oracle).

A "100%" that names these five as accepted, flagged residuals is honest. A "100%" that hides them is the downsized-spec-wearing-a-green-label the user asked this audit to prevent.
