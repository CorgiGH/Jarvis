# Final Audit Report — Template × Corpus × Engine (2026-06-10)

**Question audited:** "Is the current engine currently able to turn all the different lectures into properly-taught content (visuals, demonstrations, tracking) the best way?"

**Inputs:** 200 adversarially-verified findings from 12 finder dimensions (6 subjects + engine-delivery, authoring-schema, tracking-adaptivity, viz, language, pedagogy-profile), deduplicated here to **135 distinct flaws** (23 blockers, 53 high, 46 medium, 13 low; 30 match the known-flaws ledger, 105 are new). Baseline ground truth: `build-review/tmp/audit-context-2026-06-10.md`. Zero findings were refuted outright; several were corrected in detail (noted inline).

---

## 1. Verdict

**No.** The engine cannot teach any of the six subjects today, and the distance to "all lectures, properly taught, the best way" is not a polish gap — it is four missing layers stacked on top of each other.

**What the engine can teach TODAY: 0% of the corpus.**
- The trust-net has never run in production: `kc_verification_status` = 0 rows, live DB schema 3 columns behind the code (TRUST-01/02). The lesson endpoint 404s on anything non-faithful, so the faithful-gated queue is empty for every user — including PA's 6 authored KCs.
- 4 of 5 subjects (ALO, POO, PS, SO-RC) have literally zero KCs — `content/*/kcs/` is `.gitkeep` (CONTENT-01).
- The only functioning surface is the `/review` flip-card deck: 828 LLM-generated cards, all `kc_id=NULL` (invisible to mastery/queue/adaptivity), ~all English (hard-line violation), self-graded, zero visuals, sitting entirely outside the trust-net.

**After the smallest unblock (DB backup → migrate 3 cols → verifyContent → sync): ~2% of the corpus, badly.**
- 6 KCs from PA lecture 1 (1 of ~9 PA lectures; foundational vocabulary, inverted relative to actual exam weight) become servable — through a lesson surface that is **not the locked template**. The shipped `LessonScreen` is a 3-step read-echo-recall: beat ① predict gate never fires (`prediction_options` hardcoded `[]`, no schema field to author it), beat ② attempt does not exist, beat ③ animated reveal does not exist (no viz on the lesson wire at all), beat ④ `definition_ro` is hardcoded null, beat ⑤ check does not exist, and completion writes **nothing** — no mastery, no FSRS, no attempt row. Both free-text inputs are ungraded dead ends.
- The locked 5-beat template exists only as 3 throwaway vanilla-JS+D3 demos (`build-review/viz-fork-demo/`: MergeSort, Dijkstra, QR-Householder) covering exactly 3 algorithm-trace concept types. Porting them to the app is a rewrite, not an extraction (no BeatOrchestrator, no callout system, no dwell hook, no beat state machine in React).

**To reach "all lectures, properly taught," every one of these must land — none exists today:**
1. **Content:** author ~40–120 KCs × 5 subjects + prerequisite graphs + per-beat teaching fields — against a raw corpus where PA lectures 2–8 are PDF-only, PS's 10 course files are watermark-garbled beyond quotation, ALO has 3/10 courses as slide-title dumps, SO's corpus is POSIX programming (not the OS theory the viz components target), and RC has 3 thin lectures.
2. **Schema:** the KC schema has **no field for any of the 5 beats** (no predict prompt/options/callback, no attempt sub-task, no reveal steps/callouts, no definition_ro, no canonical_answer) and the curate-tutor skill specs none of them; the verbatim-attribution trust rule structurally conflicts with the invented concrete instances beat ① requires.
3. **Engine:** a real 5-beat orchestrator with beat-selection by concept-type × mastery, graded beats, completion writes, lesson→drill handoff (the `/oggi` "Începe" button is currently a no-op), and a grader that can handle numbers, code, and non-statistics misconceptions (today: LLM-judge only, 3 hardcoded mean/median misconception codes, "You do NOT execute the code").
4. **Viz + language:** 24 of 25 built viz components unreachable (registry has 1 entry, zero production KCs carry `viz_id`, lesson wire has no `viz_id` field, 17/24 components are zero-prop hardcoded scenarios, G4 architecture fork unresolved); and a Romanian enforcement layer that does not exist anywhere (generator, grader, validator, UI chrome, DB cards — all English-leaking).

**Template-fit reality:** the template is demonstrated only for algorithm traces. Proofs, definition-heavy theory, probability, OS timing/interleaving, code-reading, and API taxonomy — the majority of the actual exam corpus — have **no beat mapping**, and several (formal proofs, reduction constructions) are likely unmappable without new beat variants.

**Timing:** finals are June 1–21. Today is June 10. No combination of the above fits in the remaining ~11 days; there is currently no Plan-B path (e.g., fixing the 828 cards) in any plan doc.
